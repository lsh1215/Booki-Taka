# 토픽과 파티션

> 출처: 카프카 핵심 가이드 Ch1·Ch6, 실전카프카 Ch3·Ch4, 아파치 카프카 애플리케이션 프로그래밍 Ch3
> 태그: #topic #partition #offset #segment #record #replication #isr #log-compaction

## 이 토픽의 근본 문제

대량 데이터를 어떻게 병렬 처리하면서도 순서를 보장하고, 장애 시에도 데이터를 잃지 않을 것인가.

카프카는 이 문제를 세 계층으로 해결한다.
- **논리적 분리**: 토픽으로 데이터 스트림을 구분
- **물리적 병렬화**: 파티션으로 단일 토픽을 여러 브로커에 분산
- **내구성 보장**: 리플리케이션과 ISR로 브로커 장애를 투명하게 처리

## 전체 구조

```
[Topic: orders]
  ├── Partition 0  ──── Broker 1 (Leader)  ←── Producer
  │     Offset: 0,1,2,...                        (키 기반 or 라운드로빈)
  │     Segment: 00000000000000000000.log
  │
  ├── Partition 1  ──── Broker 2 (Leader)
  │     Offset: 0,1,2,...
  │
  └── Partition 2  ──── Broker 3 (Leader)
        Offset: 0,1,2,...

[Replication: factor=3]
  Partition 0:  Broker1(Leader) ← Broker2(Follower) ← Broker3(Follower)
  Partition 1:  Broker2(Leader) ← Broker1(Follower) ← Broker3(Follower)
  Partition 2:  Broker3(Leader) ← Broker1(Follower) ← Broker2(Follower)

[Segment 구조 (Partition 0, Broker 1)]
  /data/orders-0/
    ├── 00000000000000000000.log        ← 실제 레코드 데이터
    ├── 00000000000000000000.index      ← 오프셋 → 파일 위치 매핑
    ├── 00000000000000000000.timeindex  ← 타임스탬프 → 오프셋 매핑
    ├── 00000000000001000000.log        ← 다음 세그먼트 (비활성)
    └── 00000000000002000000.log        ← 활성 세그먼트 (현재 쓰기 중)
```

## 핵심 개념

### 토픽 (Topic)

**정의**
데이터를 구분하는 논리적 단위. 관계형 DB의 테이블, 파일시스템의 폴더에 비유할 수 있다. 프로듀서는 특정 토픽에 메시지를 쓰고, 컨슈머는 특정 토픽을 구독한다.

**왜 필요한가**
토픽이 없으면 모든 메시지가 단일 스트림에 섞인다. 결제 이벤트, 로그인 이벤트, 재고 변경 이벤트가 뒤섞이면 컨슈머는 자신에게 필요한 메시지를 필터링하기 위해 전체 스트림을 읽어야 한다. 토픽으로 분리하면 컨슈머는 관심 있는 데이터만 소비한다.

**동작 원리**
- 토픽은 [파티션](#파티션-partition)의 논리적 묶음이다
- 토픽 생성 시 파티션 수와 [리플리케이션 팩터](#리플리케이션-replication)를 지정한다
- 브로커는 토픽 메타데이터를 ZooKeeper(구버전) 또는 KRaft(신버전)에 저장한다

**이름 규칙**
- 허용 문자: 영문자, 숫자, `.`, `_`, `-`
- 최대 길이: 249자
- `.`과 `_` 혼용 금지: 내부적으로 `.`을 `_`로 변환하므로 `my.topic`과 `my_topic`이 충돌 가능
- 권장 패턴: `{환경}.{팀명}.{서비스}.{포맷}` → 예) `prd.marketing-team.sms-platform.json`

**설계 트레이드오프**
| 결정 | 장점 | 단점 |
|------|------|------|
| 세분화된 토픽 (1이벤트 = 1토픽) | 컨슈머 필터링 불필요, ACL 관리 용이 | 토픽 수 폭발, 관리 복잡도 증가 |
| 통합 토픽 (이벤트 유형을 헤더로 구분) | 토픽 수 감소 | 컨슈머가 불필요한 메시지도 역직렬화 필요 |

**관련 개념**: [파티션](#파티션-partition), [리플리케이션](#리플리케이션-replication)

### 파티션 (Partition)

**정의**
토픽을 물리적으로 분할한 단위. 각 파티션은 순서가 보장되는 불변(immutable) 로그다. 파티션 번호는 0부터 시작한다.

**왜 필요한가**
단일 로그(파티션 1개)는 처리량이 단일 브로커의 I/O 성능에 제한된다. 파티션을 늘리면 데이터를 여러 브로커에 분산하여 수평 확장이 가능하다. **파티션 수 = 해당 토픽을 동시에 읽을 수 있는 최대 컨슈머 수**이다.

```
[파티션이 1개인 경우]                [파티션이 3개인 경우]
  Topic: payments                      Topic: payments
  Partition 0 (Broker 1)               Partition 0 (Broker 1) ← Consumer A
    ← Consumer (병목)                  Partition 1 (Broker 2) ← Consumer B
                                       Partition 2 (Broker 3) ← Consumer C
```

**동작 원리**
- 프로듀서가 메시지를 보내면 파티셔너(Partitioner)가 어느 파티션으로 보낼지 결정
  - 키가 있으면: `hash(key) % partition_count` → 같은 키는 항상 같은 파티션
  - 키가 없으면: 스티키 파티셔너(Sticky Partitioner, 2.4+) 또는 라운드로빈
- 각 파티션은 하나의 [리더](#리플리케이션-replication)만 가지며, 읽기/쓰기는 리더를 통해서만 수행

**파티션 수 결정 기준**
```
권장 파티션 수 = max(목표_처리량 / 프로듀서_처리량, 목표_처리량 / 컨슈머_처리량)

예시: 목표 1GB/s, 프로듀서 100MB/s, 컨슈머 50MB/s
      = max(1000/100, 1000/50) = max(10, 20) = 20개
```

**설계 트레이드오프**
| 파티션 수 증가 시 | 효과 |
|-----------------|------|
| 병렬 처리량 증가 | 긍정 |
| 파일 핸들 수 증가 (파티션당 세그먼트 파일 다수) | 부정 |
| 리더 선출 시간 증가 (브로커 장애 시) | 부정 |
| 컨트롤러 부하 증가 | 부정 |
| end-to-end 지연 증가 가능 | 부정 |

**중요**: 파티션 수는 **늘릴 수만 있고 줄일 수 없다**. 줄이려면 토픽을 삭제하고 재생성해야 한다. 키 기반 라우팅 사용 중이라면 파티션 수 변경 시 같은 키가 다른 파티션으로 이동할 수 있다.

**관련 개념**: [토픽](#토픽-topic), [오프셋](#오프셋-offset), [세그먼트](#세그먼트-segment), [레코드](#레코드-record)

### 오프셋 (Offset)

**정의**
파티션 내 각 레코드에 부여되는 고유한 순번. 0부터 시작하며 단조 증가(monotonically increasing)한다. 파티션 간 오프셋은 독립적이다(Partition 0의 offset 5와 Partition 1의 offset 5는 다른 메시지).

**왜 필요한가**
오프셋이 없으면 컨슈머가 어디까지 읽었는지 추적할 수 없다. 컨슈머 재시작 시 처음부터 다시 읽거나, 못 읽은 메시지를 놓치게 된다.

**동작 원리**
```
[Partition 0]
  Offset:  0      1      2      3      4      5
           ┌──────┬──────┬──────┬──────┬──────┬──────
           │ Rec0 │ Rec1 │ Rec2 │ Rec3 │ Rec4 │ ...
           └──────┴──────┴──────┴──────┴──────┴──────
                                        ↑
                              Consumer 커밋 위치 (committed offset)
                              → 재시작 시 offset 4부터 재개
```

- 컨슈머는 읽은 오프셋을 `__consumer_offsets` 토픽에 커밋
- 커밋 방식:
  - **자동 커밋**: `enable.auto.commit=true`, `auto.commit.interval.ms` 주기로 자동 커밋
  - **수동 커밋**: `commitSync()` (동기, 느리지만 안전), `commitAsync()` (비동기, 빠르지만 실패 시 재시도 복잡)
- **오프셋 리셋**: `auto.offset.reset` 설정
  - `earliest`: 파티션의 첫 번째 오프셋부터
  - `latest`: 컨슈머 그룹 등록 이후 새로 들어오는 메시지부터

**설계 트레이드오프**
- 오프셋은 **파티션 내에서만 순서를 보장**한다
- 전역 순서(Global Order)가 필요하면 파티션을 1개로 제한해야 하는데, 이는 처리량 병목을 유발한다
- 정확히 한 번(Exactly-Once) 처리가 필요하면 트랜잭션 API 또는 멱등성 프로듀서를 사용해야 한다

**관련 개념**: [파티션](#파티션-partition), [레코드](#레코드-record)

### 레코드 (Record)

**정의**
카프카에서 데이터를 전달하는 최소 단위. 메시지(Message) 또는 이벤트(Event)라고도 한다.

**구조**
```
┌─────────────────────────────────────────────────────┐
│  Record                                             │
│  ┌──────────┬──────────┬──────────┬──────────────┐  │
│  │  Key     │  Value   │  Header  │  Timestamp   │  │
│  │ (byte[]) │ (byte[]) │ (K=V 쌍) │ (ms, epoch)  │  │
│  └──────────┴──────────┴──────────┴──────────────┘  │
│  + Offset (브로커가 파티션에 쓸 때 부여)              │
└─────────────────────────────────────────────────────┘
```

**키(Key)의 역할**
- 키가 없으면(null): 라운드로빈 또는 스티키 파티셔닝으로 분배
- 키가 있으면: `murmur2 hash(key) % partition_count` → 같은 키는 항상 같은 파티션
- 같은 파티션 내에서 순서가 보장되므로, 특정 엔티티(예: 사용자 ID)의 이벤트 순서를 보장하려면 키로 해당 ID를 사용

**설계 트레이드오프**
| 상황 | 문제 |
|------|------|
| 키 분포가 불균등 (특정 키에 이벤트 집중) | 파티션 핫스팟(Hotspot) 발생 → 해당 파티션의 컨슈머만 과부하 |
| 키를 파티션 수 변경 후 유지 | 같은 키가 다른 파티션으로 이동 → 순서 보장 깨짐 |
| 값(Value) 크기가 큰 레코드 | `max.message.bytes` 설정 필요, 브로커 메모리 압박 |

**관련 개념**: [파티션](#파티션-partition), [오프셋](#오프셋-offset)

### 세그먼트 (Segment)

**정의**
파티션의 물리적 파일 단위. 하나의 파티션은 여러 세그먼트 파일로 구성된다. 세그먼트 파일 이름은 해당 세그먼트의 첫 번째 오프셋으로 명명된다.

**파일 구성**
```
/data/kafka/orders-0/
  ├── 00000000000000000000.log        ← 레코드 바이너리 데이터
  ├── 00000000000000000000.index      ← 오프셋 → 파일 바이트 위치 매핑 (희소 인덱스)
  ├── 00000000000000000000.timeindex  ← 타임스탬프 → 오프셋 매핑
  ├── 00000000000001048576.log        ← 비활성 세그먼트 (읽기만 가능)
  └── 00000000000002097152.log        ← 활성 세그먼트 (현재 쓰기 중)
```

**활성 세그먼트 vs 비활성 세그먼트**
- **활성(Active) 세그먼트**: 파티션당 하나만 존재. 새 레코드는 여기에 추가(append)된다
- **비활성(Inactive) 세그먼트**: 이미 롤(roll)된 세그먼트. 읽기만 가능
- [로그 압착](#로그-압착-log-compaction)과 보존 정책(Retention)은 비활성 세그먼트에만 적용된다

**세그먼트 롤링(Rolling) 조건** (하나라도 만족하면 새 세그먼트 생성)
```
1. 크기 초과:  log.segment.bytes         기본 1GB
2. 시간 초과:  log.roll.hours            기본 168시간(7일)
3. 인덱스 가득 참: log.index.size.max.bytes  기본 10MB
```

**희소 인덱스(Sparse Index)**
인덱스 파일은 모든 오프셋을 저장하지 않는다. `log.index.interval.bytes`(기본 4096) 바이트마다 하나의 인덱스 엔트리를 저장한다. 특정 오프셋 탐색 시 인덱스에서 가장 가까운 위치를 찾은 후, `.log` 파일을 순차 스캔한다.

```
[인덱스 탐색 과정]
1. offset=537 검색 요청
2. .index 파일에서 537보다 작은 가장 큰 인덱스 엔트리 찾기 → (512, position=8192)
3. .log 파일의 byte 8192부터 순차 스캔 → offset=537 레코드 발견
```

**설계 트레이드오프**
| 세그먼트 크기 | 장점 | 단점 |
|-------------|------|------|
| 크게 설정 | 세그먼트 롤링 빈도 낮음, I/O 효율적 | 삭제/압착 단위가 커짐 → 디스크 공간 낭비 증가 |
| 작게 설정 | 세밀한 보존 정책 적용 가능 | 파일 수 증가, 파일 핸들 과다 사용 |

**관련 개념**: [파티션](#파티션-partition), [로그 압착](#로그-압착-log-compaction)

### 리플리케이션 (Replication)

**정의**
파티션 데이터를 여러 브로커에 복제하는 메커니즘. 브로커 장애 시 데이터 유실을 방지하고 가용성을 높인다.

**왜 필요한가**
복제 없이 브로커 1대가 장애를 겪으면 해당 브로커의 파티션 데이터는 완전히 유실된다. `replication.factor=3`이면 3대의 브로커가 동시에 장애를 겪어야 데이터가 유실된다.

**동작 원리**
```
[replication.factor = 3, 파티션 0]

  Broker 1 (Leader)   Broker 2 (Follower)   Broker 3 (Follower)
  ┌──────────────┐    ┌──────────────┐      ┌──────────────┐
  │ Partition 0  │    │ Partition 0  │      │ Partition 0  │
  │ offset 0,1,2 │ ←─ │ offset 0,1,2 │ ←─   │ offset 0,1,2 │
  └──────────────┘    └──────────────┘      └──────────────┘
       ↑
  Producer/Consumer
  (모든 읽기/쓰기는 리더를 통해)
```

- **리더(Leader)**: 해당 파티션의 읽기/쓰기를 담당하는 단일 브로커
- **팔로워(Follower)**: 리더로부터 데이터를 Pull 방식으로 복제, 읽기 요청을 처리하지 않음 (기본 설정)
- **리더 선출**: 현재 리더가 장애를 겪으면 [ISR](#isr-in-sync-replicas) 내 팔로워가 새 리더로 선출됨

**주요 설정**
```
replication.factor         복제 수 (보통 3, 최소 2)
min.insync.replicas        쓰기 성공으로 인정되는 최소 동기화 레플리카 수
                           (보통 replication.factor - 1)
acks=all                   모든 ISR이 레코드를 받을 때까지 확인 대기
```

**설계 트레이드오프**
| replication.factor | 내구성 | 디스크 비용 | 네트워크 비용 |
|-------------------|--------|------------|--------------|
| 1 | 매우 낮음 | 1x | 낮음 |
| 2 | 중간 | 2x | 중간 |
| 3 | 높음 (권장) | 3x | 높음 |

**관련 개념**: [파티션](#파티션-partition), [ISR](#isr-in-sync-replicas)

### ISR (In-Sync Replicas)

**정의**
리더와 충분히 동기화된 레플리카(리더 포함)의 집합. 카프카는 ISR 내의 레플리카만 새 리더 후보로 간주한다.

**왜 필요한가**
모든 팔로워가 리더를 따라잡고 있다는 보장이 없다. 느린 팔로워를 리더로 선출하면 최근 메시지가 유실될 수 있다. ISR은 "충분히 최신 상태"인 레플리카만 리더 후보로 제한하여 데이터 안전성을 보장한다.

**동작 원리**
```
[ISR 판단 기준]
팔로워가 replica.lag.time.max.ms (기본 10초) 내에
리더의 로그 끝(LEO: Log End Offset)을 따라잡지 못하면 ISR에서 제외

[정상 상태]          [팔로워 지연 시]
ISR = {0, 1, 2}     ISR = {0, 1}  ← 브로커 2가 ISR에서 제외
Broker 0 (Leader)   Broker 0 (Leader)
Broker 1 (Follower) Broker 1 (Follower)
Broker 2 (Follower) Broker 2 (느린 팔로워, OSR)
```

- **HW(High Watermark)**: ISR의 모든 레플리카가 복제 완료한 가장 높은 오프셋. 컨슈머는 HW까지만 읽을 수 있다.
- **LEO(Log End Offset)**: 각 레플리카의 마지막 오프셋 + 1

```
[HW와 LEO]
Leader LEO:    offset 5
Follower1 LEO: offset 4
Follower2 LEO: offset 5

HW = min(ISR의 모든 LEO) = 4
컨슈머는 offset 3까지만 읽을 수 있음
```

**리더 장애 시나리오**
```
1. unclean.leader.election.enable = false (기본, 권장)
   → ISR 내에서만 리더 선출 → 데이터 유실 없음, but ISR이 비면 쓰기 불가(가용성 감소)

2. unclean.leader.election.enable = true
   → ISR 외 팔로워도 리더 선출 가능 → 가용성 증가, but 뒤처진 데이터 유실 가능
```

**설계 트레이드오프**
| 설정 | 데이터 안전성 | 가용성 |
|------|------------|--------|
| `replica.lag.time.max.ms` 짧게 | 엄격한 ISR → 안전 | 네트워크 일시 지연에도 ISR 제외 → 가용성 저하 |
| `replica.lag.time.max.ms` 길게 | 느슨한 ISR → 위험 | 가용성 향상 |
| `min.insync.replicas=2` + `acks=all` | 강력한 보장 | ISR 2개 미만 시 쓰기 실패 |

**관련 개념**: [리플리케이션](#리플리케이션-replication), [파티션](#파티션-partition)

### 로그 압착 (Log Compaction)

**정의**
같은 키를 가진 레코드 중 오래된 것을 삭제하고 각 키의 최신 값만 유지하는 보존 정책. `cleanup.policy=compact`로 설정한다.

**왜 필요한가**
```
[시간 기반 삭제(delete) 정책의 문제]
7일 후 오래된 세그먼트를 삭제
→ user_id=42의 현재 상태(최신 주소, 설정)를 7일이 지나면 조회 불가

[로그 압착으로 해결]
key=user_42의 이전 값 모두 삭제 → 최신 값만 유지
→ 디스크 무한 증가 없이 현재 상태 항상 조회 가능
```

**동작 원리**
```
[압착 전]                          [압착 후]
Key:  A  B  A  C  B  A  C         Key:  B  A  C
Val:  1  1  2  1  2  3  2         Val:  2  3  2
Off:  0  1  2  3  4  5  6         Off:  4  5  6
                                         ↑ 최신 값만 보존
```

- 압착은 **비활성 세그먼트**에만 수행된다 (활성 세그먼트는 압착 안됨)
- `log.cleaner.threads`만큼의 백그라운드 스레드가 주기적으로 압착 수행
- 압착 후에도 각 키의 마지막 레코드(값이 null인 묘비 레코드 포함)는 보존됨
- **묘비(Tombstone) 레코드**: 키의 값이 null인 레코드 → 해당 키 삭제를 의미. 압착 후 `delete.retention.ms`(기본 24시간) 동안 유지 후 완전 삭제

**용도**
- DB CDC(Change Data Capture): 각 행의 최신 상태를 카프카에 유지
- Kafka Streams 상태 저장소(State Store)의 changelog 토픽
- 설정(Config) 변경 이벤트: 최신 설정값만 필요

**설계 트레이드오프**
| 압착 | 장점 | 단점 |
|------|------|------|
| 키 기반 최신값 유지 | 영구적인 상태 조회 가능, 디스크 효율 | 이력(History) 조회 불가 |
| 백그라운드 작업 | 쓰기 성능에 영향 적음 | 압착 주기 동안 중복 데이터 존재 |
| CPU/IO 사용 | - | 압착 스레드가 리소스 소비 |

**delete vs compact 비교**
```
cleanup.policy=delete   → 오래된 세그먼트를 시간(log.retention.hours) 또는
                          크기(log.retention.bytes) 기준으로 통째로 삭제
cleanup.policy=compact  → 키 기반 최신값만 유지, 시간 기반 삭제 없음
cleanup.policy=compact,delete → 두 정책 동시 적용 (압착 + 오래된 세그먼트 삭제)
```
