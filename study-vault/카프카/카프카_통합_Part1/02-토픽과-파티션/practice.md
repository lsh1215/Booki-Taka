## ★ 기본

**Q1.** 카프카에서 토픽과 파티션의 관계를 설명하라. 토픽이 데이터를 저장하는 방식과, 파티션이 이 과정에서 어떤 역할을 하는지 구분하여 서술하라.

<details>
<summary>정답 보기</summary>

토픽은 데이터를 구분하는 **논리적 단위**이며, 파티션은 토픽을 구성하는 **물리적 단위**다.

토픽 자체는 데이터를 직접 저장하지 않는다. 토픽에 쓰인 메시지는 실제로는 해당 토픽의 파티션 중 하나에 저장된다. 파티션은 순서가 보장되는 불변 로그로, 메시지는 파티션의 끝에 append-only 방식으로 추가된다.

하나의 토픽은 하나 이상의 파티션으로 구성되며, 파티션들은 여러 브로커에 분산될 수 있다. 이를 통해 단일 토픽의 데이터가 여러 브로커에 나뉘어 저장되고, 병렬 처리가 가능해진다.

요약:
- 토픽 = 논리적 이름 (DB의 테이블과 유사)
- 파티션 = 실제 데이터가 저장되는 물리적 로그 단위
- 순서 보장 범위 = 파티션 내부 (토픽 전체가 아님)

</details>

**Q2.** 오프셋(Offset)은 파티션 내에서 어떻게 부여되는가? 아래 상황에서 컨슈머가 재시작 후 어느 오프셋부터 메시지를 읽을지 설명하라.

> Partition 0에 offset 0~9 메시지가 존재한다. 컨슈머가 offset 6까지 처리한 뒤 offset 6을 커밋하고 비정상 종료되었다.

<details>
<summary>정답 보기</summary>

오프셋은 파티션 내에서 0부터 시작하여 메시지가 추가될 때마다 1씩 단조 증가하는 고유 순번이다. 파티션 간 오프셋은 독립적이다.

컨슈머가 커밋한 오프셋은 `__consumer_offsets` 토픽에 저장된다. 커밋된 오프셋은 "이 오프셋까지 처리 완료"를 의미한다.

재시작 후 컨슈머는 커밋된 오프셋 **다음인 offset 7**부터 읽기를 재개한다.

- 커밋된 오프셋 6 = offset 6 포함 이전까지 처리 완료
- 다음 읽기 시작점 = 6 + 1 = offset 7
- offset 7, 8, 9 메시지를 순서대로 처리하게 됨

만약 offset 6을 커밋하기 전에 종료되었다면, `auto.offset.reset` 설정 또는 이전 커밋 위치에 따라 offset 0 또는 마지막으로 커밋된 오프셋 이후부터 재개된다.

</details>

**Q3.** 카프카 레코드에서 키(Key)가 있을 때와 없을 때 파티션 분배 방식이 어떻게 달라지는가? 각 방식의 목적을 함께 설명하라.

<details>
<summary>정답 보기</summary>

**키가 있을 때 (Key-based Partitioning)**

`murmur2 hash(key) % partition_count` 공식으로 파티션을 결정한다. 동일한 키는 항상 동일한 파티션으로 라우팅된다.

목적: 특정 엔티티(예: 사용자 ID, 주문 ID)의 이벤트를 같은 파티션에 모아 **순서를 보장**한다. 예를 들어 `user_id=42`를 키로 사용하면, 해당 사용자의 모든 이벤트가 하나의 파티션에 순서대로 저장된다.

**키가 없을 때 (Null Key)**

카프카 2.4 이전에는 라운드로빈(Round-Robin) 방식으로 파티션을 순환하며 분배했다. 2.4 이후에는 스티키 파티셔너(Sticky Partitioner)를 사용하여 배치가 가득 찰 때까지 같은 파티션에 메시지를 모은 뒤 다음 파티션으로 이동한다.

목적: 특정 순서가 필요 없을 때 모든 파티션에 **균등하게 부하를 분산**한다.

주의: 키를 사용할 경우, 특정 키에 메시지가 집중되면 해당 파티션에 핫스팟이 발생할 수 있다.

</details>

**Q4.** 카프카 토픽 이름 규칙에서 `.`과 `_`를 혼용하면 안 되는 이유를 설명하라.

<details>
<summary>정답 보기</summary>

카프카는 내부 메트릭 처리 시 토픽 이름의 `.`을 `_`로 변환한다. 이로 인해 `my.topic`과 `my_topic`은 내부적으로 동일한 이름으로 처리되어 충돌이 발생할 수 있다.

예를 들어 `orders.v2`라는 토픽이 존재할 때 `orders_v2` 토픽을 생성하려 하면, 카프카는 경고를 발생시키거나 예상치 못한 동작을 보일 수 있다.

따라서 토픽 이름에는 `.`과 `_` 중 하나만 일관되게 사용해야 한다. 팀 내에서 명명 규칙을 정할 때 이 제약을 반영하는 것이 중요하다. 권장 패턴인 `prd.marketing-team.sms-platform.json`처럼 구분자로 `.`만 사용하거나, 반대로 `-`와 `_`만 사용하는 방식으로 통일한다.

</details>

## ★★ 심화

**Q5.** 파티션 수를 결정할 때 고려해야 할 트레이드오프를 3가지 이상 서술하고, 파티션 수를 **늘린 후 줄일 수 없는** 이유를 설명하라.

<details>
<summary>정답 보기</summary>

**파티션 수 증가 시 트레이드오프**

이점:
1. **처리량 증가**: 파티션 수만큼 컨슈머를 병렬로 운영 가능. 최대 컨슈머 수 = 파티션 수.
2. **수평 확장**: 파티션을 여러 브로커에 분산하여 단일 브로커 병목 해소.

비용:
1. **파일 핸들 증가**: 파티션당 여러 세그먼트 파일(.log, .index, .timeindex)이 생성되어 OS 파일 핸들 수가 증가한다.
2. **리더 선출 시간 증가**: 브로커 장애 시 컨트롤러가 모든 영향받은 파티션의 리더를 재선출해야 하므로, 파티션 수가 많을수록 복구 시간이 늘어난다.
3. **메모리 사용 증가**: 브로커는 파티션마다 버퍼를 유지하므로 메모리 사용량이 비례하여 증가한다.
4. **end-to-end 지연 증가 가능성**: 파티션이 많아지면 배치 처리 효율이 낮아질 수 있다.

**줄일 수 없는 이유**

파티션을 줄이면 기존에 그 파티션에 저장된 데이터를 어디로 이동할지 결정할 수 없다. 카프카는 파티션 데이터의 병합(merge)이나 재배치를 지원하지 않는다.

또한 키 기반 파티셔닝을 사용하는 경우, 파티션 수가 변경되면 `hash(key) % partition_count`의 결과가 달라져 동일한 키가 이전과 다른 파티션으로 라우팅된다. 기존 데이터와의 일관성이 깨지므로 안전하게 줄이는 방법이 없다.

파티션 수를 줄이려면 토픽을 삭제하고 원하는 파티션 수로 재생성해야 한다.

</details>

**Q6.** 세그먼트 롤링(Segment Rolling)이 발생하는 세 가지 조건을 설명하고, 인덱스 파일이 희소 인덱스(Sparse Index)인 이유와 이로 인해 특정 오프셋을 탐색하는 방법을 서술하라.

<details>
<summary>정답 보기</summary>

**세그먼트 롤링 조건**

현재 활성 세그먼트에서 다음 세 조건 중 하나라도 만족하면 새 세그먼트가 생성된다.

1. **크기 초과** (`log.segment.bytes`, 기본 1GB): 활성 세그먼트의 .log 파일 크기가 설정값을 초과하면 롤링.
2. **시간 초과** (`log.roll.hours`, 기본 168시간/7일): 세그먼트가 생성된 후 설정된 시간이 지나면 롤링. 오래된 데이터를 보존 정책으로 삭제할 수 있게 해주는 역할도 한다.
3. **인덱스 파일 가득 참** (`log.index.size.max.bytes`, 기본 10MB): .index 또는 .timeindex 파일이 가득 차면 롤링.

**희소 인덱스인 이유**

모든 오프셋에 대한 인덱스 엔트리를 저장하면 인덱스 파일 자체가 지나치게 커진다. 카프카는 `log.index.interval.bytes`(기본 4096) 바이트마다 하나의 인덱스 엔트리를 저장하는 희소(Sparse) 방식을 사용한다. 이로써 인덱스 파일을 메모리에 전부 올릴 수 있어 탐색 속도를 높이면서도 디스크 공간을 절약한다.

**특정 오프셋 탐색 방법**

1. .index 파일을 읽어, 찾으려는 오프셋보다 크지 않은 가장 큰 인덱스 엔트리를 이진 탐색으로 찾는다.
2. 해당 인덱스 엔트리가 가리키는 .log 파일의 바이트 위치로 이동한다.
3. 그 위치부터 .log 파일을 순차적으로 스캔하여 원하는 오프셋을 가진 레코드를 찾는다.

예: offset=537 검색 시 인덱스에서 (offset=512, position=8192) 엔트리를 찾고, .log 파일의 8192바이트 위치부터 스캔하여 offset=537 레코드를 찾는다.

</details>

**Q7.** ISR(In-Sync Replicas)에서 팔로워가 제외되는 조건은 무엇인가? ISR과 `min.insync.replicas`, `acks=all` 설정이 함께 작동하는 방식을 설명하고, ISR이 1개만 남은 상황에서 `min.insync.replicas=2`일 때 어떤 일이 발생하는지 서술하라.

<details>
<summary>정답 보기</summary>

**팔로워가 ISR에서 제외되는 조건**

팔로워가 `replica.lag.time.max.ms`(기본 10초) 이내에 리더의 LEO(Log End Offset)를 따라잡지 못하면 ISR에서 제외된다. 네트워크 지연, 팔로워 GC 일시 정지, 과부하 등으로 복제가 지연될 때 발생한다.

**ISR + min.insync.replicas + acks=all 동작**

- `acks=all`: 프로듀서가 보낸 메시지를 ISR에 속한 모든 레플리카가 수신 확인해야 프로듀서에게 성공 응답을 반환한다.
- `min.insync.replicas=2`: 쓰기 성공으로 인정되려면 ISR에 최소 2개의 레플리카가 있어야 한다.

이 두 설정은 함께 작동하여 최소 2개의 레플리카가 데이터를 복제했을 때만 쓰기를 성공으로 처리한다. 이는 1개의 브로커가 장애를 겪어도 데이터가 적어도 1개의 다른 레플리카에 남아 있음을 보장한다.

**ISR이 1개만 남은 상황**

ISR이 리더 1개만 남고 `min.insync.replicas=2`인 경우:

브로커는 프로듀서의 쓰기 요청에 `NotEnoughReplicasException`을 반환한다. 즉 **쓰기가 실패**한다. 이는 가용성을 희생하고 데이터 안전성을 선택하는 설계다.

이 상태에서 쓰기를 강제로 허용하려면 `unclean.leader.election.enable=true`로 설정하거나 `min.insync.replicas`를 낮춰야 하는데, 두 방법 모두 데이터 유실 위험을 수반한다.

</details>

**Q8.** 로그 압착(Log Compaction)과 시간 기반 삭제(Log Deletion, `cleanup.policy=delete`)의 차이를 설명하고, 각각이 적합한 사용 사례를 2가지씩 제시하라. 또한 묘비(Tombstone) 레코드가 무엇인지 설명하라.

<details>
<summary>정답 보기</summary>

**로그 압착 vs 시간 기반 삭제**

| 구분 | Log Compaction (`compact`) | Log Deletion (`delete`) |
|------|--------------------------|------------------------|
| 삭제 단위 | 키 기반 - 같은 키의 이전 값 삭제 | 세그먼트 단위 - 오래된 세그먼트 통째로 삭제 |
| 보존 기준 | 각 키의 최신 값 영구 보존 | 시간(`log.retention.hours`) 또는 크기(`log.retention.bytes`) |
| 이력 조회 | 불가 (최신값만 남음) | 보존 기간 내 가능 |
| 적용 대상 | 비활성 세그먼트 | 보존 기간 초과한 비활성 세그먼트 |

**Log Compaction 적합 사례**
1. **DB CDC(Change Data Capture)**: 각 행의 최신 상태(현재 주소, 설정값 등)만 필요하고, 영구적으로 조회 가능해야 할 때.
2. **Kafka Streams 상태 저장소 Changelog**: 상태 복원 시 각 키의 최신 상태만 필요하므로 압착이 효율적.

**Log Deletion 적합 사례**
1. **애플리케이션 로그**: 최근 7일치 로그만 분석에 사용하고 이후 데이터는 불필요한 경우.
2. **실시간 메트릭/모니터링 이벤트**: 최신 데이터만 소비하며, 오래된 데이터는 가치가 없는 경우.

**묘비(Tombstone) 레코드**

키는 존재하지만 값(Value)이 `null`인 레코드다. 로그 압착 시 해당 키를 완전히 삭제하라는 신호로 사용된다.

동작: 압착 중 특정 키의 가장 최신 레코드가 묘비 레코드이면, 압착 후에도 그 묘비 레코드는 `delete.retention.ms`(기본 24시간) 동안 유지된다. 이는 컨슈머가 해당 키의 삭제 사실을 인지할 시간을 주기 위함이다. 이후 묘비 레코드도 완전히 삭제된다.

</details>

## ★★★ 고급

**Q9.** 다음 시나리오를 읽고 발생하는 문제와 원인, 해결책을 서술하라.

> 전자상거래 서비스에서 주문 이벤트를 처리하는 토픽 `orders`를 `replication.factor=3`, `min.insync.replicas=2`, `acks=all`로 운영 중이다. 피크 타임에 Broker 2, Broker 3이 동시에 네트워크 지연을 겪으며 ISR에서 제외되었다. Broker 1만 ISR에 남은 상태에서 프로듀서가 메시지를 전송하려 하고 있다.
>
> (1) 현재 상황에서 어떤 일이 발생하는가?
> (2) 운영자가 즉시 선택할 수 있는 두 가지 대응 방안과 각각의 리스크를 설명하라.
> (3) 이 상황을 사전에 예방하기 위한 설계 방안을 제시하라.

<details>
<summary>정답 보기</summary>

**(1) 현재 상황**

ISR에 Broker 1만 남아 있고 `min.insync.replicas=2`이므로, ISR 수(1)가 최솟값(2)보다 적다. 브로커는 프로듀서의 쓰기 요청에 `NotEnoughReplicasException`을 반환한다. 프로듀서는 `retries` 설정에 따라 재시도하지만 ISR이 회복되지 않으면 최종적으로 쓰기 실패가 발생한다. 주문 이벤트가 카프카에 저장되지 않아 서비스 장애로 이어질 수 있다.

**(2) 즉각적 대응 방안**

**방안 A: min.insync.replicas를 1로 임시 낮추기**

```bash
kafka-configs.sh --bootstrap-server broker:9092 \
  --entity-type topics --entity-name orders \
  --alter --add-config min.insync.replicas=1
```

- 효과: 즉시 쓰기 가능. ISR이 1개여도 메시지 수용.
- 리스크: Broker 1마저 장애를 겪으면 커밋되었지만 복제되지 않은 데이터 유실. 프로듀서는 성공으로 인식했지만 실제로는 소실될 수 있음.

**방안 B: 네트워크 지연 해소 후 ISR 자연 복구 대기**

- 효과: 설정 변경 없이 안전하게 ISR 복구를 기다림. 복구 후 정상화.
- 리스크: 복구까지 걸리는 시간 동안 쓰기 불가. 서비스 다운타임 발생. `replica.lag.time.max.ms` 이상 지연이 지속되면 OSR 팔로워가 따라잡는 데 시간이 더 걸림.

**(3) 사전 예방 설계**

1. **replication.factor를 브로커 수와 여유롭게 설정**: 브로커 3대 클러스터에서 `replication.factor=3`, `min.insync.replicas=2`는 1대 장애를 허용한다. 브로커를 5대로 늘리면 2대 장애까지 허용 가능.

2. **브로커 분리**: Broker 2, 3이 동일 네트워크 스위치나 랙에 있어 동시 지연이 발생한 것이라면, 레플리카를 서로 다른 랙(rack)에 배치하는 `broker.rack` 설정을 적용한다.

3. **모니터링 알람 설정**: `kafka.server:type=ReplicaManager,name=UnderReplicatedPartitions` 메트릭이 0보다 커지면 즉시 알람. ISR 제외 전에 조기 개입 가능.

4. **프로듀서 측 재시도 버퍼**: 프로듀서에 충분한 `buffer.memory`와 `max.block.ms`를 설정하여 일시적 ISR 부족 시 메시지를 버퍼에 유지하고, ISR 복구 후 재전송하도록 한다.

5. **`min.insync.replicas` 조정**: 가용성이 내구성보다 중요한 토픽은 `min.insync.replicas=1`로 설정하되, 중요 토픽과 분리하여 운영.

</details>

**Q10.** 아래 조건에서 최적의 파티션 수를 추론하고, 키 기반 라우팅 사용 시 주의해야 할 두 가지 문제점과 해결 방법을 서술하라.

> - 목표 처리량: 500 MB/s
> - 프로듀서 1개 인스턴스의 파티션당 쓰기 속도: 50 MB/s
> - 컨슈머 1개 인스턴스의 파티션당 읽기 속도: 25 MB/s
> - 메시지 키: 상품 카테고리 ID (총 10개 카테고리, 일부 카테고리가 전체 트래픽의 70% 차지)

<details>
<summary>정답 보기</summary>

**최적 파티션 수 추론**

기본 공식을 적용한다.

```
필요 파티션 수 = max(
  목표_처리량 / 프로듀서_처리량,
  목표_처리량 / 컨슈머_처리량
)
= max(500/50, 500/25)
= max(10, 20)
= 20개
```

컨슈머 처리 속도가 병목이므로 최소 20개의 파티션이 필요하다. 여기에 향후 트래픽 증가와 브로커 장애 시 여유를 고려하여 24~30개로 설정하는 것이 현실적이다. 단, 파티션 수는 브로커 수의 배수로 설정하면 파티션이 균등하게 분산된다.

**키 기반 라우팅 시 주의 문제점**

**문제 1: 파티션 핫스팟(Hotspot)**

카테고리 ID를 키로 사용 시, 전체 트래픽의 70%를 차지하는 일부 카테고리가 해시 결과에 따라 특정 파티션으로 집중될 수 있다. 해당 파티션의 리더 브로커가 과부하를 겪고, 그 파티션을 담당하는 컨슈머만 처리 지연이 발생한다. 파티션을 추가해도 핫스팟 카테고리가 여전히 같은 파티션으로 해시될 수 있어 근본적 해결이 어렵다.

해결 방법:
- 핫스팟 키에 접미사를 붙여 여러 파티션으로 분산: `category_id=5` → `5-0`, `5-1`, `5-2` 처럼 3개의 가상 키로 분리
- 단, 이 경우 동일 카테고리의 메시지가 여러 파티션에 분산되어 순서 보장이 깨지므로, 순서가 필요 없는 집계 처리에만 적합
- 또는 카테고리 ID 대신 상품 ID처럼 더 세분화된 키를 사용하여 해시 분포를 균등하게 만든다

**문제 2: 파티션 수 변경 시 라우팅 불일치**

현재 20개 파티션에서 나중에 24개로 늘리면, `hash(key) % 20`과 `hash(key) % 24`의 결과가 달라진다. 동일한 카테고리 ID가 이전과 다른 파티션으로 라우팅된다. 컨슈머가 파티션 단위로 상태를 관리하거나 순서에 의존한다면 데이터 일관성이 깨진다.

해결 방법:
- 파티션 수를 처음부터 충분히 크게 설정하여 나중에 변경할 필요를 최소화한다
- 파티션 수 변경이 불가피한 경우, 기존 토픽을 유지하면서 새 파티션 수의 토픽을 생성하고, 마이그레이션 기간 동안 두 토픽을 병행 운영한다
- 파티션 수 변경 시 컨슈머가 상태를 초기화하거나 재처리를 수행하는 절차를 사전에 정의해 둔다

</details>
