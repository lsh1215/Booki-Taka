# 핵심 아키텍처

> 출처: 카프카 핵심 가이드 Ch1·Ch6, 실전카프카 Ch1·Ch3, 아파치 카프카 애플리케이션 프로그래밍 Ch1·Ch3
> 태그: #broker #cluster #zookeeper #kraft #controller #distributed-system

## 이 토픽의 근본 문제

대규모 실시간 데이터를 안정적으로 수집/전달하려면 어떤 아키텍처가 필요한가.

프로듀서와 컨슈머가 직접 통신하면 강결합이 발생하고, 시스템 중 하나가 느려지면 전체가 영향을 받는다. 수십 개의 시스템이 서로 데이터를 주고받아야 하는 환경에서는 연결 수가 N² 로 증가해 관리 불가능 상태가 된다. 카프카는 이 문제를 브로커 클러스터를 중간에 두는 방식으로 해결한다.

## 전체 구조

```
                    ┌─────────────────────────────────────────┐
                    │           카프카 클러스터                  │
                    │                                         │
  Producer 1 ──────►│  Broker 1 (Controller)                  │──────► Consumer Group A
                    │    ├─ Partition 0 (Leader)               │
  Producer 2 ──────►│    └─ Partition 1 (Follower)             │──────► Consumer Group B
                    │                                         │
  Producer 3 ──────►│  Broker 2                               │
                    │    ├─ Partition 0 (Follower)             │
                    │    └─ Partition 1 (Leader)               │
                    │                                         │
                    │  Broker 3                               │
                    │    └─ Partition 0 (Follower)             │
                    └─────────────────────────────────────────┘
                                      │
                          ┌───────────┴───────────┐
                          │                       │
                   ZooKeeper 앙상블           (또는 KRaft)
                   (메타데이터 관리)          (내장 Raft 합의)
```

## 핵심 개념

### 브로커 (Broker)

**정의**: 카프카 클러스터를 구성하는 개별 서버 프로세스. 프로듀서로부터 메시지를 수신하여 디스크에 저장하고, 컨슈머의 요청에 응답해 메시지를 전달한다.

**왜 필요한가**: 브로커가 없으면 프로듀서와 컨슈머가 직접 연결되어야 한다. 이 경우 N개의 프로듀서와 M개의 컨슈머 사이에 N×M 개의 연결이 생긴다. 한쪽이 느려지면 다른 쪽도 영향을 받는 강결합 구조가 되고, 처리 속도 차이를 흡수할 버퍼가 없어 전체 파이프라인이 가장 느린 컴포넌트 속도로 제한된다.

**동작 원리**:
1. 프로듀서가 특정 토픽 파티션의 리더 브로커에게 메시지 전송
2. 리더 브로커가 메시지를 로컬 로그(디스크)에 순차 기록
3. 팔로워 브로커들이 리더로부터 메시지를 복제 (풀 방식)
4. ISR(In-Sync Replica) 에 속한 팔로워들이 복제를 완료하면 커밋 처리
5. 컨슈머가 리더 브로커에게 오프셋 기반으로 메시지 요청 → 응답

**설계 트레이드오프**:
- Stateless 설계: 오프셋 관리를 컨슈머에게 위임 → 브로커 자체는 단순해지고 스케일아웃이 용이하지만, 컨슈머가 오프셋을 잘못 관리하면 중복 소비나 메시지 누락 가능
- 페이지 캐시 의존: JVM 힙 대신 OS 페이지 캐시를 활용 → GC 정지 없이 대용량 데이터 처리 가능하지만, 브로커 재시작 시 캐시가 초기화되어 초반 성능 저하 발생
- 디스크 순차 쓰기: 랜덤 I/O 대신 항상 로그 끝에 추가(append-only) → HDD에서도 높은 쓰기 처리량 달성, 대신 특정 메시지 수정/삭제는 불가

**관련 개념**: [클러스터](#클러스터-cluster), [컨트롤러](#컨트롤러-controller)

### 클러스터 (Cluster)

**정의**: 동일한 카프카 서비스를 제공하는 여러 브로커의 논리적 그룹. 하나의 클러스터는 컨트롤러 역할을 하는 브로커를 포함하며, 공통의 메타데이터(브로커 목록, 토픽/파티션 정보, 복제 구성)를 공유한다.

**왜 필요한가**: 단일 브로커만 운영하면 단일 장애점(SPOF)이 생겨 브로커가 장애를 겪는 순간 전체 서비스가 중단된다. 또한 단일 서버의 디스크 용량과 네트워크 대역폭 한계로 처리량이 제한된다. 클러스터는 수평 확장을 통해 이 두 문제를 동시에 해결한다.

**동작 원리**:
- 수평 확장(Scale-out): 브로커를 추가하면 파티션이 새 브로커에 분산되어 처리량이 증가
- 복제(Replication): 각 파티션은 복제 인수(replication.factor)만큼 복수의 브로커에 저장 → 일부 브로커 장애 시에도 데이터 유실 없이 서비스 지속
- 파티션 분산: 토픽의 파티션들이 여러 브로커에 균등하게 분산되어 부하가 나뉨

**설계 트레이드오프**:
- 브로커 수 증가 → 클러스터 내 복제 트래픽이 증가하고, 컨트롤러의 메타데이터 관리 부하도 증가
- 복제 인수 높임 → 내구성은 높아지지만 디스크 사용량과 복제 네트워크 트래픽이 그 배수로 증가
- 파티션 수 증가 → 병렬 처리량은 늘지만 리더 선출 오버헤드와 파일 핸들 수 증가

**관련 개념**: [브로커](#브로커-broker), [컨트롤러](#컨트롤러-controller)

### 주키퍼 (ZooKeeper) / KRaft

**정의**: 카프카 클러스터의 메타데이터를 관리하고 분산 합의(consensus)를 제공하는 외부 또는 내장 시스템.

**ZooKeeper 방식**:
- 카프카 클러스터 외부에서 동작하는 별도의 분산 코디네이션 서비스
- 브로커 등록/탈퇴, 컨트롤러 선출, 토픽 설정 등 클러스터 메타데이터를 ZNode 계층 구조로 저장
- 카프카와 별도로 ZK 앙상블(최소 3대 권장)을 운영해야 하므로 인프라 복잡도 증가
- 모든 브로커가 ZK를 직접 바라보는 구조 → ZK가 병목이 되거나, ZK 장애 시 클러스터 전체 영향

**KRaft(Kafka Raft) 방식**:
- 카프카 2.8부터 프리뷰, 3.3부터 프로덕션 사용 가능 (4.0에서 ZK 완전 제거)
- 카프카 브로커 자체에 Raft 합의 알고리즘을 내장하여 메타데이터를 내부 토픽(`__cluster_metadata`)으로 관리
- 별도 ZK 앙상블 불필요 → 운영 단순화, 컨테이너 환경 배포 간소화
- 컨트롤러 쿼럼(3~5대의 컨트롤러 지정 브로커)이 Raft로 리더를 선출하고 메타데이터 변경을 처리

**설계 트레이드오프**:
- ZooKeeper: 수년간 검증된 성숙한 시스템, 카프카 생태계의 풍부한 운영 경험. 단, 외부 의존성, 별도 운영 비용, 대규모 파티션 환경에서 메타데이터 동기화 병목 발생
- KRaft: ZK 의존성 제거로 운영 단순화, 메타데이터 변경 속도 향상, 컨트롤러 페일오버 시간 단축. 단, 상대적으로 짧은 프로덕션 운영 이력

**관련 개념**: [컨트롤러](#컨트롤러-controller), [클러스터](#클러스터-cluster)

### 컨트롤러 (Controller)

**정의**: 클러스터 내 하나의 브로커에 부여되는 특수 역할. 파티션 리더 선출, 브로커 장애 감지, 메타데이터 변경 조율을 담당한다. 일반 브로커 역할과 컨트롤러 역할을 동시에 수행한다.

**왜 필요한가**: 파티션 리더 브로커가 장애를 겪었을 때, "누가 새 리더를 선출할 것인가"를 결정하는 주체가 없으면 클러스터가 일관성 없는 상태에 빠진다. 컨트롤러는 이 메타데이터 변경 작업을 직렬화하여 처리함으로써 클러스터 전체의 일관성을 보장한다.

**동작 원리 (ZooKeeper 모드)**:
1. 브로커 시작 시 ZK에 `/controller` 노드 등록 경쟁 → 먼저 등록한 브로커가 컨트롤러
2. 컨트롤러는 ZK의 브로커 목록(`/brokers/ids`) 변화를 감시(watch)
3. 브로커 이탈 감지 → 해당 브로커가 리더인 파티션 목록 확인
4. ISR(In-Sync Replica) 목록에서 새 리더를 선택 → 모든 브로커에 LeaderAndIsr 요청 전파
5. 컨트롤러 자신이 장애를 겪으면 ZK 세션 만료 → 다른 브로커가 `/controller` 노드 재등록 경쟁

**동작 원리 (KRaft 모드)**:
1. 사전에 지정된 컨트롤러 쿼럼 브로커들이 Raft 프로토콜로 액티브 컨트롤러를 선출
2. 액티브 컨트롤러가 메타데이터 변경 사항을 `__cluster_metadata` 토픽에 기록
3. 모든 브로커가 해당 토픽을 팔로우하여 최신 메타데이터를 유지
4. 컨트롤러 페일오버 시 Raft 재선출로 수 초 내 복구 (ZK 모드 대비 빠름)

**설계 트레이드오프**:
- 컨트롤러가 단일 노드이므로 메타데이터 변경 처리가 직렬화됨 → 대규모 클러스터에서 파티션 리더 선출 폭풍(rebalance storm) 발생 시 병목 가능
- KRaft에서는 컨트롤러 쿼럼을 통해 이 단일 장애점을 완화

**관련 개념**: [주키퍼/KRaft](#주키퍼-zookeeper--kraft), [클러스터](#클러스터-cluster)

### 분산 시스템 설계

**카프카가 높은 처리량을 달성하는 4가지 핵심 설계**:

**1. 페이지 캐시 활용**
- 브로커는 JVM 힙 메모리 대신 OS가 관리하는 페이지 캐시(파일 시스템 캐시)에 의존
- 메시지 쓰기 시 디스크가 아닌 페이지 캐시에 먼저 기록 → OS가 백그라운드에서 디스크에 플러시
- 최근 메시지 읽기는 디스크 접근 없이 페이지 캐시에서 직접 서빙
- 효과: JVM GC 정지 없음, 브로커 재시작 후에도 OS 캐시는 유지됨 (프로세스 재시작과 캐시는 독립)
- 주의: 브로커에 JVM 힙을 과도하게 할당하면 페이지 캐시로 쓸 수 있는 메모리가 줄어 역효과

**2. 배치 전송 (Batch Processing)**
- 프로듀서는 메시지를 개별 전송하지 않고 `batch.size`와 `linger.ms` 설정에 따라 묶어서 전송
- 브로커도 컨슈머에게 응답 시 단건이 아닌 최대 `fetch.max.bytes`만큼 묶어서 응답
- 효과: 네트워크 라운드트립 감소, 요청당 오버헤드(헤더, 직렬화 비용) 분산
- 트레이드오프: `linger.ms`를 높이면 처리량은 증가하지만 개별 메시지의 종단간 지연시간(latency) 증가

**3. 제로 카피 (Zero-Copy)**
- 일반 파일 전송: 디스크 → 커널 버퍼 → 유저 공간 → 소켓 버퍼 → 네트워크 (4번 복사)
- 카프카 제로 카피: `sendfile()` 시스템 콜로 커널 공간에서 디스크 → 소켓 버퍼 직접 전송 (2번 복사)
- 효과: CPU 사용률 감소, 메모리 복사 횟수 감소 → 컨슈머가 많아도 브로커 CPU 부하가 선형적으로 증가하지 않음
- 제약: SSL/TLS 암호화 적용 시 유저 공간에서 암호화 처리가 필요해 제로 카피 불가

**4. 순차 디스크 I/O**
- 카프카 로그는 append-only 구조 → 항상 파일 끝에만 쓰기
- HDD의 랜덤 I/O(~100 IOPS)와 달리 순차 쓰기는 수백 MB/s 처리량 달성 가능
- SSD 대비 저렴한 HDD로도 높은 쓰기 성능 확보

**트레이드오프 요약**:

| 설계 | 처리량 | 지연시간 | 복잡도 |
|------|--------|----------|--------|
| 페이지 캐시 | 높음 | 낮음 | OS 의존 |
| 배치 전송 | 높음 | 증가 | 설정 조율 필요 |
| 제로 카피 | 높음 | 낮음 | TLS 불가 |
| 순차 I/O | 높음 | 낮음 | 수정/삭제 불가 |

### 요청 처리 모델

**아키텍처**: 네트워크 스레드풀과 IO 스레드풀을 분리하는 Reactor 패턴

```
Client
  │
  ▼
[Acceptor Thread]
  │  (연결 수락 후 네트워크 스레드에 배정)
  ▼
[Network Thread Pool] ─── num.network.threads (기본값: 3)
  │  클라이언트 요청 수신 → 요청 큐(Request Queue)에 삽입
  │  응답 큐(Response Queue)에서 꺼내 클라이언트에게 응답
  ▼
[Request Queue]
  │
  ▼
[IO Thread Pool] ─── num.io.threads (기본값: 8)
  │  요청 큐에서 꺼내 실제 처리 (디스크 읽기/쓰기, 복제 등)
  │  처리 완료 후 응답 큐에 삽입
  ▼
[Response Queue]
```

**각 스레드의 역할**:
- Acceptor Thread: TCP 연결 수락, 네트워크 스레드에 연결 배정 (1개)
- Network Thread: 소켓에서 바이트 읽기/쓰기, 요청 파싱, 응답 직렬화 (입출력 분리)
- IO Thread: 요청의 실제 처리 담당. 디스크 I/O, 메시지 유효성 검증, ISR 확인, 오프셋 업데이트

**요청 유형**:
- Produce Request: 프로듀서가 메시지를 쓸 때
- Fetch Request: 컨슈머 또는 팔로워 브로커가 메시지를 읽을 때
- Metadata Request: 클라이언트가 파티션 리더 정보를 조회할 때
- LeaderAndIsr Request: 컨트롤러가 리더 변경을 브로커에 전파할 때

**주요 설정**:
- `num.network.threads`: 클라이언트 수가 많을 때 증가 (기본값 3)
- `num.io.threads`: 디스크 처리가 병목일 때 증가 (기본값 8, 디스크 수에 비례 권장)
- `queued.max.requests`: 요청 큐 최대 크기. 초과 시 네트워크 스레드가 블락됨

**설계 트레이드오프**:
- 네트워크/IO 분리로 디스크 I/O 지연이 네트워크 처리를 블락하지 않음
- 단, 요청 큐가 쌓이면 지연시간이 급증 → 큐 길이 모니터링 필수
- IO 스레드 수를 과도하게 늘리면 디스크 경합 발생 가능
