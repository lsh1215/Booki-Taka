### Q1. 브로커의 역할

카프카 브로커가 메시지를 처리하는 순서를 올바르게 나열한 것은?

1. 프로듀서로부터 메시지 수신 → 메모리(JVM 힙)에 저장 → 컨슈머 요청 시 힙에서 전달
2. 프로듀서로부터 메시지 수신 → 디스크에 순차 기록 → 컨슈머 요청 시 전달
3. 프로듀서로부터 메시지 수신 → 컨슈머에게 즉시 전달 → 비동기로 디스크에 기록
4. 프로듀서로부터 메시지 수신 → 컨트롤러에 전달 → 컨트롤러가 컨슈머에게 분배

<details><summary>정답 보기</summary>

**정답: 2**

브로커는 프로듀서로부터 메시지를 수신하면 로그 파일에 순차적으로 기록(append)한다. 컨슈머가 오프셋과 함께 읽기 요청을 보내면 브로커는 해당 오프셋부터 메시지를 응답한다.

- 1번 오답: 브로커는 JVM 힙이 아닌 OS 페이지 캐시를 활용한다. JVM 힙에 메시지를 저장하면 GC 정지가 발생해 처리량이 급감한다.
- 3번 오답: 브로커는 컨슈머에게 즉시 푸시하지 않는다. 컨슈머가 브로커에게 풀(pull) 방식으로 요청한다.
- 4번 오답: 컨트롤러는 데이터 흐름에 개입하지 않는다. 컨트롤러는 메타데이터 관리(리더 선출 등)만 담당한다.

</details>

### Q2. 페이지 캐시

카프카 브로커가 JVM 힙 대신 OS 페이지 캐시를 활용하는 이유로 가장 적절한 것은?

1. 페이지 캐시는 JVM 힙보다 더 많은 용량을 제공하기 때문에
2. 페이지 캐시를 사용하면 JVM GC 정지 없이 대용량 데이터를 처리할 수 있기 때문에
3. 페이지 캐시는 브로커 재시작 후에도 데이터가 영구적으로 보존되기 때문에
4. 페이지 캐시에 저장된 데이터는 암호화가 자동으로 적용되기 때문에

<details><summary>정답 보기</summary>

**정답: 2**

JVM 힙에 대용량 데이터를 올려두면 GC가 발생할 때마다 수백 밀리초에서 수 초 단위의 정지(stop-the-world)가 일어난다. OS 페이지 캐시는 JVM 관리 밖에 있어 GC 영향을 받지 않는다. 또한 최근에 쓴 메시지는 페이지 캐시에 남아 있어 디스크 접근 없이 빠르게 서빙된다.

- 1번 오답: 용량 자체가 핵심 이유가 아니다. JVM 힙도 서버 메모리 한도 내에서 크게 설정할 수 있다. 핵심은 GC 영향 여부다.
- 3번 오답: 페이지 캐시는 휘발성 메모리 기반이다. 서버 전원이 꺼지거나 재부팅되면 소실된다. 단, 브로커 프로세스만 재시작하면 OS는 살아있으므로 캐시가 유지된다는 점은 맞다.
- 4번 오답: 페이지 캐시는 암호화 기능을 제공하지 않는다.

</details>

### Q3. 컨트롤러의 주요 역할

카프카 컨트롤러가 수행하는 핵심 역할을 모두 고른 것은?

가. 파티션 리더 선출
나. 브로커 장애 감지
다. 프로듀서가 보낸 메시지의 유효성 검증
라. 메타데이터 변경 사항을 모든 브로커에 전파

1. 가, 나
2. 가, 나, 라
3. 나, 다, 라
4. 가, 나, 다, 라

<details><summary>정답 보기</summary>

**정답: 2**

컨트롤러는 클러스터 메타데이터를 관리하는 역할을 담당한다.

- 가(O): 브로커 장애 시 해당 브로커가 리더인 파티션에 대해 ISR 목록 기반으로 새 리더를 선출한다.
- 나(O): ZooKeeper 모드에서는 ZK 세션 만료를 감지하고, KRaft 모드에서는 브로커 상태를 직접 추적한다.
- 다(X): 메시지 유효성 검증은 IO 스레드가 담당한다. 컨트롤러는 데이터 경로(data plane)에 개입하지 않는다.
- 라(O): 리더 변경 등 메타데이터 변경이 발생하면 LeaderAndIsr 요청을 통해 모든 브로커에 전파한다.

</details>

### Q4. ZooKeeper와 KRaft 비교

다음 중 KRaft 모드가 ZooKeeper 모드와 비교해 개선한 사항으로 옳은 것은?

1. 카프카 브로커 수를 무제한으로 늘릴 수 있게 되었다.
2. 외부 코디네이션 시스템 의존성을 제거하여 운영 복잡도가 낮아졌다.
3. 브로커 장애 시 데이터 손실이 발생하지 않는 무결성이 보장된다.
4. 컨슈머 오프셋 관리가 브로커 내부에서 자동으로 처리된다.

<details><summary>정답 보기</summary>

**정답: 2**

KRaft는 카프카 내부에 Raft 합의 알고리즘을 구현하여 메타데이터 관리를 카프카 자체적으로 처리한다. 별도로 ZooKeeper 앙상블을 구축하고 운영할 필요가 없어져 인프라 구성과 운영이 단순해진다.

- 1번 오답: 브로커 수 제한은 ZK/KRaft 선택과 직접적인 관련이 없다. 실제로 KRaft는 대규모 파티션 처리가 더 효율적이지만, "무제한"은 사실이 아니다.
- 3번 오답: 데이터 내구성은 복제 인수(replication.factor)와 `acks` 설정으로 결정된다. KRaft 전환이 데이터 무결성을 보장하지는 않는다.
- 4번 오답: 컨슈머 오프셋 관리는 `__consumer_offsets` 내부 토픽에 저장되며, ZK/KRaft 모드와 관계없이 동일하게 동작한다.

</details>

### Q5. 클러스터 확장 시나리오

현재 3개의 브로커로 구성된 카프카 클러스터에 브로커를 3개 추가하여 총 6개로 확장했다. 다음 중 이 변경으로 즉각적으로 발생하는 일은?

1. 기존 토픽의 파티션들이 6개 브로커에 자동으로 재분산된다.
2. 신규 브로커가 클러스터에 등록되지만 기존 파티션은 자동으로 이동하지 않는다.
3. 컨트롤러가 새로운 브로커를 감지하고 즉시 리밸런싱을 시작한다.
4. 신규 브로커 추가 후 기존 토픽의 복제 인수가 자동으로 높아진다.

<details><summary>정답 보기</summary>

**정답: 2**

카프카는 브로커를 추가해도 기존 파티션을 자동으로 재분산하지 않는다. 신규 브로커는 클러스터에 등록되고 이후 새로 생성되는 토픽의 파티션은 신규 브로커에도 배정된다. 기존 파티션을 신규 브로커로 이동하려면 `kafka-reassign-partitions.sh` 도구를 사용하거나 Cruise Control 같은 자동화 도구를 별도로 운영해야 한다.

- 1번 오답: 자동 재분산은 발생하지 않는다. 이는 의도적인 설계다. 예기치 않은 파티션 이동은 대규모 복제 트래픽을 유발해 운영 중인 서비스에 영향을 줄 수 있다.
- 3번 오답: 컨트롤러는 브로커 이탈 시 리더 재선출을 담당하지, 브로커 추가 시 자동 리밸런싱을 수행하지 않는다.
- 4번 오답: 복제 인수는 토픽 생성 시 설정되며, 브로커 추가로 자동 변경되지 않는다.

</details>

### Q6. 배치 전송과 지연시간 트레이드오프

프로듀서 설정에서 `linger.ms=100`으로 설정했을 때 발생하는 현상으로 가장 정확한 설명은?

1. 메시지가 100ms마다 한 번씩 브로커에 전송되어 처리량이 감소한다.
2. 프로듀서는 최대 100ms까지 대기하며 배치를 채운 후 전송하므로 처리량은 높아지고 개별 메시지의 지연시간은 증가할 수 있다.
3. 브로커가 100ms 동안 메시지를 버퍼링한 후 컨슈머에게 전달한다.
4. 100ms 이내에 `batch.size`에 도달하지 않으면 메시지가 폐기된다.

<details><summary>정답 보기</summary>

**정답: 2**

`linger.ms`는 프로듀서가 배치를 전송하기 전 추가 메시지를 기다리는 최대 시간이다. 이 시간 동안 더 많은 메시지가 배치에 모이면 한 번의 네트워크 요청으로 다수의 메시지를 전송할 수 있어 처리량이 높아진다. 반면 개별 메시지 입장에서는 최대 100ms의 추가 대기 시간이 생긴다.

- 1번 오답: `linger.ms`는 "최대 대기 시간"이다. `batch.size`에 먼저 도달하면 100ms를 채우지 않아도 즉시 전송한다. 처리량이 감소하는 것이 아니라 증가한다.
- 3번 오답: `linger.ms`는 프로듀서 설정이다. 브로커의 버퍼링 동작을 제어하지 않는다.
- 4번 오답: 카프카는 메시지를 폐기하지 않는다. 배치가 꽉 차지 않아도 `linger.ms` 시간이 지나면 현재까지 쌓인 배치를 전송한다.

</details>

### Q7. 제로 카피의 제약

카프카가 제로 카피(Zero-Copy) 기술을 활용하여 높은 처리량을 달성하지만, 특정 상황에서는 제로 카피를 사용할 수 없다. 다음 중 제로 카피가 비활성화되는 상황은?

1. 컨슈머 그룹에 여러 컨슈머가 동시에 읽기 요청을 보낼 때
2. 파티션의 복제 인수(replication.factor)가 3 이상일 때
3. 브로커와 클라이언트 간 TLS/SSL 암호화가 활성화된 경우
4. 배치 크기(batch.size)가 기본값보다 클 때

<details><summary>정답 보기</summary>

**정답: 3**

제로 카피는 커널 공간에서 디스크 버퍼를 소켓 버퍼로 직접 복사하는 `sendfile()` 시스템 콜을 활용한다. TLS/SSL 암호화를 적용하면 데이터를 암호화하기 위해 유저 공간으로 데이터를 가져와야 하므로 제로 카피를 사용할 수 없게 된다. 대신 Java NIO의 일반 채널 기반 전송으로 동작한다.

- 1번 오답: 다수의 컨슈머가 동시에 읽어도 제로 카피는 정상 동작한다. 오히려 제로 카피의 장점이 극대화되는 상황이다.
- 2번 오답: 복제 인수는 제로 카피 적용 여부와 관련이 없다. 팔로워 브로커가 리더로부터 데이터를 페치할 때도 제로 카피가 활용된다.
- 4번 오답: 배치 크기는 제로 카피 적용 여부에 영향을 주지 않는다.

</details>

### Q8. 요청 처리 모델 병목 진단

카프카 브로커의 지연시간이 급증했다. 모니터링 결과 다음과 같은 지표를 확인했다.

- 네트워크 처리 스레드 CPU: 10% (정상)
- IO 스레드 CPU: 95% (매우 높음)
- 디스크 I/O 사용률: 90% (매우 높음)
- 요청 큐(Request Queue) 크기: 지속 증가

가장 적절한 조치는?

1. `num.network.threads` 값을 늘린다.
2. `num.io.threads` 값을 늘리거나 디스크를 추가한다.
3. `queued.max.requests` 값을 늘린다.
4. 새로운 브로커를 클러스터에 추가한 후 파티션을 재분산한다.

<details><summary>정답 보기</summary>

**정답: 2**

진단 결과를 분석하면:
- 네트워크 스레드 CPU가 낮다 → 네트워크 처리는 병목이 아님
- IO 스레드 CPU와 디스크 I/O가 모두 높다 → IO 처리가 병목
- 요청 큐가 쌓인다 → IO 스레드가 요청을 처리하는 속도보다 네트워크 스레드가 요청을 쌓는 속도가 빠름

`num.io.threads`를 늘리면 더 많은 스레드가 디스크 작업을 병렬 처리할 수 있다. 디스크 자체가 포화 상태라면 디스크(물리적 드라이브 또는 마운트 포인트)를 추가하고 IO 스레드를 늘리는 것이 근본 해결책이다.

- 1번 오답: 네트워크 스레드는 CPU 사용률이 10%로 여유가 충분하다. 늘려도 효과 없다.
- 3번 오답: 큐 크기를 늘리면 큐에 더 많은 요청을 쌓을 수 있지만, 처리 속도는 그대로이므로 지연시간만 더 늘어난다.
- 4번 오답: 장기적인 해결책으로는 유효하지만, 브로커 추가 후 파티션 재분산을 완료하기까지 시간이 걸린다. 현재 병목 원인이 IO 스레드 수와 디스크 처리량 한계이므로 2번이 더 직접적인 조치다.

</details>

### Q9. 컨트롤러 장애와 리더 선출 과정

ZooKeeper 모드에서 운영 중인 카프카 클러스터에서 컨트롤러 역할을 맡은 브로커(Broker 1)가 갑자기 장애를 겪었다. 이후 새로운 컨트롤러가 선출되고 서비스가 복구되는 과정을 순서대로 서술하시오. (ZooKeeper, 브로커, ISR, LeaderAndIsr 요청 개념을 포함할 것)

<details><summary>정답 보기</summary>

**복구 과정 (순서)**:

1. **ZooKeeper 세션 만료 감지**: Broker 1이 장애를 겪으면 ZooKeeper와의 하트비트가 끊어진다. ZK 세션 타임아웃(`zookeeper.session.timeout.ms`) 이후 ZK는 `/controller` 임시(Ephemeral) 노드를 자동으로 삭제한다.

2. **컨트롤러 재선출**: `/controller` 노드 삭제를 감지한 나머지 브로커들(Broker 2, 3 등)이 동시에 해당 노드를 등록하려 경쟁한다. ZooKeeper의 원자적 연산에 의해 가장 먼저 성공한 하나의 브로커만 새 컨트롤러가 된다 (예: Broker 2).

3. **브로커 이탈 파티션 파악**: Broker 2(신규 컨트롤러)는 ZK에서 `/brokers/ids` 목록을 조회하여 Broker 1이 이탈했음을 확인한다. Broker 1이 리더를 맡고 있던 파티션 목록을 조회한다.

4. **새 리더 선출**: 각 파티션의 ISR(In-Sync Replica) 목록을 확인한다. ISR에서 첫 번째로 등록된 살아있는 브로커를 새 리더로 선정한다. ISR이 비어있으면 `unclean.leader.election.enable` 설정에 따라 ISR 외 복제본을 리더로 허용할지 결정한다.

5. **LeaderAndIsr 전파**: 신규 컨트롤러(Broker 2)는 영향받은 파티션의 새 리더/팔로워 정보를 담은 `LeaderAndIsr` 요청을 모든 살아있는 브로커에게 전송한다. 각 브로커는 자신이 어떤 파티션의 리더/팔로워인지 최신 상태로 업데이트한다.

6. **클라이언트 메타데이터 갱신**: 프로듀서/컨슈머는 다음 메타데이터 요청 또는 에러 응답을 통해 파티션 리더 변경을 인지하고 새 리더 브로커로 재연결한다.

**핵심 포인트**: 이 전체 과정은 수 초 내에 완료되며, 이 시간 동안 영향받은 파티션으로의 쓰기/읽기는 일시적으로 실패할 수 있다. `min.insync.replicas`, `acks=-1` 설정이 되어있는 경우 ISR이 해당 수 아래로 내려가면 프로듀서는 `NotEnoughReplicasException`을 받는다.

</details>

### Q10. 분산 시스템 설계 종합 분석

다음 시나리오를 읽고 질문에 답하시오.

**시나리오**: 실시간 주문 처리 시스템에서 카프카를 도입했다. 현재 설정은 다음과 같다.
- `replication.factor=3`, `min.insync.replicas=2`, `acks=all`
- `linger.ms=50`, `batch.size=64KB`
- 브로커 3대, 파티션 12개
- TLS/SSL 암호화 활성화

운영팀은 두 가지 문제를 호소한다:
① 최근 주문 폭증 시간대에 메시지 처리량이 브로커 1대당 100MB/s 이상 올라가지 않는다.
② 결제 완료 이벤트의 종단간 지연시간이 평균 200ms인데, 이를 20ms 이하로 줄여야 한다.

각 문제의 원인을 아키텍처 관점에서 분석하고, 설정 변경 외에 아키텍처 수준에서 가능한 개선 방안을 제시하시오.

<details><summary>정답 보기</summary>

**① 처리량 한계 분석**

**원인 분석**:
- TLS/SSL이 활성화되어 있어 제로 카피(Zero-Copy)가 비활성화된 상태다. 일반 NIO 기반 전송이 사용되므로 CPU 사용률이 증가하고 네트워크 처리 효율이 낮아진다.
- `acks=all` + `min.insync.replicas=2` 설정으로 인해 리더 브로커가 쓰기를 확정하려면 2개의 팔로워 브로커가 복제를 완료했다는 응답을 기다려야 한다. 네트워크 왕복이 추가로 발생한다.
- 배치 크기 64KB는 처리량 최적화 관점에서 다소 작을 수 있다.

**아키텍처 수준 개선 방안**:
- 파티션 수를 12개에서 늘린다. 파티션이 병렬 처리의 단위이므로, 브로커 수에 비례하거나 그 배수로 설정하면 CPU 코어를 더 활용할 수 있다.
- 브로커 수를 늘리고 파티션을 재분산하여 부하를 분산한다.
- TLS 오프로딩을 적용한다. 로드밸런서나 별도 프록시에서 TLS를 종료하고 내부 브로커 간 통신은 평문으로 처리하면 제로 카피를 부분적으로 활용 가능하다. 단, 네트워크 보안 정책 검토 필요.
- 브로커에 고성능 네트워크 인터페이스(10GbE 이상)와 NVMe SSD를 사용하여 물리적 한계를 높인다.

**② 지연시간 분석**

**원인 분석**:
- `linger.ms=50`이 가장 직접적인 원인이다. 프로듀서가 배치를 채우기 위해 최대 50ms를 대기하므로, 개별 메시지는 최대 50ms의 추가 지연이 발생한다.
- `acks=all` 설정으로 리더 + 팔로워 복제 완료 대기가 추가된다. 네트워크 왕복 시간(RTT)에 따라 수십 ms가 추가될 수 있다.
- 두 지연을 합하면 50ms + 복제 지연 + 처리 시간으로 200ms 전후 도달 가능

**아키텍처 수준 개선 방안**:
- 결제 완료 이벤트 전용 토픽을 분리하고 `linger.ms=0`으로 설정하여 메시지 생성 즉시 전송하도록 한다. 배치 처리가 필요한 토픽과 지연시간이 민감한 토픽을 아키텍처 수준에서 분리하는 것이 핵심이다.
- 지연시간이 민감한 파티션의 리더를 컨슈머와 물리적으로 가까운 브로커에 배치한다 (랙 인식(Rack Awareness) 설정 활용).
- `acks=1`로 변경하면 팔로워 복제 대기가 없어 지연이 줄지만, 리더 장애 시 미복제 메시지 손실 위험이 있다. 결제 시스템의 경우 `acks=all`을 유지하되 팔로워 브로커를 리더와 동일 데이터센터 내에 두어 복제 지연을 최소화하는 것이 더 안전하다.
- 결제 완료 후 카프카 전송을 비동기 처리하고 응답을 먼저 반환하는 방식으로 종단간 지연을 체감상 줄이는 애플리케이션 레벨 설계도 고려할 수 있다.

**핵심 교훈**: 처리량(throughput)과 지연시간(latency)은 근본적으로 트레이드오프 관계다. 배치 크기와 `linger.ms`를 높이면 처리량이 높아지지만 지연시간이 증가하고, 낮추면 반대다. 사용 사례별로 토픽과 프로듀서 설정을 분리하는 것이 실무적인 해결책이다.

</details>
