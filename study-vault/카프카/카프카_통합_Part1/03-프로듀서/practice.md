## ★ 기본 (4문제)

**Q1.** 카프카 프로듀서에서 `acks=1`로 설정했을 때, 메시지 유실이 발생할 수 있는 시나리오를 설명하라.

<details><summary>정답 보기</summary>

`acks=1`은 리더 파티션이 메시지를 로컬 로그에 기록한 시점에 프로듀서에 ack를 반환한다. 이 순간 팔로워들은 아직 해당 메시지를 복제하지 않은 상태일 수 있다.

유실 시나리오:
1. 리더가 ack를 보낸 직후 장애로 다운된다.
2. 팔로워 중 하나가 새 리더로 선출된다.
3. 새 리더는 이전 리더가 ack를 보낸 메시지를 복제받지 못한 상태다.
4. 컨슈머는 해당 메시지를 영구히 볼 수 없다. 메시지 유실 발생.

방지하려면 `acks=all`과 `min.insync.replicas=2` 이상을 함께 사용해야 한다.

</details>

**Q2.** `batch.size=16384`, `linger.ms=0`으로 설정된 프로듀서가 있다. 메시지를 초당 10건씩 매우 낮은 속도로 전송한다면 배치 효과가 거의 없는 이유를 설명하라.

<details><summary>정답 보기</summary>

`linger.ms=0`이면 Sender 스레드는 RecordAccumulator에 메시지가 하나라도 있으면 즉시 전송한다. 초당 10건이라는 낮은 전송 속도에서는 메시지가 배치에 쌓이기 전에 Sender가 가져가 버린다. 결과적으로 대부분의 배치는 메시지 1건만 담고 전송된다. `batch.size`를 아무리 크게 설정해도 효과가 없다.

배치 효과를 얻으려면 `linger.ms`를 일정 값(예: 10~20ms)으로 설정해 메시지가 쌓이는 시간을 확보해야 한다. 이를 통해 낮은 전송률에서도 여러 메시지를 하나의 배치로 묶어 전송할 수 있다.

</details>

**Q3.** 카프카 프로듀서에서 커스텀 시리얼라이저 대신 Avro + Schema Registry를 권장하는 이유를 두 가지 이상 설명하라.

<details><summary>정답 보기</summary>

1. **스키마 진화(Schema Evolution) 지원**: Avro는 하위 호환(Backward Compatibility)과 상위 호환(Forward Compatibility)을 선언적으로 관리한다. 필드 추가 시 기본값을 지정하면 기존 컨슈머가 새 메시지를 읽을 수 있다. 커스텀 직렬화는 필드 하나 추가에도 프로듀서·컨슈머 코드를 동시에 변경해야 한다.

2. **스키마 중앙 관리**: Schema Registry에 스키마가 버전별로 저장되므로 어떤 프로듀서가 어떤 버전의 스키마로 메시지를 썼는지 추적 가능하다. 커스텀 직렬화는 이 정보를 별도로 관리해야 하고 누락되기 쉽다.

3. **타입 안전성**: 스키마 정의에서 타입 오류를 컴파일 타임에 잡을 수 있다. 커스텀 직렬화는 런타임에야 직렬화 오류가 발견되는 경우가 많다.

4. **상호운용성**: Avro/Protobuf는 언어 중립적이므로 Python 프로듀서와 Java 컨슈머가 같은 스키마로 통신 가능하다.

</details>

**Q4.** 키 없이 메시지를 전송할 때 Kafka 2.4 이전의 라운드로빈 파티셔닝과 스티키 파티셔닝의 차이를 설명하고, 스티키 파티셔닝이 처리량에 유리한 이유를 설명하라.

<details><summary>정답 보기</summary>

**라운드로빈 (Kafka 2.4 이전)**
메시지가 올 때마다 다음 파티션으로 순환 배정한다. 메시지 3개를 3개 파티션에 보내면 파티션별 배치 크기는 1이다. Sender는 각 파티션에 대해 개별 ProduceRequest를 보내야 하므로 브로커에 3번의 요청이 발생한다.

**스티키 파티셔닝 (Kafka 2.4+)**
현재 배치가 `batch.size`에 도달하거나 `linger.ms`가 경과할 때까지 같은 파티션에 메시지를 쌓는다. 메시지 3개가 모두 파티션 0번 배치에 들어가고 한 번의 ProduceRequest로 전송된다.

**처리량에 유리한 이유**:
- 브로커 요청 수 감소: 여러 메시지를 하나의 요청으로 전송
- 배치 압축률 향상: 배치가 클수록 중복 패턴(JSON 필드명 등)이 많아 압축 효과 증가
- 네트워크 오버헤드 감소: TCP/Kafka 프로토콜 헤더 비용을 메시지 수가 아닌 배치 수로 나눔

</details>

## ★★ 중급 (4문제)

**Q5.** 다음 설정으로 프로듀서를 구성했다. 브로커의 ISR이 1개(리더만 남음)로 줄어들었을 때 어떤 일이 발생하는지 설명하라.

```java
props.put("acks", "all");
props.put("min.insync.replicas", "2");   // 브로커 설정
```

<details><summary>정답 보기</summary>

ISR이 1개(리더만)로 줄어든 상태에서 `acks=all`은 "현재 ISR 전체"의 확인을 요구한다. ISR이 1개이면 리더 하나만 확인하면 되므로, `acks=all`만 있다면 리더 응답만으로 성공 처리될 수 있다.

그러나 `min.insync.replicas=2`가 함께 설정되면 브로커가 "쓰기를 허용하기 위한 최소 ISR 수"를 강제한다. ISR이 2 미만인 파티션에 쓰기를 시도하면 브로커는 `NotEnoughReplicasException`을 반환한다.

결과:
- 프로듀서는 `NotEnoughReplicasException`을 받는다.
- `retries` 설정에 따라 재시도하지만 ISR이 복구되지 않으면 계속 실패한다.
- `delivery.timeout.ms` 초과 시 `TimeoutException`이 최종 발생한다.
- **메시지는 쓰이지 않는다.** 이것이 의도된 동작이다. 복제 없는 쓰기를 허용하면 장애 시 유실되기 때문이다.

이 설정 조합은 "데이터 유실 없음"을 위해 가용성을 포기하는 선택이다.

</details>

**Q6.** 멱등적 프로듀서(`enable.idempotence=true`)를 활성화했을 때 내부적으로 중복을 방지하는 메커니즘을 PID와 시퀀스 번호를 중심으로 설명하라. 또한 이 방식이 프로듀서 프로세스 재시작 시 정확히 한 번을 보장하지 못하는 이유를 설명하라.

<details><summary>정답 보기</summary>

**중복 방지 메커니즘**

프로듀서 초기화 시 브로커로부터 PID(Producer ID)를 할당받는다. 이후 각 메시지에 `(PID, 파티션 번호, SequenceNumber)` 튜플이 부착된다. 시퀀스 번호는 파티션별로 0부터 단조 증가한다.

브로커는 파티션별로 `(PID, 파티션)` 쌍에 대한 마지막 시퀀스 번호를 메모리에 유지한다. 수신한 메시지의 시퀀스 번호가 이미 처리한 것이면 조용히 무시(중복 제거)한다. 시퀀스 번호에 Gap이 있으면 `OutOfOrderSequenceException`을 반환해 재전송을 유도한다.

이 덕분에 네트워크 오류로 ack가 유실되어 프로듀서가 재전송해도 브로커에 중복이 저장되지 않는다.

**프로세스 재시작 시 보장 안 되는 이유**

PID는 프로듀서 세션(프로세스 생존 기간)에만 유효하다. 프로듀서가 재시작되면 브로커로부터 새 PID를 할당받는다. 이전 PID와 새 PID는 브로커 입장에서 완전히 다른 프로듀서다. 브로커는 이전 PID로 보낸 마지막 시퀀스 번호를 새 PID와 연결하지 않는다.

따라서 프로세스 재시작 전에 전송 중이던 메시지가 재시작 후 재전송되면, 새 PID로 인식되어 중복 감지가 작동하지 않는다. 프로세스 재시작 간 정확히 한 번을 보장하려면 `transactional.id`를 설정한 트랜잭션 프로듀서를 사용해야 한다.

</details>

**Q7.** `buffer.memory`가 가득 찼을 때 프로듀서 애플리케이션에서 어떤 일이 일어나는지 `max.block.ms`와 연관지어 설명하라. 이 상황이 지속되면 어떻게 대처해야 하는가?

<details><summary>정답 보기</summary>

**버퍼 가득 참 시 동작**

`buffer.memory`(기본 32MB)가 가득 차면 `send()` 메서드가 블로킹 상태로 진입한다. Sender 스레드가 배치를 브로커에 전송하고 버퍼를 회수할 때까지 대기한다.

`max.block.ms`(기본 60,000ms = 60초) 이내에 버퍼 공간이 생기면 정상적으로 메시지를 추가하고 반환한다. `max.block.ms`를 초과해도 버퍼가 확보되지 않으면 `TimeoutException`이 발생한다.

**원인 분석**

버퍼가 지속적으로 가득 차는 상황은 "프로듀서가 보내는 속도 > Sender가 처리하는 속도"임을 의미한다.
- 브로커가 느리거나 응답하지 않는 경우
- 네트워크 병목
- `acks=all`에서 ISR 복제가 느린 경우
- 프로듀서 메시지 생성 속도가 비정상적으로 높은 경우

**대처 방법**

1. `buffer.memory` 증가: 일시적 버스트를 흡수할 버퍼 확보
2. `batch.size`와 `linger.ms` 조정: 더 효율적인 배치로 Sender 처리량 향상
3. 압축 활성화: 네트워크 전송량 감소
4. 파티션 수 증가: 병렬 전송 가능한 파티션 확보
5. 브로커 측 성능 점검: ISR 복제 지연, 디스크 I/O 등
6. 프로듀서 인스턴스 수 증가 (단, 파티션 수가 뒷받침되어야 함)

</details>

**Q8.** 키 기반 파티셔닝을 사용하는 프로듀서에서 특정 키가 전체 메시지의 90%를 차지할 때 발생하는 문제와 해결 방법을 설명하라.

<details><summary>정답 보기</summary>

**문제: 핫스팟 파티션**

키 기반 파티셔닝은 `murmur2(key) % numPartitions`로 파티션을 결정한다. 특정 키가 전체 메시지의 90%를 차지하면 해당 키의 해시가 매핑된 파티션에 부하가 집중된다.

발생하는 문제:
- 해당 파티션을 담당하는 브로커 리더의 CPU·디스크·네트워크 과부하
- 다른 파티션은 놀고 있는 반면 핫스팟 파티션만 포화 상태
- 프로듀서의 해당 파티션 배치가 계속 가득 차 `buffer.memory` 고갈 유발 가능
- 컨슈머 측에서 해당 파티션의 컨슈머만 과부하

**해결 방법**

1. **키 설계 변경**: 값 쏠림의 근본 원인인 키 설계를 재검토. 키에 타임스탬프나 랜덤 suffix를 추가해 카디널리티 증가. (예: `userId` → `userId-{random 0~9}`)

2. **커스텀 파티셔너 구현**: 특정 핫 키를 감지해 여러 파티션에 분산 배정. 단, 같은 키가 여러 파티션에 분산되므로 순서 보장이 깨진다는 트레이드오프 존재.

3. **파티션 수 재설계**: 파티션 수를 늘려도 특정 키의 해시값은 하나이므로 근본 해결이 안 됨. 오히려 커스텀 파티셔너가 필요한 상황임을 인식.

4. **비즈니스 로직 검토**: 특정 키가 몰리는 것이 데이터 특성이라면 해당 키에 대한 처리 경로를 별도 토픽으로 분리.

</details>

## ★★★ 심화 (2문제)

**Q9.** Kafka Streams 애플리케이션에서 "소비 → 처리 → 생산"을 정확히 한 번 보장하기 위해 트랜잭션 프로듀서가 필요한 이유를 설명하라. `transactional.id`, 에포크 증가, 좀비 프로듀서 펜싱이 이 보장에서 각각 어떤 역할을 하는지 서술하라.

<details><summary>정답 보기</summary>

**왜 트랜잭션이 필요한가**

Kafka Streams의 처리 패턴은 다음과 같다.
1. 입력 토픽에서 메시지 소비 (컨슈머 오프셋 전진)
2. 처리 후 출력 토픽에 결과 쓰기 (프로듀서)

멱등적 프로듀서만으로는 "출력 쓰기"의 중복은 방지하지만, "오프셋 커밋"과 "출력 쓰기"가 원자적이지 않다. 예를 들어 출력 쓰기는 성공했지만 오프셋 커밋 전에 장애가 나면 재시작 시 같은 메시지를 다시 처리해 출력을 중복 생산한다. 트랜잭션은 오프셋 커밋과 출력 쓰기를 하나의 원자 단위로 묶는다.

**transactional.id의 역할**

`transactional.id`는 프로듀서 인스턴스를 식별하는 논리적 ID다. 프로세스 재시작 후에도 같은 ID를 사용하면 Transaction Coordinator가 이전 세션을 인식하고 상태를 이어받는다. 이를 통해 프로세스 재시작 간에도 트랜잭션 연속성이 유지된다.

**에포크 증가의 역할**

같은 `transactional.id`로 새 프로듀서가 초기화될 때 Transaction Coordinator는 에포크(epoch) 번호를 1 증가시킨다. 이후 구 프로듀서(낮은 에포크)가 보내는 모든 요청은 브로커가 `ProducerFencedException`으로 거부한다. 이로써 구 프로듀서는 더 이상 쓰기를 진행할 수 없다.

**좀비 프로듀서 펜싱의 역할**

분산 시스템에서 "죽었다고 생각한" 프로듀서가 실제로는 살아서 계속 메시지를 쓰는 좀비 현상이 발생할 수 있다. 예: GC 일시 정지, 네트워크 파티션 복구 후 재접속. 에포크 기반 펜싱은 오래된 에포크의 프로듀서를 자동으로 차단하므로, 새 프로듀서와 구 좀비 프로듀서가 동시에 쓰는 상황을 방지한다. 이것이 없으면 두 프로듀서가 동시에 같은 `transactional.id`로 쓰기를 시도해 정확히 한 번 보장이 깨진다.

**전체 보장 흐름**

```
재시작 전 프로듀서 (에포크 5) → 좀비
재시작 후 프로듀서 (에포크 6)
  initTransactions() → 에포크 6 확인, 에포크 5 펜싱
  beginTransaction()
  send(출력 메시지)
  sendOffsetsToTransaction(입력 오프셋)  ← 오프셋 커밋도 트랜잭션에 포함
  commitTransaction()
  → 출력 쓰기 + 오프셋 커밋이 원자적으로 완료
```

컨슈머가 `isolation.level=read_committed`이면 미완료 트랜잭션의 메시지는 노출되지 않는다.

</details>

---

**Q10.** 프로듀서 전달 보장 수준(delivery guarantee)을 "최대 한 번(at-most-once)", "최소 한 번(at-least-once)", "정확히 한 번(exactly-once)"으로 나눌 때, 각 수준을 달성하기 위한 카프카 프로듀서 설정 조합을 구체적으로 제시하고, 각각이 어떤 장애 시나리오에서 어떤 결과를 내는지 비교하라.

<details><summary>정답 보기</summary>

**최대 한 번 (At-Most-Once)**

설정:
```java
acks=0                          // 응답 대기 없음
retries=0                       // 재시도 없음
```

또는:
```java
acks=1
retries=0
```

동작: 메시지를 한 번 보내고 결과를 확인하지 않는다. 전송 실패 시 재시도하지 않는다.

장애 시나리오:
- 네트워크 오류 → 메시지 유실. 프로듀서는 알지 못함.
- 리더 장애 → ack를 못 받았지만 재시도 없음 → 유실

언제 사용: 광고 클릭 로그, 게임 이벤트처럼 일부 유실이 허용되고 지연을 극단적으로 낮춰야 할 때.

---

**최소 한 번 (At-Least-Once)**

설정:
```java
acks=all
retries=Integer.MAX_VALUE        // 사실상 무한 재시도
delivery.timeout.ms=120000
min.insync.replicas=2            // 브로커 설정
```

동작: 성공 ack를 받을 때까지 재시도한다. 네트워크 오류로 ack가 유실되면 이미 저장된 메시지를 다시 보낸다.

장애 시나리오:
- 네트워크 오류로 ack 유실 → 프로듀서 재전송 → 브로커에 중복 저장
- 리더 장애 → ISR 복제가 완료된 경우 새 리더에 정상 존재, 미복제 시 재시도로 재전송

결과: 메시지 유실 없음. 단, 중복 가능.

언제 사용: 금융 이체처럼 유실은 절대 안 되고 컨슈머 측에서 멱등 처리(중복 제거 로직)가 가능한 경우.

---

**정확히 한 번 (Exactly-Once)**

설정:
```java
// 단일 파티션 EOS (멱등적 프로듀서)
enable.idempotence=true          // Kafka 3.0 기본값
acks=all                         // 자동 강제
retries=Integer.MAX_VALUE        // 자동 강제
max.in.flight.requests.per.connection=5

// 다중 파티션 EOS (트랜잭션)
transactional.id=unique-producer-id
isolation.level=read_committed   // 컨슈머 설정
```

동작:
- 멱등적 프로듀서: PID + 시퀀스 번호로 브로커에서 중복 제거. 동일 세션 내 재전송에 대해 정확히 한 번.
- 트랜잭션: 여러 파티션에 대한 쓰기와 오프셋 커밋을 원자적으로 처리.

장애 시나리오:
- 네트워크 오류로 ack 유실 → 재전송 → 브로커가 시퀀스 번호로 중복 감지 → 조용히 무시. 정확히 한 번.
- 트랜잭션 중 프로세스 장애 → 재시작 시 에포크 증가, 구 트랜잭션 어보트 → 새 트랜잭션 시작. 원자성 유지.
- 프로듀서 재시작(멱등만) → 새 PID → 재시작 전 미확인 메시지가 중복될 수 있음. 트랜잭션 필요.

언제 사용: Kafka Streams 처리 파이프라인, 결제·주문처럼 중복도 유실도 허용되지 않는 시스템.

---

**비교 요약**

| 수준 | 유실 | 중복 | 오버헤드 | 복잡도 |
|------|------|------|----------|--------|
| At-Most-Once | 가능 | 없음 | 최소 | 낮음 |
| At-Least-Once | 없음 | 가능 | 중간 | 중간 |
| Exactly-Once | 없음 | 없음 | 높음 | 높음 |

트레이드오프의 핵심: 정확히 한 번은 공짜가 아니다. 트랜잭션 코디네이터 왕복, 시퀀스 번호 추적, 트랜잭션 마커 쓰기가 추가되므로 처리량 감소와 지연 증가를 감수해야 한다.

</details>
