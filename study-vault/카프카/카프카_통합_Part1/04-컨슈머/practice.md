## ★ 기본

**Q1.** 토픽에 파티션이 6개, 컨슈머 그룹에 컨슈머가 4개 있다. 파티션은 어떻게 배분되는가? 또한 컨슈머를 10개로 늘리면 어떻게 되는가?

<details><summary>정답 보기</summary>

파티션 6개, 컨슈머 4개인 경우: 파티션은 컨슈머보다 많으므로 일부 컨슈머가 여러 파티션을 담당한다. RangeAssignor 기준으로 C1:[P0,P1], C2:[P2,P3], C3:[P4], C4:[P5] 형태가 될 수 있으며, 할당 전략에 따라 분포가 다르다.

컨슈머를 10개로 늘리면: 파티션 수(6)보다 컨슈머 수(10)가 많으므로 6개 컨슈머만 파티션을 하나씩 담당하고, 나머지 4개 컨슈머는 유휴(idle) 상태가 된다. 파티션 수 이상으로 컨슈머를 늘려도 병렬 처리 성능이 향상되지 않는다.

</details>

**Q2.** `enable.auto.commit=true`로 설정했을 때 오프셋이 언제 커밋되는가? 이 설정이 가진 데이터 유실 위험을 설명하라.

<details><summary>정답 보기</summary>

`auto.commit.interval.ms`(기본 5000ms) 간격이 지난 후 다음 `poll()` 호출 시, 이전 `poll()`에서 반환된 마지막 오프셋을 커밋한다.

데이터 유실 위험: `poll()`로 레코드를 받은 뒤, 처리 완료 전에 `auto.commit.interval.ms`가 경과하고 다음 `poll()`이 호출되면 아직 처리하지 않은 레코드의 오프셋이 커밋될 수 있다. 이후 컨슈머가 크래시하면 해당 메시지는 이미 커밋된 것으로 간주되어 재처리되지 않아 유실된다.

</details>

**Q3.** 서로 다른 `group.id`를 가진 두 컨슈머 그룹이 같은 토픽을 구독하면 어떻게 되는가?

<details><summary>정답 보기</summary>

두 컨슈머 그룹은 서로 독립적으로 같은 토픽의 모든 메시지를 각각 소비한다. 각 그룹은 자신만의 오프셋을 `__consumer_offsets`에 별도로 관리하므로, 한 그룹의 오프셋이 다른 그룹에 영향을 주지 않는다. 이것이 카프카에서 pub/sub 패턴을 구현하는 방식이다. 예를 들어 하나의 주문 토픽을 "결제 처리 그룹"과 "주문 통계 그룹"이 각자 독립적으로 소비할 수 있다.

</details>

**Q4.** 리밸런스가 발생하는 조건 4가지를 나열하라.

<details><summary>정답 보기</summary>

3. `session.timeout.ms` 내에 heartbeat가 수신되지 않아 컨슈머 장애로 판정될 때
4. `max.poll.interval.ms`를 초과하여 `poll()`이 호출되지 않을 때
5. (추가) 구독 중인 토픽의 파티션 수가 변경될 때

</details>

## ★★ 응용

**Q5.** `commitSync()`와 `commitAsync()`의 차이를 비교하고, 각각 어떤 상황에서 사용하는 것이 적합한가?

<details><summary>정답 보기</summary>

**commitSync():**
- 동기(블로킹) 방식. 브로커로부터 커밋 응답을 받을 때까지 스레드가 대기
- 실패 시 재시도 가능 (retriable 에러에 한해 자동 재시도)
- 처리량이 낮아지지만 커밋 성공이 보장됨
- 적합한 상황: 컨슈머 종료 시 마지막 커밋, 커밋 실패가 치명적인 경우

**commitAsync():**
- 비동기(논블로킹) 방식. 커밋 요청을 보내고 바로 반환
- 실패 시 자동 재시도 없음 (오프셋 순서 역전 문제 방지)
- 처리량이 높지만 커밋 실패를 콜백에서 별도 처리해야 함
- 적합한 상황: 일반적인 처리 루프 내 커밋, 높은 처리량이 필요할 때

**권장 패턴:** 평상시에는 `commitAsync()`로 처리량을 높이고, 컨슈머 종료 시 `finally` 블록에서 `commitSync()`로 마지막 커밋을 보장한다.

</details>

**Q6.** Eager 리밸런스와 Cooperative 리밸런스의 차이를 설명하고, Cooperative 방식이 어떻게 중단 시간을 줄이는지 설명하라.

<details><summary>정답 보기</summary>

**Eager 리밸런스:**
리밸런스 시작 시 모든 컨슈머가 보유한 파티션을 전부 해제(revoke)한다. 이후 새로운 할당이 완료될 때까지 모든 컨슈머가 메시지 소비를 중단한다(Stop-The-World). 구현이 단순하지만 전체 중단이 발생한다.

**Cooperative 리밸런스:**
두 단계로 진행된다.
1단계: 코디네이터가 재배치가 필요한 파티션만 식별하여 해당 파티션을 보유한 컨슈머에게 반납을 요청한다. 나머지 파티션을 가진 컨슈머는 계속 처리한다.
2단계: 반납된 파티션만 새롭게 재할당한다.

이 방식은 이동이 필요 없는 파티션은 중단 없이 계속 처리되므로, 전체 처리 중단 없이 부분적으로만 잠시 중단된다. `CooperativeStickyAssignor`가 이 방식을 지원한다.

</details>

**Q7.** `max.poll.interval.ms`가 왜 존재하는가? 이 값을 초과하면 어떤 일이 발생하며, 어떻게 해결할 수 있는가?

<details><summary>정답 보기</summary>

**존재 이유:**
heartbeat는 별도 스레드에서 전송되므로 메인 스레드에서 처리가 오래 걸려도 heartbeat는 계속 전송된다. 따라서 `session.timeout.ms`만으로는 "처리 로직이 멈춘 컨슈머"를 감지할 수 없다. `max.poll.interval.ms`는 `poll()` 호출 간격을 제한함으로써 처리 로직이 응답 불가 상태인 컨슈머를 감지한다.

**초과 시 발생하는 일:**
그룹 코디네이터가 해당 컨슈머를 그룹에서 이탈시키고 리밸런스를 트리거한다. 컨슈머 자신은 계속 실행되지만 파티션 할당이 취소된다.

**해결 방법:**
1. `max.poll.records`를 줄여 한 번에 처리하는 레코드 수를 감소시킨다
2. `max.poll.interval.ms`를 늘려 더 긴 처리 시간을 허용한다
3. 무거운 처리 로직을 별도 스레드나 비동기로 분리하고 메인 루프에서는 빠르게 `poll()`을 호출한다

</details>

**Q8.** `StickyAssignor`와 `RoundRobinAssignor`의 차이를 설명하고, 리밸런스 시 각각 어떻게 동작하는지 예시로 설명하라.

<details><summary>정답 보기</summary>

**RoundRobinAssignor:**
전체 파티션을 컨슈머에게 순서대로 하나씩 돌아가며 할당한다. 균등 배분이 장점이지만, 리밸런스 시 기존 할당을 고려하지 않아 많은 파티션이 이동할 수 있다.

예시: C1:[P0,P2,P4], C2:[P1,P3,P5]에서 C3 추가 시
→ C1:[P0,P3], C2:[P1,P4], C3:[P2,P5] (기존 할당 무관하게 재배분, 많은 이동 발생)

**StickyAssignor:**
리밸런스 후에도 기존 할당을 최대한 유지하면서 균등 배분을 달성한다. 불필요한 파티션 이동을 최소화하여 리밸런스 비용을 줄인다.

예시: C1:[P0,P2,P4], C2:[P1,P3,P5]에서 C3 추가 시
→ C1:[P0,P2], C2:[P1,P3], C3:[P4,P5] (각자 기존 파티션 유지, P4와 P5만 이동)

StickyAssignor는 계산이 복잡하지만 파티션 이동 횟수가 적어 리밸런스 중 처리 중단이 최소화된다.

</details>

## ★★★ 심화

**Q9.** 다음 상황을 분석하라. 컨슈머가 메시지를 처리하고 DB에 저장한 뒤 `commitAsync()`를 호출했다. 직후 컨슈머가 크래시되어 재시작되었다. 어떤 문제가 발생할 수 있으며, 이를 해결하기 위한 전략 두 가지를 설명하라.

<details><summary>정답 보기</summary>

**발생 가능한 문제 - 중복 처리:**
`commitAsync()`는 비동기이므로 크래시 시 커밋 요청이 브로커에 도달하지 못했을 가능성이 있다. 재시작된 컨슈머는 커밋되지 않은 오프셋부터 다시 읽으므로, 이미 DB에 저장된 메시지를 다시 처리하게 된다.

**해결 전략 1 - 멱등성(Idempotency) 보장:**
DB 저장 로직을 멱등적으로 설계한다. 메시지 고유 ID를 DB에 함께 저장하고, 저장 전에 중복 여부를 확인한다. `INSERT ... ON CONFLICT DO NOTHING` 또는 `UPSERT` 패턴을 사용하여 같은 메시지가 두 번 처리되어도 결과가 동일하게 만든다.

**해결 전략 2 - 트랜잭션적 처리 (정확히 한 번):**
DB 저장과 오프셋 커밋을 하나의 트랜잭션으로 묶는다. 오프셋을 `__consumer_offsets` 대신 DB에 직접 저장하고, 메시지 처리와 오프셋 업데이트를 같은 DB 트랜잭션으로 실행한다. 이렇게 하면 처리와 오프셋 기록이 원자적으로 보장된다.

</details>

**Q10.** 컨슈머 그룹의 그룹 코디네이터와 그룹 리더의 역할을 정확히 구분하고, 새 컨슈머가 그룹에 참가할 때부터 파티션을 할당받기까지의 전체 과정을 단계별로 설명하라.

<details><summary>정답 보기</summary>

**역할 구분:**

그룹 코디네이터 (브로커 측):
- `hash(group.id) % __consumer_offsets 파티션 수`로 결정된 파티션의 리더 브로커
- 컨슈머 멤버십 관리 (등록/이탈 처리)
- heartbeat 수신 및 세션 타임아웃 모니터링
- 리밸런스 트리거 및 조율
- 각 컨슈머에 파티션 할당 결과 전파
- 오프셋 저장 및 조회

그룹 리더 (첫 번째로 JoinGroup을 보낸 컨슈머):
- 파티션 할당 계획 실제 계산 (Assignment Strategy 실행)
- 계산된 할당 계획을 SyncGroup 요청으로 코디네이터에 전달

**참가 과정 단계별 설명:**

1. **FindCoordinator:** 새 컨슈머가 임의의 브로커에 `FindCoordinator` 요청을 보내 자신의 그룹을 담당하는 코디네이터 브로커 주소를 받는다.

2. **JoinGroup 요청:** 컨슈머가 코디네이터에 `JoinGroup` 요청을 보낸다. 지원하는 Assignment Strategy 목록을 함께 전달한다.

3. **그룹 리더 선출:** 코디네이터는 처음 JoinGroup을 보낸 컨슈머를 그룹 리더로 선출한다. 모든 컨슈머의 JoinGroup 응답에 현재 그룹 멤버 목록을 포함시키되, 리더에게만 모든 멤버 정보를 전달한다.

4. **할당 계획 수립:** 그룹 리더는 코디네이터로부터 받은 멤버 목록과 토픽 파티션 정보를 바탕으로 Assignment Strategy에 따라 파티션 할당 계획을 계산한다.

5. **SyncGroup 요청:** 그룹 리더가 계산된 할당 계획을 `SyncGroup` 요청으로 코디네이터에 전달한다. 다른 컨슈머들도 `SyncGroup`을 보내지만 할당 정보 없이 보낸다.

6. **할당 전파:** 코디네이터가 각 컨슈머의 `SyncGroup` 응답으로 해당 컨슈머에게 배정된 파티션 목록을 전달한다.

7. **처리 시작:** 각 컨슈머는 할당받은 파티션에서 `poll()`을 시작한다.

</details>
