# 핵심 아키텍처 연습 문제

> 난이도: ★ 기초 / ★★ 응용 / ★★★ 심화

---

## 문제 1 ★

RDBMS의 LIKE 검색과 엘라스틱서치의 역색인 검색이 동일한 데이터를 대상으로 동작할 때 성능 차이가 발생하는 근본 이유는 무엇인가?

<details>
<summary>정답 보기</summary>

RDBMS의 LIKE 검색은 테이블 전체를 순차 스캔(Full Table Scan)한다. 데이터가 100만 건이면 매 쿼리마다 최대 100만 번 비교를 수행한다. 인덱스를 사용할 수 없는 `LIKE '%keyword%'` 패턴의 경우 O(N)이다.

역색인은 색인 시점에 "단어 → 문서 목록" 구조를 미리 구성한다. 검색 시에는 해당 단어를 역색인에서 조회하기만 하면 되므로 O(1)에 문서 목록을 얻는다. 데이터가 아무리 많아도 검색 속도가 일정하게 유지된다.

핵심 차이: RDBMS는 검색 시점에 비교 작업이 일어나고, 역색인은 색인 시점에 미리 처리한다. 이 사전 처리 비용이 쓰기를 느리게 하는 대신 읽기를 극적으로 빠르게 만든다.

</details>

## 문제 2 ★

엘라스틱서치에서 인덱스를 생성할 때 `number_of_shards`를 3으로 설정했다. 이후 데이터가 10배 증가해 샤드당 데이터가 너무 커졌다. 이 문제를 해결하기 위해 운영 중인 인덱스의 `number_of_shards`를 6으로 변경할 수 있는가?

<details>
<summary>정답 보기</summary>

변경할 수 없다.

샤드 수는 인덱스 생성 시점에 결정되며, 이후 변경이 불가능하다. 이유는 데이터가 샤드에 라우팅될 때 `shard = hash(document_id) % number_of_shards` 공식을 사용하기 때문이다. 샤드 수가 바뀌면 기존 모든 문서의 위치가 달라져 데이터 일관성이 깨진다.

해결책:
- 새 인덱스를 원하는 샤드 수(6개)로 생성하고 기존 데이터를 Reindex API로 이전
- 또는 샤드 분할(Split) API를 사용 (기존 샤드 수의 배수로만 분할 가능)

이 때문에 인덱스 초기 설계 시 데이터 증가량을 예측해 샤드 수를 충분히 잡는 것이 중요하다.

</details>

## 문제 3 ★

레플리카 샤드가 같은 노드에 Primary Shard와 함께 배치되면 안 되는 이유는 무엇인가?

<details>
<summary>정답 보기</summary>

레플리카의 목적은 고가용성(Failover)이다. 같은 노드에 Primary와 Replica가 함께 있으면 해당 노드가 장애를 일으킬 때 Primary와 Replica 모두 소실되어 복구가 불가능해진다.

엘라스틱서치는 이를 방지하기 위해 같은 번호의 Primary Shard와 Replica Shard를 반드시 서로 다른 노드에 배치하도록 강제한다. 따라서 노드가 1대뿐인 개발 환경에서 레플리카를 설정하면 Replica는 배치될 곳이 없어 UNASSIGNED 상태가 된다.

</details>

## 문제 4 ★

"엘라스틱서치에 문서를 색인했는데 1초 후에야 검색 결과에 나타났다"는 현상의 원인을 설명하라.

<details>
<summary>정답 보기</summary>

NRT(Near Real-Time) 특성 때문이다.

색인된 문서는 즉시 루씬 세그먼트가 되지 않고 인메모리 버퍼에 먼저 쌓인다. Refresh 작업이 수행되어야 새 세그먼트가 OS Page Cache에 올라가고 IndexSearcher가 이 세그먼트를 읽을 수 있게 된다.

기본 `refresh_interval`은 1초이므로 색인 직후부터 최대 1초까지 검색 결과에 반영되지 않는다. 즉시 검색 가능하게 하려면 색인 후 `POST /index/_refresh`를 명시적으로 호출하거나, 색인 API 파라미터에 `?refresh=true`를 추가하면 된다.

</details>

## 문제 5 ★

세그먼트 불변성(Immutability)이 없다면 어떤 문제가 생기는가? 최소 두 가지를 설명하라.

<details>
<summary>정답 보기</summary>

1. **동시성 문제**: 다중 스레드 환경에서 세그먼트를 동시에 읽고 쓰면 데이터 일관성 보장을 위해 Lock이 필요하다. Lock 경합은 심각한 성능 저하를 야기한다. 불변이면 Lock이 불필요하다.

2. **OS 시스템 캐시 효율 저하**: 데이터가 변경될 때마다 OS의 Page Cache를 무효화하고 다시 로드해야 한다. 캐시 미스가 잦아지면 검색마다 디스크 I/O가 발생해 성능이 급락한다. 불변이면 한번 캐시에 올라간 세그먼트는 계속 재사용 가능하다.

추가로: 역색인을 재구성하는 CPU/메모리 비용이 지속적으로 발생하고, 캐시 적중률이 낮아져 메모리 효율이 떨어진다.

</details>

## 문제 6 ★★

다음 두 작업의 차이를 설명하라.
- 루씬 Flush
- 엘라스틱서치 Flush

<details>
<summary>정답 보기</summary>

**루씬 Flush**:
- 인메모리 버퍼의 데이터를 OS Page Cache에 `write()` 시스템 콜로 기록하여 새 세그먼트 파일 생성
- 이후 루씬의 IndexReader가 ReOpen되어 새 세그먼트가 검색 대상에 포함됨
- 실제 디스크에 영속화되지 않음 (OS가 적절한 시점에 디스크에 쓸 뿐)
- 엘라스틱서치에서 이 작업을 "Refresh"라고 부름

**엘라스틱서치 Flush**:
- 내부적으로 루씬 Commit(`fsync()` 시스템 콜)을 수행하여 OS Page Cache → 실제 디스크 동기화
- 새 Translog를 시작하고 이전 Translog를 삭제
- 매우 비용이 큰 작업 (I/O 집약적)

정리: 루씬 Flush = 검색 가능성 확보 / ES Flush = 영속성 보장 + Translog 정리

</details>

## 문제 7 ★★

Translog의 목적과, ES Flush가 Translog와 어떻게 상호작용하는지 설명하라.

<details>
<summary>정답 보기</summary>

**Translog의 목적**:
루씬 Commit은 비용이 크므로 자주 수행할 수 없다. Commit과 Commit 사이에 장애가 발생하면 인메모리/Page Cache의 데이터가 소실된다. Translog는 모든 색인/삭제/수정 작업을 즉시 fsync로 기록하는 Write-Ahead Log로, 이 간격의 데이터 유실을 방지한다.

**ES Flush와의 상호작용**:
1. 색인 요청 → 루씬 작업 수행 → Translog에 즉시 기록 (fsync)
2. Translog는 계속 커짐 (모든 변경사항 누적)
3. ES Flush 트리거 (5초 주기 또는 Translog 크기 임계치):
   - 루씬 Commit 수행 (세그먼트들을 디스크에 영속화)
   - Commit이 완료된 이전까지의 Translog 삭제
   - 새 빈 Translog 파일 생성
4. 장애 후 복구: 마지막 Commit 이후의 Translog 내용을 재실행하여 복구

Translog 크기 = Flush 주기 × 쓰기 처리량. Flush 주기가 길수록 Translog가 커지고 복구 시간이 길어진다.

</details>

## 문제 8 ★★

세그먼트 Merge가 수행될 때 서비스(검색)는 중단되는가? Merge 중 검색 요청이 들어오면 어떻게 처리되는지 설명하라.

<details>
<summary>정답 보기</summary>

중단되지 않는다. Merge는 검색 서비스를 중단하지 않고 백그라운드에서 수행된다.

동작 방식:
1. IndexWriter가 Merge 대상 세그먼트들의 **복제본**을 생성
2. 복제본들을 하나의 새 세그먼트로 병합하는 작업 시작
3. **이 작업 중 IndexSearcher는 원본 세그먼트를 계속 사용해 검색 서비스 유지**
4. 병합 완료 후 원본 세그먼트와 새 세그먼트를 교체
5. 교체 후 원본 세그먼트(삭제 비트 포함 데이터) 물리 삭제

이 Copy-on-Write 방식 덕분에 Merge가 진행되는 동안에도 검색 결과의 정확성이 보장되고 서비스가 중단 없이 운영된다.

</details>

## 문제 9 ★★★

엘라스틱서치에서 문서를 "업데이트"하면 내부적으로 어떤 일이 일어나는가? 세그먼트 불변성과 연관 지어 설명하라.

<details>
<summary>정답 보기</summary>

엘라스틱서치(루씬)의 업데이트는 실제로 "논리적 삭제 + 새 문서 색인"의 두 단계로 처리된다.

1. **논리적 삭제**: 기존 문서가 있는 세그먼트에서 해당 문서의 삭제 비트 배열 플래그를 1로 설정. 세그먼트 파일 자체는 변경되지 않으므로 불변성 유지. IndexSearcher는 검색 시 삭제 플래그를 먼저 확인하여 해당 문서를 결과에서 제외.

2. **새 세그먼트 생성**: 변경된 내용을 가진 새 문서가 인메모리 버퍼로 들어와 Refresh 시 새 세그먼트로 생성.

결과:
- 잠시 동안 "삭제된 원본 문서(비트만 설정)"와 "새 문서"가 각각 다른 세그먼트에 공존
- 검색은 삭제 비트를 확인하므로 중복 결과 없음
- 원본의 물리적 공간 해제는 Merge 작업 때까지 지연됨

이 방식이 불변성을 지키면서도 업데이트를 가능하게 하는 핵심 트릭이다. 단점은 업데이트가 많을수록 삭제된 문서가 쌓여 디스크 공간 낭비가 발생하고, Merge 전까지 실제 용량이 줄지 않는다는 점이다.

</details>

## 문제 10 ★★★

3노드 클러스터에서 Primary Shard 3개, Replica 1세트로 구성된 인덱스가 운영 중이다. 갑자기 노드 1개가 완전히 다운되었다. 이 상황에서 엘라스틱서치가 데이터 가용성을 유지하기 위해 취하는 단계를 순서대로 설명하라.

<details>
<summary>정답 보기</summary>

**전제**: 3노드(N1, N2, N3), Primary Shard(P0, P1, P2)가 각 노드에 하나씩, Replica Shard(R0, R1, R2)가 각각 다른 노드에 배치됨.

예시: N1(P0, R2), N2(P1, R0), N3(P2, R1) → N1 다운

**복구 단계**:

1. **장애 감지**: 마스터 노드가 N1과의 통신 두절을 감지 (기본 ping 타임아웃: 30초)

2. **클러스터 상태 업데이트**: 마스터가 클러스터 상태에서 N1의 모든 샤드(P0, R2)를 비활성으로 표시

3. **Replica 승격**: N1에 있던 P0의 복제본인 R0(N2에 있음)을 새로운 Primary Shard로 승격. P1은 N2에 그대로 유지. P2는 N3에 그대로 유지.

4. **새 Replica 배정**: 클러스터에 N1이 없으므로 승격된 P0의 새 Replica를 N3에 생성 (N3에 이미 P2, R1이 있으므로 R0_new를 N3에 배치). 마찬가지로 R2(원래 N1에 있던 P2의 복제본)의 새 Replica를 N2에 생성.

5. **데이터 복사**: Primary에서 새 Replica로 데이터 복사 시작 (이 동안 클러스터 상태는 yellow)

6. **복구 완료**: 모든 Replica가 정상화되면 클러스터 상태 green 복귀

핵심: 이 모든 과정에서 **서비스 중단 없이** 검색/색인 요청이 계속 처리된다. 다운된 노드의 Primary가 담당하던 요청은 승격된 Replica가 즉시 처리하기 때문이다.

</details>

## 문제 11 ★★★

대량 데이터 색인 작업(Bulk Indexing)을 수행할 때 성능을 극대화하기 위해 조정하는 ES 설정 두 가지를 말하고, 각 설정이 왜 도움이 되는지 내부 동작 원리와 연결해 설명하라.

<details>
<summary>정답 보기</summary>

**설정 1: `index.refresh_interval: -1` (Refresh 비활성화)**

원리: Refresh는 인메모리 버퍼의 데이터를 OS Page Cache에 올리고 IndexReader를 ReOpen하는 작업이다. 기본 1초 주기로 실행되면 대량 색인 중에도 1초마다 세그먼트를 생성한다. 세그먼트가 빠르게 쌓이면 Merge 작업이 자주 트리거되어 CPU와 I/O를 소모한다.

`-1`로 설정하면 자동 Refresh를 중단하여 인메모리 버퍼가 충분히 쌓일 때까지 세그먼트 생성을 미룬다. 결과적으로 더 큰 세그먼트가 생성되고 Merge 횟수가 줄어들어 쓰기 처리량이 크게 증가한다.

완료 후 `index.refresh_interval: 1s`로 복원하고 명시적으로 `_refresh`를 호출한다.

**설정 2: `index.number_of_replicas: 0` (색인 중 Replica 비활성화)**

원리: 색인 시 문서는 Primary Shard에 기록된 후 모든 Replica에도 동기적으로 복제된다. Replica가 있으면 Primary가 받은 매 요청을 Replica 수만큼 네트워크로 전송해야 한다. 이는 쓰기 지연(latency)을 높이고 네트워크 대역폭을 소모한다.

색인 중 Replica를 0으로 설정하면 Primary에만 쓰므로 네트워크 복제 오버헤드가 사라져 쓰기 처리량이 향상된다.

완료 후 `number_of_replicas`를 원래 값으로 복원하면 ES가 자동으로 복제를 수행한다.

주의: 이 방법은 색인 완료 전까지 데이터 내구성이 낮아지는 트레이드오프가 있다.

</details>
