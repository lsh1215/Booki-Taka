# 핵심 아키텍처

> 출처: 엘라스틱서치 실무가이드 Ch1-Ch2, Ch9 / 엘라스틱서치바이블 Ch1-Ch2
> 태그: #elasticsearch #lucene #inverted-index #shard #replica #segment #translog #nrt

## 이 토픽의 근본 문제

"텍스트를 빠르게 검색하면서 대용량 데이터를 장애 없이 분산 운영하려면 어떤 구조가 필요한가"

## 전체 구조

```
[ 클러스터 ]
  ├── 노드 #1 (마스터 + 데이터)
  │     ├── 인덱스 A
  │     │     ├── Primary Shard 0 ──> Lucene Index
  │     │     │     ├── Segment 0 (역색인, 불변)
  │     │     │     ├── Segment 1 (역색인, 불변)
  │     │     │     └── Commit Point (세그먼트 목록)
  │     │     └── Replica Shard 1 (노드 #2에 배치)
  │     └── Translog (장애 복구용 WAL)
  │
  ├── 노드 #2 (데이터)
  │     ├── Primary Shard 1
  │     └── Replica Shard 0 (노드 #1 Primary 복제)
  │
  └── 노드 #3 (데이터)
        ├── Primary Shard 2
        └── Replica Shard 2

색인 흐름: 문서 → In-Memory Buffer → (Refresh) → OS Page Cache → 검색 가능
                                    → (Flush/Commit) → 디스크 (영속화)
                    → Translog (즉시 기록, fsync) → 장애 복구 보장
```

## 핵심 개념

### 역색인 (Inverted Index)

**정의**: 문서 내 단어(텀)를 키로, 해당 단어가 등장하는 문서 목록을 값으로 저장하는 자료구조.
일반 색인이 "문서 → 단어 목록"이라면, 역색인은 "단어 → 문서 목록".

**왜 필요한가**: RDBMS의 LIKE 검색은 전체 테이블을 순차 스캔한다. 텍스트가 100만 행이면
매 쿼리마다 100만 번 비교가 일어난다. 역색인이 없으면 "검색어를 포함하는 문서를 빠르게
찾는 것" 자체가 불가능하다.

**동작 원리**:
```
원문 문서:
  doc1: "the quick brown fox"
  doc2: "the fox jumped high"

역색인 구조:
  "the"   → [doc1, doc2]
  "quick" → [doc1]
  "brown" → [doc1]
  "fox"   → [doc1, doc2]
  "jumped"→ [doc2]
  "high"  → [doc2]

검색: "fox" → 역색인에서 O(1) 조회 → [doc1, doc2] 즉시 반환
```

색인 시 텍스트는 애널라이저(Analyzer)를 거쳐 토큰화 → 소문자 변환 → 불용어 제거 등
처리 후 최종 "텀(term)"으로 역색인에 등록된다.

**설계 트레이드오프**:
- 읽기(검색)는 매우 빠름. 쓰기(색인)는 느림 - 역색인 재구성 필요
- 텍스트 검색에 최적화, 범위 조건/정렬/조인에는 부적합
- 세그먼트 불변성으로 인해 수정이 "삭제 + 신규 생성"으로 처리됨

**관련 개념**: 세그먼트, 애널라이저, 텀(term)

### 클러스터 / 노드 / 인덱스 / 도큐먼트

**클러스터 (Cluster)**

**정의**: 데이터를 실제로 가지고 있는 노드의 모음. 같은 클러스터 이름으로 묶인 노드들이 협력한다.

**왜 필요한가**: 단일 서버는 하드웨어 한계와 단일 장애점(SPOF)을 가진다. 클러스터 없이는
대용량 데이터 처리와 고가용성을 동시에 달성할 수 없다.

**노드 종류**:
```
마스터 노드 (node.master: true)  → 클러스터 상태 관리, 샤드 배치 결정
데이터 노드 (node.data: true)    → 실제 데이터 보유, CRUD/검색/집계
인제스트 노드 (node.ingest: true)→ 색인 전 전처리 파이프라인
코디네이팅 노드                  → 분산 처리 조율만 담당 (데이터 없음)
```

**인덱스 (Index)**: 유사한 특성의 문서 컬렉션. RDBMS의 데이터베이스에 대응.
- 인덱스명은 소문자 강제, 클러스터 내 유일해야 함
- 루씬 인덱스(세그먼트 + 검색 기능)와 구별 필요: 엘라스틱서치 인덱스 = 여러 샤드의 논리적 묶음

**도큐먼트 (Document)**: 검색 대상이 되는 JSON 형식의 데이터 단위. RDBMS의 행(row)에 대응.

**엘라스틱서치 vs RDBMS 대응**:
```
엘라스틱서치      RDBMS
인덱스(Indices)   데이터베이스
샤드(Shard)       파티션
문서(Document)    행(Row)
필드(Field)       열(Column)
매핑(Mapping)     스키마
Query DSL         SQL
```

**관련 개념**: 샤드, 레플리카, 매핑

### 샤드 (Primary Shard)와 레플리카 (Replica Shard)

**정의**:
- Primary Shard: 인덱스 데이터를 물리적으로 분산 저장하는 단위. 하나의 엘라스틱서치 샤드 = 하나의 루씬 인덱스.
- Replica Shard: Primary Shard의 복제본.

**왜 필요한가**:
- Shard 없이는: 인덱스가 커질수록 단일 하드웨어 한계에 부딪힘. 수평 확장이 불가능.
- Replica 없이는: 노드 하나가 죽으면 해당 Primary Shard의 데이터에 접근 불가. 서비스 장애.

**동작 원리**:
```
인덱스 생성 시:
  number_of_shards: 3    → Primary Shard 3개
  number_of_replicas: 1  → 각 Primary마다 복제본 1개 = Replica 3개, 총 6개 샤드

배치 원칙:
  - 같은 번호의 Primary와 Replica는 반드시 다른 노드에 배치
  - 노드 1개 장애 → 해당 노드의 Primary를 Replica가 승격

검색 시:
  - 쿼리가 인덱스에 들어오면 모든 샤드(Primary or Replica)에 분산 전달
  - 각 샤드에서 1차 결과 생성 → 코디네이팅 노드가 병합 → 최종 결과 반환
```

**설계 트레이드오프**:
- 샤드 수는 인덱스 생성 후 변경 불가 → 초기 설계가 매우 중요
- 레플리카 수는 운영 중 변경 가능 → 트래픽 증가 시 읽기 분산 조정 가능
- 샤드가 너무 많으면: 오버헤드 증가 (각 샤드마다 JVM 힙 사용)
- 샤드가 너무 적으면: 수평 확장 한계

**관련 개념**: 노드, 루씬 인덱스, 레플리카 페일오버

### 세그먼트 (Segment)와 불변성

**정의**: 루씬 내부의 역색인 자료구조 파일. 한번 디스크에 저장되면 수정이 불가능한(immutable) 단위.

**왜 필요한가**: 역색인 구조를 변경하려면 전체를 재구성해야 한다. 매 쓰기마다 재구성하면 성능이
급격히 저하된다. 불변 세그먼트를 여러 개 두고 검색은 전체 세그먼트를 합산하면 쓰기 성능과
읽기 성능을 동시에 얻을 수 있다.

**동작 원리**:
```
색인 요청 → 인메모리 버퍼에 누적
            → (일정 조건) 루씬 Flush → OS Page Cache에 새 세그먼트 생성
            → (IndexSearcher ReOpen) → 새 세그먼트가 검색 대상에 포함

커밋 포인트(Commit Point): 현재 유효한 세그먼트 목록을 관리
IndexSearcher: 커밋 포인트를 보고 모든 세그먼트를 순회하며 검색

삭제 처리:
  - 세그먼트 불변이므로 즉시 삭제 불가
  - 각 문서에 삭제 비트 배열(bit array) 존재
  - 삭제 요청 시: 비트만 1로 설정 → 검색 시 필터링
  - 실제 물리 삭제: Merge 작업 시

수정 처리:
  - 기존 문서 삭제 비트 설정 + 수정된 내용으로 새 세그먼트 생성
```

**불변성의 장점** (실무가이드 Ch9):
- 동시성 문제 회피: Lock 불필요
- OS 시스템 캐시 적극 활용 가능 (데이터 변경 없으므로 캐시 무효화 불필요)
- 높은 캐시 적중률: 메모리에서 검색 데이터 유지
- CPU/메모리 I/O 절감: 역색인 재생성 없음

**설계 트레이드오프**:
- 장점: 읽기 성능 극대화, 캐시 효율 극대화
- 단점: 수정/삭제가 "논리적 삭제 + 새 생성"으로 처리 → 실제 공간 해제는 Merge 때까지 지연
- 세그먼트가 많아질수록 검색 시 읽어야 할 파일 수 증가 → Merge 필요

**관련 개념**: 루씬, Commit Point, Merge, 역색인

### Translog

**정의**: 엘라스틱서치 샤드가 유지하는 작업 로그 파일(Write-Ahead Log). 루씬에는 없는 ES만의 개념.

**왜 필요한가**: 루씬 Commit은 비용이 크므로 자주 할 수 없다. 그러나 Commit 사이에 장애가
발생하면 인메모리의 변경사항이 모두 유실된다. Translog 없이는 장애 복구 시 데이터 유실이 불가피하다.

**동작 원리** (실무가이드 Ch9.4, 바이블 Ch2.3.5):
```
색인/삭제/수정 요청
    ↓
루씬 인덱스에 작업 수행
    ↓
Translog에 해당 내역 즉시 기록 (fsync까지 완료)
    ↓
클라이언트에 성공 응답

장애 발생 후 재시작:
    ↓
샤드 복구 단계에서 Translog 읽기
    ↓
루씬 Commit에 포함되지 못한 작업 재실행
    ↓
데이터 복구 완료

ES Flush (루씬 Commit + 새 Translog 생성):
  - Translog가 너무 커지면 샤드 복구 시간 길어짐
  - ES Flush가 백그라운드에서 주기적으로 실행
  - Flush 완료 후 이전 Translog 삭제, 새 Translog 시작
```

**설정**:
- `index.translog.durability`: `request`(기본, 매 요청마다 fsync) / `async`(주기적 fsync, 장애 시 최대 5초 유실 가능)
- `index.translog.sync_interval`: async 모드에서 fsync 주기 (기본 5초)

**설계 트레이드오프**:
- durability=request: 내구성 보장, 쓰기 지연 증가
- durability=async: 높은 쓰기 처리량, 장애 시 데이터 유실 가능

**관련 개념**: ES Flush, 루씬 Commit, 샤드 복구

### Refresh / Flush / Merge

**Refresh**

**정의**: 인메모리 버퍼의 새 세그먼트를 OS Page Cache에 올려 검색 가능 상태로 만드는 작업.
루씬의 IndexReader를 새로 열어(openIfChanged) 변경사항을 검색에 반영.

**왜 필요한가**: 색인한 문서가 즉시 검색되지 않는다. Refresh 없이는 새 데이터가 검색 결과에
나타나지 않아 NRT(Near Real-Time) 검색이 불가능하다.

**동작**:
```
기본: 1초마다 자동 Refresh (index.refresh_interval: 1s)
예외: 30초 이상 검색 쿼리가 없으면 Refresh 중단 (idle 절약)
명시적 호출: POST /my_index/_refresh
```

**Flush (ES Flush)**

**정의**: OS Page Cache의 데이터를 물리 디스크에 영속화(루씬 Commit = fsync 수행) + 새 Translog 시작.

**왜 필요한가**: OS Page Cache는 서버 재시작 시 소멸. Flush 없이는 캐시의 세그먼트가 영속화되지
않아 재시작 시 데이터 유실. 또한 Translog를 정리해 복구 시간을 단축.

**동작**:
```
루씬 Flush (write() 기반) → OS Page Cache에만 기록 (빠름, 검색 가능)
루씬 Commit (fsync 기반) → 실제 디스크 동기화 (느림, 영속성 보장)
ES Flush = 루씬 Commit + 새 Translog 생성

주기: 기본적으로 5초에 한 번 또는 Translog 크기 임계치 도달 시
명시적 호출: POST /my_index/_flush
```

**Merge**

**정의**: 다수의 작은 세그먼트를 하나의 큰 세그먼트로 병합하는 백그라운드 작업.

**왜 필요한가**: 색인이 쌓일수록 세그먼트 파일 수가 증가 → 검색 시 더 많은 파일을 열어야 함 →
검색 성능 저하. 또한 삭제 비트가 표시된 데이터가 물리적으로 공간을 계속 차지.

**동작**:
```
Merge 전: [Seg0][Seg1][Seg2][Seg3][Seg4] ... (파일 수 증가)
Merge 중: 원본 유지하며 복제본으로 병합 작업 (검색은 원본으로 계속 서비스)
Merge 후: [Seg_merged] (원본들 삭제, 삭제 비트 문서 물리 제거)

명시적 호출: POST /my_index/_forcemerge?max_num_segments=1
  ※ 추가 색인이 완전히 끝난 인덱스에만 권장 (진행 중인 색인과 충돌 가능)
```

**설계 트레이드오프**:
- Merge는 CPU/IO 비용이 크므로 운영 중에는 자동 정책에 위임
- 검색 성능 vs 쓰기 처리량 균형 필요

**3가지 작업 비교**:
```
Refresh: 검색 가능성 확보 (1초 주기, 비교적 저렴)
Flush:   영속성 보장 (5초 주기, 비쌈 - fsync)
Merge:   성능 최적화 (비정기, 가장 비쌈 - 세그먼트 재생성)
```

### NRT (Near Real-Time) 검색

**정의**: 색인한 문서가 최대 1초 이내에 검색 가능해지는 특성. "실시간"이 아닌 "근실시간"인 이유는
Refresh 주기(기본 1초) 때문.

**왜 필요한가**: 디스크 IO(fsync)를 매번 수행하면 쓰기 성능이 급격히 저하된다. OS Page Cache를
활용해 중간 단계(Refresh)를 두면 쓰기 성능과 검색 반영 속도를 동시에 얻을 수 있다.

**동작 원리**:
```
색인 요청 → 인메모리 버퍼 누적
          → (Refresh, ~1초) → OS Page Cache에 세그먼트 생성 → 검색 가능
          → (Flush, ~5초) → 디스크 동기화 → 영속성 보장

NRT = "Refresh 주기만큼의 지연이 있지만, 디스크 영속화를 기다리지 않고 검색 가능"
```

**설계 트레이드오프**:
- refresh_interval을 줄이면: 더 실시간에 가까워짐, 단 Refresh 비용 증가
- refresh_interval을 늘리면: 쓰기 처리량 증가, 단 검색 반영 지연
- 대량 색인 시: refresh_interval=-1 (Refresh 비활성화)로 설정해 성능 극대화 후 완료 후 복원

**관련 개념**: Refresh, OS Page Cache, 세그먼트

### 루씬(Lucene)과 ES의 관계

**정의**: 루씬은 Doug Cutting이 개발한 자바 기반 오픈소스 검색 라이브러리. 역색인 생성/검색의 핵심 엔진.
엘라스틱서치는 루씬을 샤드 내부에 내장하고 분산 처리 계층을 추가한 시스템.

**계층 관계**:
```
엘라스틱서치 클러스터
  └── 엘라스틱서치 인덱스 (논리적 개념)
        └── 엘라스틱서치 샤드 (= 루씬 인덱스 + ES 부가 기능)
              └── 루씬 인덱스
                    ├── IndexWriter  (색인: 세그먼트 생성)
                    ├── IndexSearcher (검색: 커밋 포인트 기반)
                    └── Segment × N (역색인 파일들)
```

**루씬 한계 vs ES 해결책**:
```
루씬 한계                     ES 해결책
----------------------------------------------------------
단일 머신 한계                샤드로 데이터 분산
루씬 인덱스 간 검색 불가      샤드들을 논리적 인덱스로 통합
고가용성 없음                 레플리카 + 페일오버
Commit 사이 데이터 유실      Translog로 WAL 구현
```

**핵심 요약** (실무가이드 Ch9.2):
- 하나의 ES 샤드 = 하나의 루씬 인덱스
- 하나의 ES 인덱스 = 여러 샤드(= 여러 루씬 인덱스)의 논리적 집합
- 루씬의 IndexWriter, IndexSearcher가 ES 검색의 실질적 엔진
- ES는 루씬의 분산화 불가능 문제를 샤드 + 클러스터로 해결

**관련 개념**: 샤드, 세그먼트, 역색인, Translog
