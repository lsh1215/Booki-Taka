### Q1. 루씬 인덱스, ES 샤드, ES 인덱스의 관계를 설명하라.

<details>
<summary>정답 보기</summary>

- **루씬 인덱스**: `IndexWriter`(색인)와 `IndexSearcher`(검색)로 구성된 단일 검색엔진. 내부에 세그먼트를 보유. 자기가 가진 세그먼트 내에서만 검색 가능.
- **ES 샤드**: 루씬 인덱스의 확장. "장애 복구 기능을 가진 작은 루씬 기반 단일 검색 서버". ES가 추가한 기능(Translog, 복제 등)을 포함. 다수의 샤드가 협력해 전체 세그먼트를 논리적으로 통합 검색.
- **ES 인덱스**: 물리적으로 분산된 ES 샤드를 논리적으로 하나의 데이터로 바라보는 추상화 레이어.

계층 관계: ES 인덱스 > ES 샤드(= 루씬 인덱스 확장) > 세그먼트

</details>

### Q2. 루씬 세그먼트 불변성(Immutability)이란 무엇이고, 왜 이 설계를 택했는가?

<details>
<summary>정답 보기</summary>

**불변성**: 한번 디스크에 생성된 세그먼트는 수정이 불가능하다. Merge 작업으로 삭제되기 전까지는 변경을 허용하지 않는다.

**이유(장점)**:
1. **동시성 문제 회피**: Lock 없이도 다중 스레드 환경에서 안전하게 읽기 가능.
2. **시스템 캐시 적극 활용**: 데이터가 변경되지 않으므로 캐시 수명이 길어짐.
3. **높은 캐시 적중률**: 읽기 연산 비중이 높은 검색엔진에서 매우 유리.
4. **리소스 절감**: 부분 변경 시에도 역색인 전체를 재생성하지 않아도 됨.

**단점**: 수정 불가능하므로 수정/삭제 처리가 복잡(비트 배열로 삭제 표시, Merge 시 실제 삭제).

</details>

### Q3. 루씬에서 Flush, Commit, Merge의 차이를 설명하고, ES에서 각각에 해당하는 용어를 말하라.

<details>
<summary>정답 보기</summary>

| 루씬 작업 | 설명 | ES 용어 |
|-----------|------|---------|
| **Flush** | 인메모리 버퍼 → 세그먼트 생성 → `write()`로 OS 커널 캐시에 기록. 검색 가능해짐. 물리 디스크 보장 없음. | **Refresh** |
| **Commit** | `fsync()`로 OS 커널 캐시 → 물리 디스크 동기화. 무거운 작업. 영구 보존 보장. | **Flush** |
| **Merge** | 다수의 세그먼트를 하나로 합침. 삭제 데이터의 물리 삭제. Commit 동반. 검색 성능 향상. | **Optimize API / Force Merge API** |

ES의 기본 Refresh 주기: **1초** (NRT 검색의 근거)
ES의 기본 Flush 주기: **5초** 또는 Translog 일정 크기 초과 시

</details>

### Q4. Translog의 역할과 동작 순서를 설명하라.

<details>
<summary>정답 보기</summary>

**역할**: ES 샤드의 장애 복구를 위한 특수 파일. 루씬 Commit 전 장애 발생 시 데이터 유실 방지.

**동작 순서**:
1. 색인 요청 → Translog에 먼저 기록
2. 루씬 인메모리 버퍼로 데이터 전달
3. Refresh(1초) → 인메모리 버퍼 → 세그먼트 생성 → 검색 가능. Translog는 계속 유지.
4. Translog가 일정 크기 이상 → Flush 수행
5. 커밋 포인트 디스크 기록 → 시스템 캐시 물리 디스크 동기화 → Translog 내역 삭제

**복구 활용**: 직전 루씬 Commit 이후 모든 내역이 Translog에 기록되어 있으므로, 샤드 크래시 시 Translog 재실행으로 완전 복구 가능.

**주의**: Translog 크기가 클수록 복구 시간 증가. 적절한 Flush 주기 설정 필요.

</details>

### Q5. `_seq_no`와 `_primary_term`이 필요한 이유를 설명하라.

<details>
<summary>정답 보기</summary>

**`_seq_no` 필요 이유**: 분산 환경에서 복제 요청의 순서가 역전될 수 있다. 예를 들어 views=1로 색인 후 views=2로 색인했는데 복제본 샤드에 views=2가 먼저 도착하고 views=1이 나중에 도착하면 최종값이 역전된다. `_seq_no`가 더 작은 요청은 현재 문서보다 이전 버전이므로 무시함으로써 이 문제를 해결한다.

**`_primary_term` 필요 이유**: 주 샤드 노드에 장애가 생겨 복제본 샤드가 새 주 샤드로 승격될 때, 이전 주 샤드의 작업과 새 주 샤드의 작업을 구분해야 한다. 주 샤드가 새로 지정될 때마다 `_primary_term`이 1씩 증가한다.

이 두 값의 조합이 ES의 **낙관적 동시성 제어(Optimistic Concurrency Control)**의 핵심이다.

</details>

## 응용 (40%)

### Q6. Query Phase와 Fetch Phase가 왜 분리되어 있는지 설명하고, 각 단계에서 반환하는 데이터의 차이를 말하라.

<details>
<summary>정답 보기</summary>

**분리 이유**: 효율성 때문이다. 예를 들어 1000개 샤드에서 각 10개씩 상위 문서를 뽑아야 한다면 Query Phase에서 10,000개의 결과가 조정 노드에 모이고, 이를 병합해 실제로 반환할 최상위 10개만 결정한다. 그 10개 문서의 실제 내용만 Fetch Phase에서 요청한다. 만약 처음부터 _source를 포함해 10,000개를 모두 전송하면 네트워크 비용이 막대하다.

**각 단계 반환 데이터**:
- **Query Phase**: 쿼리에 매칭된 상위 문서의 **docId + 유사도 점수** 목록. _source 없음.
- **Fetch Phase**: 확정된 최종 상위 N개 문서의 **실제 내용** (`_source`, 메타데이터 등).

**샤드 레벨 요청 캐시**도 이 분리를 반영한다. QueryPhase 결과만 캐시하며 FetchPhase는 항상 실제 읽기가 필요하다.

</details>

### Q7. 글로벌 체크포인트와 로컬 체크포인트의 차이를 설명하고, 샤드 복구 시 어떻게 활용되는지 말하라.

<details>
<summary>정답 보기</summary>

**로컬 체크포인트**: 각 샤드(주/복제본)가 seq_no 기준으로 **빠짐없이 순차적으로 완료**한 마지막 번호. 예) seq_no 1,2,3,5,7을 완료 → 로컬 체크포인트 = 3 (4번이 누락됐으므로). 완료될 때마다 주 샤드에 보고.

**글로벌 체크포인트**: 주 샤드가 모든 복제본의 로컬 체크포인트를 비교해 **가장 낮은 값**으로 결정. 이 번호까지는 모든 샤드에 반영이 완료됐음을 보장.

**복구 활용**:
1. 주 샤드와 복구 대상 샤드의 글로벌 체크포인트 비교
2. 같으면 추가 복구 불필요
3. 다르면 체크포인트 이후 작업만 재처리 → 세그먼트 파일 통째로 복사보다 훨씬 효율적

단, 샤드 이력 보존 기간(기본 12시간)이 지난 경우에는 작업 재처리 대신 세그먼트 파일 통째로 복사.

</details>

### Q8. 샤드 레벨 요청 캐시와 노드 레벨 쿼리 캐시의 차이를 설명하고, 언제 어떤 캐시를 활용해야 하는지 설명하라.

<details>
<summary>정답 보기</summary>

**샤드 레벨 요청 캐시**:
- 캐시 키: 검색 요청 전체(완전히 같은 요청이어야 적중)
- 캐시 값: QueryPhase 전체 결과(매칭 문서 목록, 점수, 집계 결과 등)
- 기본 조건: `size=0` 또는 `requestCache=true` 명시
- 무효화: Refresh 시

**노드 레벨 쿼리 캐시**:
- 캐시 키: 루씬 Query 인스턴스(equals/hashCode 기준)
- 캐시 값: 쿼리에 매칭된 문서의 비트 배열(DocIdSet)
- 조건: 필터 문맥 쿼리 + 세그먼트에 10,000개+ 문서 + 충분한 수행 빈도
- bool 쿼리에서 필터 문맥 하위 쿼리 일부가 겹쳐도 부분 캐시 혜택

**활용 지침**:
- **집계 중심 서비스**: 요청 캐시가 효과적. `size=0` 집계 쿼리가 반복될 때.
- **일반 검색 반복**: `requestCache=true` 명시 + QueryPhase 결과 캐시. FetchPhase는 여전히 수행됨.
- **필터 조건 재사용**: bool 쿼리의 filter 절을 최대한 활용 → 노드 레벨 쿼리 캐시 혜택.
- **같은 노드 우선**: `preference=<해시문자열>`로 같은 노드를 우선 → 캐시 적중률 향상.

</details>

### Q9. conjunction(AND) 검색에서 DocIdSetIterator를 활용한 최적화 메커니즘을 설명하라.

<details>
<summary>정답 보기</summary>

**핵심 원리**: AND 조건에서 한 DISI가 doc ID N을 가리키면, 다른 DISI는 N보다 작은 문서를 검사할 필요가 없다. `advance(N)`으로 N 이상의 첫 번째 매칭 문서로 바로 건너뛸 수 있다.

**구체적 동작**:
1. 각 하위 쿼리의 DocIdSetIterator를 `cost()`(매칭 추정 문서 수) 기준 오름차순 정렬. cost가 가장 작은 것이 lead1, 두 번째가 lead2, 나머지는 others.
2. lead1.advance(target) 수행 → lead1의 현재 docId를 doc으로 저장.
3. lead2.advance(doc) 수행 → lead2의 docId가 doc과 같지 않으면 lead1을 다시 advance. 두 DISI가 같은 문서를 가리킬 때까지 번갈아 advance.
4. lead1과 lead2가 같은 문서를 가리키면 others의 DISI도 순서대로 advance.
5. 모든 DISI가 같은 문서를 가리키면 → 그 문서가 모든 조건에 매칭.

**효율성**: 공통으로 매칭될 가능성이 낮은 조건(cost가 작은 것)을 먼저 기준으로 삼아 필요없는 문서를 최대한 건너뜀.

</details>

## 심화 (20%)

### Q10. `search_type=dfs_query_then_fetch`가 필요한 상황과 그 단점을 설명하라.

<details>
<summary>정답 보기</summary>

**필요한 상황**: 샤드 수가 많고 각 샤드의 문서 수가 불균일할 때. 기본 `query_then_fetch`는 각 샤드가 자기 로컬 통계(텀 빈도 등)로 유사도 점수를 계산하므로, 샤드마다 통계가 달라 점수 편차가 발생한다. 특히 소규모 데이터에서 특정 샤드에 특정 용어가 집중된 경우 점수가 왜곡될 수 있다.

**DFS(Distributed Frequency Search) 동작**:
1. DfsPhase: 모든 샤드에서 텀 빈도, 문서 빈도 등 통계 데이터를 사전 수집 → 조정 노드에서 전역 통계 합산.
2. 각 샤드는 이 전역 통계로 유사도 점수 계산 → 모든 샤드 간 점수 일관성 보장.

**단점**:
- 추가 라운드트립(DfsPhase) 발생 → 지연 시간 증가.
- 각 샤드에서 추가 작업 수행 → CPU, 네트워크 비용 증가.
- 실제로 데이터가 충분히 많고 샤드 간 문서 수가 고르면 기본 방식과 결과 차이가 크지 않음.

**권장**: 일반적으로는 기본 `query_then_fetch`를 사용. 점수 정확도가 중요하고 성능 비용을 감당할 수 있을 때만 `dfs_query_then_fetch` 사용.

</details>

### Q11. ES 인덱스의 주 샤드 수를 운영 중에 변경할 수 없는 근본적인 이유를 루씬 세그먼트 특성과 연결해 설명하라.

<details>
<summary>정답 보기</summary>

**근본 원인**: 루씬 세그먼트의 **불변성(Immutability)**.

각 샤드는 내부에 독립적인 루씬 인스턴스를 가진다. 루씬은 Stand-Alone 검색엔진이므로 자신의 샤드에만 집중하고, 다른 샤드의 존재를 모른다.

만약 샤드 수를 5개에서 10개로 늘린다면:
1. 기존 5개 루씬 인스턴스가 각각 보유하던 세그먼트를 분할해 새 5개 루씬으로 이전해야 함.
2. 그런데 세그먼트는 불변이므로 분할 자체가 불가능.
3. 억지로 분할하면 역색인 구조를 처음부터 다시 만들어야 함 → 사실상 전체 재색인과 동일한 비용.

따라서 ES는 이를 허용하지 않고 Reindex API로 새 인덱스를 만들어 전체 재색인하는 방법만 제공한다.

**레플리카는 변경 가능한 이유**: 레플리카 추가는 기존 주 샤드를 단순 복사만 하면 되므로 세그먼트 불변성을 깨지 않는다.

</details>

### Q12. 노드 레벨 쿼리 캐시가 TermQuery를 캐시하지 않는 이유를 설명하고, 노드 레벨 쿼리 캐시 적중률을 높이기 위한 전략을 설명하라.

<details>
<summary>정답 보기</summary>

**TermQuery 캐시 제외 이유**: 루씬이 TermQuery를 "캐시 없이도 충분히 빠른 쿼리"로 판단하기 때문이다. TermQuery는 역색인을 직접 조회하므로 이미 매우 빠르다. 캐시를 적재하고 관리하는 오버헤드가 성능 이득보다 클 수 있어 캐시에서 제외된다. 마찬가지로 MatchAllDocsQuery, MatchNoDocsQuery, DocValuesFieldExistsQuery도 제외된다.

**적중률 향상 전략**:
1. **필터 문맥 최대 활용**: bool 쿼리에서 점수와 무관한 조건은 반드시 filter 절로. 유사도 점수 계산 불필요 쿼리만 캐시되므로 filter 절 사용이 핵심.
2. **쿼리 구조 일관성**: 캐시 키가 Query 인스턴스이므로 같은 조건은 항상 같은 구조로 생성되어야 적중. 동적 파라미터 위치 고정.
3. **세그먼트 크기 확보**: 10,000개 이상 문서를 가진 세그먼트에서만 캐시. 샤드 크기를 너무 작게 하면 캐시 조건 미달.
4. **preference 활용**: 같은 문자열 preference를 지정해 같은 노드로 요청 유도 → 캐시 재사용.
5. **Refresh 빈도 관리**: 캐시는 Refresh 시 무효화. 불필요한 Refresh 최소화.

</details>
