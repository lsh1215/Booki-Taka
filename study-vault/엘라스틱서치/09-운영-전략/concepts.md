## 1. Alias 전략

### 정의
alias는 이미 존재하는 인덱스를 다른 이름으로도 가리키도록 하는 기능이다. 하나의 alias가 하나 이상의 인덱스를 가리키도록 지정할 수 있다. 엘라스틱서치에서 인덱스 이름을 지정해야 하는 거의 모든 곳에는 인덱스 이름 대신 alias를 넣을 수 있다.

### 왜 필요한가
운영 중에 인덱스의 매핑이나 설정을 크게 변경해야 할 때, 서비스 코드를 수정하지 않고 인덱스를 교체할 수 있다. 서비스에서는 실제 인덱스 이름이 아니라 alias를 가리키도록 설계하면, alias가 가리키는 인덱스만 변경하면 운영 중에 새 인덱스로 넘어갈 수 있다.

단순 로그성 데이터가 아니라 서비스에 직접 활용되는 데이터를 들고 있는 인덱스나, 요건 등이 변할 것이 예상되는 인덱스라면 모두 alias를 사전에 지정하는 것이 중요하다.

### 동작 원리

**alias 지정**
```json
POST _aliases
{
  "actions": [
    {
      "add": {
        "index": "my_index",
        "alias": "my_alias_name"
      }
    },
    {
      "add": {
        "index": "my_index2",
        "alias": "my_alias_name",
        "is_write_index": true
      }
    }
  ]
}
```

- 여러 인덱스를 가리키는 alias로 검색하면 모든 인덱스를 대상으로 검색
- 여러 인덱스를 가리키는 alias는 단건 문서 조회 작업의 대상이 될 수 없음 (단일 인덱스 alias여야 함)
- 업데이트, 삭제 등 쓰기 작업은 `is_write_index`를 true로 지정한 인덱스 대상으로 작업
- 여러 인덱스를 가리키는 alias 안에 `is_write_index`를 지정한 인덱스가 없으면 쓰기 작업 불가

**alias 해제**
```json
POST _aliases
{
  "actions": [
    {
      "remove": {
        "index": "my_index2",
        "alias": "my_alias_name"
      }
    }
  ]
}
```

**alias 조회**
```
GET _alias/my_alias_name
```

**실무 패턴: 무중단 인덱스 교체**
1. `service-data-v1` 인덱스 생성, `service-data` alias가 `service-data-v1`을 가리키도록 설정
2. 모든 서비스는 `service-data` alias 대상으로 색인, 조회, 검색
3. 인덱스에 큰 변화가 필요할 때 `service-data-v2` 새로 생성
4. `service-data` alias가 `service-data-v1` 대신 `service-data-v2`를 가리키도록 변경
5. 새 인덱스 생성에는 reindex API 활용

### 설계 트레이드오프
| 관점 | 내용 |
|---|---|
| 장점 | 무중단 인덱스 교체, 서비스 코드 변경 없음, 여러 인덱스를 단일 엔드포인트로 검색 |
| 단점 | 여러 인덱스 alias의 쓰기 작업은 is_write_index 필수, 단건 조회 불가 |
| 필수 조건 | reindex를 위해서 미리 alias를 설정해 두는 것을 잊지 않아야 함 |

## 2. 롤오버 (Rollover)

### 정의
alias를 활용할 때 하나의 alias에 여러 인덱스를 묶고 한 인덱스에만 `is_write_true`를 지정하는 구성에서, 쓰기를 담당하는 인덱스 내 샤드의 크기가 너무 커지면 새로운 인덱스를 생성해서 같은 alias 안에 묶은 뒤 `is_write_true`를 새 인덱스로 옮기는 방식으로 운영한다. 롤오버는 이러한 작업을 한 번에 묶어서 수행하는 기능이다.

### 왜 필요한가
대용량 시계열 데이터를 단일 인덱스에 계속 색인하면 샤드 크기가 과도하게 커져 검색 성능이 저하된다. 롤오버를 사용하면 인덱스를 자동으로 교체하여 적정 크기를 유지할 수 있다.

### 동작 원리

**롤오버 수행**
```
POST [롤오버 대상]/_rollover
```

롤오버 대상으로는 alias 이름 또는 데이터 스트림 이름이 들어간다.

**인덱스 이름 규칙**
- 롤오버를 수행할 alias 내 `is_write_true` 인덱스의 이름은 반드시 `.*-\d+$` 패턴을 따라야 함
- `test-index-000001`: 롤오버 가능
- `test-index`: 롤오버 수행 불가
- `test-index-000001`에 롤오버 수행하면 `test-index-000002`가 자동 생성, `is_write_true`가 넘어감
- 숫자 부분은 반드시 6자리일 필요 없으나, 롤오버가 자동 생성하는 이름은 6자리로 맞춤

**날짜 포함 인덱스 이름 생성**
```
PUT %3Ctest-index-%7Bnow%2Fd%7D-000001%3E
```
`<test-index-{now/d}-000001>` URL 인코딩. 생성 시각에 맞춰 `test-index-2022.01.09-000001`과 같은 이름으로 생성됨.

**새 인덱스 이름 직접 명시**
```
POST [롤오버 대상 alias]/_rollover/[생성할 새 인덱스 이름]
```
이 경우 기존 `is_write_true` 인덱스와 새 인덱스 이름 모두 특별한 형식을 따를 필요 없음.

### 설계 트레이드오프
| 관점 | 내용 |
|---|---|
| ILM 연동 | 인덱스 생명주기 관리 기능과 묶어 활용하면 조건 기반 자동 롤오버 가능 |
| 이름 규칙 | alias 기반 롤오버는 `.*-\d+$` 패턴 필수, 데이터 스트림은 이름 직접 지정 불가 |
| 자동화 | ILM의 롤오버 액션에 max_age, max_primary_shard_size 등 조건 지정 가능 |

## 3. 데이터 스트림 (Data Stream)

### 정의
데이터 스트림은 내부적으로 여러 개의 인덱스(뒷받침 인덱스, backing indices)로 구성된다. 검색을 수행할 때는 해당 데이터 스트림에 포함된 모든 인덱스를 대상으로 검색하고, 문서를 추가 색인할 때는 가장 최근에 생성된 단일 인덱스에 새 문서가 들어간다.

데이터 스트림은 마치 여러 인덱스를 묶고 `is_write_true` 인덱스를 하나 둔 alias 구성과 유사하게 동작한다.

### 왜 필요한가
alias를 인덱스 템플릿과 연계해서 시계열 데이터 사용 패턴에 맞게 정형화하고 간단하게 사용할 수 있도록 정제한 것이 데이터 스트림이다. ILM과의 연계를 염두에 두고 개발된 기능이다.

### 동작 원리

**뒷받침 인덱스 특성**
- 모두 hidden 속성
- 이름 패턴 고정: `.ds-<데이터 스트림 이름>-<yyyy.MM.dd>-<세대 수>`
- 세대 수: 000001부터 시작해 1씩 증가하는 0 패딩된 6자리 자연수
- 롤오버 시 이름 직접 지정 불가

**alias와의 차이점**
- 반드시 인덱스 템플릿과 연계해서 생성해야 함
- 문서 추가는 가능하지만 업데이트 작업 불가
- 반드시 `@timestamp` 필드(date 또는 date_nanos 타입)가 포함된 문서만 취급

**생성 절차**
1. ILM 정책 생성 (선택이나 권장)
2. 컴포넌트 템플릿 생성 (매핑, 설정)
3. `data_stream: {}` 포함한 인덱스 템플릿 생성
4. 데이터 스트림 생성

```
PUT _data_stream/my-data-stream-first
```

**문서 추가** (`_create` 사용)
```json
PUT my-data-stream-first/_create/1
{
  "hello": "world",
  "@timestamp": "2022-01-09T13:49:45.003Z"
}
```

**업데이트/삭제**: `update_by_query` 또는 `delete_by_query` 사용, 또는 뒷받침 인덱스 이름과 `_id` 직접 지정

**삭제**
```
DELETE _data_stream/my-data-stream-first
```

### 설계 트레이드오프
| 관점 | 내용 |
|---|---|
| 적합한 용도 | 모니터링용 지표, 로그 등 버려도 큰 문제 없는 시계열 데이터 |
| 부적합한 용도 | 장애로 인한 특정 시간대 데이터를 재처리해야 하는 경우 (역등하게 재처리하기 어려움) |
| 제약 | 업데이트 불가, @timestamp 필수, 인덱스 템플릿 필수 |

## 4. 인덱스 생명주기 관리 (ILM)

### 정의
Index Lifecycle Management(ILM)는 인덱스를 hot-warm-cold-frozen-delete 페이즈로 구분해서 지정한 기간이 지나면 인덱스를 다음 페이즈로 전환시키고 이때 지정한 작업을 수행하도록 하는 기능이다.

### 왜 필요한가
대규모 시계열 데이터를 운영할 때 오래된 데이터는 성능이 낮은 노드로 옮기고 결국 삭제하는 일련의 과정을 자동화할 수 있다. 롤오버, 세그먼트 병합, 데이터 티어 이동, 삭제 등을 정책으로 정의해 자동 운영한다.

### 동작 원리

**각 페이즈 콘셉트**
| 페이즈 | 상태 |
|---|---|
| hot | 현재 업데이트가 수행되고 있고 읽기 작업도 가장 많은 상태 |
| warm | 인덱스에 더 이상 업데이트가 수행되지는 않지만 읽기 작업은 들어오는 상태 |
| cold | 업데이트 없음, 읽기 작업도 가끔씩. 검색은 되어야 하나 속도가 느려도 괜찮은 상황 |
| frozen | 업데이트 없음, 읽기 작업도 거의 없음. 속도가 상당히 느려도 괜찮은 상황 (엔터프라이즈 등급) |
| delete | 인덱스가 더 이상 필요 없고 삭제되어도 무방한 상태 |

**페이즈 전환**
- 시간 조건 기반: 인덱스가 생성된 이후 얼마만큼의 시간이 지났는지
- 현재 페이즈에서 수행해야 하는 액션을 모두 끝냈는지 체크
- 두 조건 모두 만족하면 다음 페이즈로 전환
- 인덱스에 미할당 샤드가 남아 yellow 상태이더라도 다음 페이즈로 전환됨

**페이즈 액션 확인 주기**
- `indices.lifecycle.poll_interval` 설정 (기본값: 10분)
- 경쟁 상태 회피를 위해 작업 하나를 여러 세부 단계로 쪼개 수행 가능

**페이즈별 액션 종류**

| 액션 | 가능한 페이즈 | 설명 |
|---|---|---|
| 롤오버 | hot | 지정된 조건 만족 시 롤오버 수행 |
| 읽기 전용으로 만들기 | hot(롤오버 함께), warm, cold | 인덱스를 읽기 전용으로 변경 |
| 세그먼트 병합 | hot(롤오버 함께), warm | 세그먼트를 지정한 개수로 강제 병합 |
| shrink | hot(롤오버 함께), warm | 읽기 전용 후 shrink 수행해 새 인덱스 생성 |
| 인덱스 우선순위 변경 | hot, warm, cold | 복구 우선순위 변경 (값이 클수록 우선 복구) |
| 할당 | warm, cold | 노드 속성 지정 통한 샤드 할당 필터링 변경 |
| migrate | warm, cold | 데이터 티어에 맞는 노드로 인덱스 이동 |
| 동결 | cold | 인덱스 동결 (7.7 이후 지원 중단 선언) |
| 스냅샷 대기 | delete | 지정 SLM 정책으로 스냅샷 백업 완료까지 대기 후 삭제 |
| 삭제 | delete | 인덱스 삭제 |

**ILM 정책 생성 예시**
```
PUT _ilm/policy/test-ilm-policy
{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_primary_shard_size": "4gb",
            "max_age": "1d"
          }
        }
      },
      "delete": {
        "min_age": "15d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

**인덱스에 ILM 정책 적용**
- 인덱스 설정에 `index.lifecycle.name` 설정
- 인덱스 템플릿을 통해 자동 적용 가능 (데이터 스트림과 연계 시 권장)

**주의사항**
- 각 페이즈에 진입하면 ILM은 페이즈 정의를 인덱스 메타데이터에 캐싱
- 정책 업데이트 시 변경점이 안전하게 적용 가능한 종류라면 캐싱된 페이즈 정의도 업데이트

### 설계 트레이드오프
| 관점 | 내용 |
|---|---|
| 자동화 수준 | 롤오버, 이동, 삭제까지 정책으로 자동 운영 가능 |
| 페이즈 자유도 | warm에도 업데이트 허용 가능, hot 다음 바로 delete도 가능 - 반드시 콘셉트에 맞출 필요 없음 |
| SLM 연동 | delete 페이즈에서 스냅샷 대기 액션으로 안전하게 삭제 가능 |
| frozen | 엔터프라이즈 등급에서만 사용 가능 (검색 가능한 스냅샷 변환) |

## 5. 스냅샷 생명주기 관리 (SLM)

### 정의
Snapshot Lifecycle Management(SLM)는 지정한 시각에 지정한 내용의 스냅샷을 찍고 지정한 기간보다 오래된 스냅샷은 삭제하는 등의 작업을 자동으로 수행하는 정책을 등록해 스냅샷을 관리하는 기능이다.

### 왜 필요한가
스냅샷은 동작 중인 엘라스틱서치 클러스터의 인덱스를 백업하는 유일한 공식 방법이다. 파일 시스템 레벨의 단순 복사는 데이터 정합성을 깨뜨릴 수 있어 공식 지원하지 않는다. SLM을 통해 스냅샷 생성과 삭제를 자동화할 수 있다.

### 동작 원리

**스냅샷 저장소 등록**
스냅샷을 찍으려면 먼저 스냅샷 저장소를 등록해야 한다. 지원 저장소:
- 공유 파일 시스템(fs): NFS 등 공유 파일 시스템
- HDFS: 공식 플러그인 설치 필요
- Amazon S3, Microsoft Azure, Google Cloud Storage: 공식 플러그인 제공
- 소스 전용 저장소(source): 필드 값과 인덱스 메타데이터만 저장 (최소 크기)

```json
PUT _snapshot/[저장소 이름]
{
  "type": "fs",
  "settings": {
    "location": "/my/nfs/mount/path"
  }
}
```

`elasticsearch.yml`에 `path.repo` 설정으로 사전 등록 필요:
```yaml
path:
  repo:
    - /my/nfs/mount/path
```

**스냅샷 생성**
```
PUT _snapshot/[저장소 이름]/[스냅샷 이름]?wait_for_completion=false
{
  "indices": "my-index-202108*, my-index-202109*"
}
```

**스냅샷 조회**
```
GET _snapshot/[저장소 이름]/[스냅샷 이름]
```
스냅샷 이름에 `_all` 지정하면 저장소 내 모든 스냅샷 조회.

**스냅샷 복구**
```json
POST _snapshot/[저장소 이름]/[스냅샷 이름]/_restore
{
  "indices": "some-index-2021080*",
  "include_global_state": false
}
```

`include_global_state: true`이면 persistent 클러스터 설정, 레거시 아닌 인덱스 템플릿, 인제스트 파이프라인, ILM 정책을 완전히 삭제 후 스냅샷 내용으로 대체 (주의 필요).

**저장소 분리 (성능 고려)**
- 스냅샷은 증분 백업 방식: 작업 시작 시 저장소 내 모든 스냅샷 정보를 메모리로 올리는 선행 작업 수행
- 저장소에 스냅샷이 많을수록 느려짐
- 기본 저장소당 최대 스냅샷 개수: 500개
- 너무 무거운 저장소는 마스터 노드에 부담 - 대량 스냅샷은 다른 저장소 추가해 나눠 관리

**SLM 정책 생성**
```json
PUT _slm/policy/[정책 이름]
{
  "name": "<my-daily-snapshot-{now/d}>",
  "schedule": "0 30 17 * * ?",
  "repository": "fs-repo",
  "config": {
    "indices": ["<my-index-{now/d-2d}-*>"],
    "include_global_state": false
  },
  "retention": {
    "expire_after": "7d",
    "max_count": 100
  }
}
```

- `schedule`: cron 문법, UTC 기준
- `retention.expire_after`: 스냅샷 유지 기간
- `retention.max_count`: 최대 유지 개수

**만료된 스냅샷 삭제 스케줄 설정**
```json
PUT _cluster/settings
{
  "persistent": {
    "slm.retention.schedule": "0 30 10 * * ?"
  }
}
```
기본값: `0 30 1 * * ?`. SLM 정책으로 생성한 스냅샷만 삭제 (수동 스냅샷은 삭제 안 됨).

### 설계 트레이드오프
| 관점 | 내용 |
|---|---|
| 하위 호환성 | 메이저 버전이 둘 이상 차이나면 스냅샷 호환 불가 (5버전 스냅샷을 7버전에서 복구 불가) |
| 임시 해결 | 중간 버전 임시 클러스터를 세팅해 복구 후 재작업 필요 |
| 전역 상태 | include_global_state=false 기본값으로 복구하는 것이 안전 |

## 6. Reindex 전략

### 정의
reindex는 원본 인덱스 내 문서의 `_source`를 읽어서 대상 인덱스에 새로 색인하는 작업이다.

### 왜 필요한가
- 인덱스의 매핑이나 설정을 크게 변경해야 할 때 (기존 인덱스는 매핑 변경 불가)
- 메이저 버전 업그레이드 전 이전 메이저 버전에서 생성된 인덱스를 새로 만들 때
- `_source`가 비활성화되면 reindex도 불가능하고 버전 업그레이드도 어려워짐 - 기본값 유지 필수

### 동작 원리

**기본 reindex**
```json
POST _reindex
{
  "source": {
    "index": "source_index"
  },
  "dest": {
    "index": "target-index",
    "conflicts": "abort"
  }
}
```
`conflicts: proceed`로 지정하면 충돌 발생 문서는 건너뛰고 진행.

**검색 쿼리로 대상 제한**
```json
POST _reindex
{
  "source": {
    "index": "source_index",
    "query": { ... }
  },
  "dest": {
    "index": "target-index"
  }
}
```

**Painless 스크립트 지정**
재인덱싱 작업의 상세 동작을 변경 가능 (필드 변환, 추가 등).

**비동기 실행**
`wait_for_completion=false`로 비동기 요청. `tasks API`로 진척 확인 및 취소 가능.

**스로틀링**: `requests_per_second` 설정

**슬라이싱**: `slices` 설정으로 병렬 처리

**shrink vs reindex**
| 구분 | reindex | shrink |
|---|---|---|
| 방식 | 문서를 새로 색인 | 세그먼트를 하드 링크 |
| 제약 | 상대적으로 자유로움 | 읽기 전용, 한 노드에 모든 샤드, green 상태 필수 |
| 매핑 | 새 설정으로 변경 가능 | 샤드 개수 외 기존 설정 유지 |
| 결론 | 운영 중 안전한 선택 | ILM 내 warm 전환 시 자동 사용 고려 |

**split (샤드 개수 늘리기)**
- 원본의 세그먼트를 하드 링크, 인덱스 생성 당시 `index.number_of_routing_shards` 설정 필요
- `index.blocks.write: true` 읽기 전용 상태에서만 작업 가능
- 실무에서는 alias + reindex 조합이 더 많이 사용

### 설계 트레이드오프
| 관점 | 내용 |
|---|---|
| 무중단 | alias와 함께 사용하면 서비스 중단 없이 마이그레이션 가능 |
| 비용 | 대용량 데이터의 경우 비동기 + 스로틀링 + 슬라이싱으로 부하 조절 필수 |
| _source | 반드시 활성화 상태 유지 (비활성화 시 reindex, 업데이트, 버전 업그레이드 모두 불가) |

## 7. 샤드 운영 전략

### 적정 샤드 크기와 개수

#### 공식 가이드라인 (바이블 기준)
- 샤드 하나당 20~40GB 크기가 적절하다고 소개하나, 실제 운영 경험상 20GB만 되어도 느리고 무거운 경우가 많다. **수 GB 내외 수준에서 조정하는 것이 좋다.**
- **힙 1GB당 샤드 20개 이하**: 약 32GB 힙 기준으로 노드당 640 샤드 이하

```
GET _cat/shards?v&s=store:desc    # 샤드 크기 확인
GET _cat/health?v                  # 전체 샤드 수 확인
GET _cat/allocation?v              # 노드별 샤드 할당 확인
```

#### 샤드 개수 결정 고려 요소

**모든 노드가 충분히 일을 하고 있는지**
- 특정 인덱스의 주 샤드와 복제본 샤드가 모든 노드에 고르게 퍼지도록 설정
- 노드 대수가 n대라면 `number_of_shards`를 n의 배수로 지정
- 단, 서비스상 자주 호출되지 않는 인덱스나 크기가 너무 작은 인덱스는 모든 노드에 배치할 필요 없음

**미래 데이터 증가 고려**
- 시계열 인덱스 이름 전략 사용 시: 템플릿의 `number_of_shards` 값을 봐 가며 변경 가능
- 단일 인덱스: 미래에 커질 데이터 사이즈를 넉넉히 예상해서 설정. 버틸 수 없으면 결국 reindex 필요

**테스트 수행**
- 서비스 환경마다 상황이 다름. 특히 성능에 민감한 인덱스라면 실제 서비스 조건과 유사한 조건으로 테스트 수행 후 결정

### 샤드 할당 필터링과 데이터 티어 구조

#### 노드 속성과 샤드 할당 의식 속성
특정 노드에만 샤드를 배분하고 싶을 때, 노드에 커스텀 속성을 지정하고 이를 샤드 할당 의식 속성(awareness attributes)으로 지정한다.

```yaml
# elasticsearch.yml
node.attr.my_rack_id: rack_one
```

```json
PUT _cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.awareness.attributes": "my_rack_id"
  }
}
```

같은 샤드의 주 샤드와 복제본 샤드를 서로 다른 부분 집합에 배분한다 (강제는 아님).

#### 클러스터 단위 샤드 할당 필터링
```json
PUT _cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.include.my_rack_id": "rack_one,rack_two"
  }
}
```
- `include`: 지정한 속성들 중 적어도 하나 이상을 가진 노드에 할당
- `require`: 지정한 속성들을 모두 가진 노드에 할당
- `exclude`: 지정한 속성들 중 하나라도 가진 노드에는 할당하지 않음

가장 전형적인 사용 예: 특정 노드를 클러스터에서 제거하기 전에 샤드를 먼저 이동시킬 때

#### 인덱스 단위 샤드 할당 필터링
```json
PUT my_index/_settings
{
  "index.routing.allocation.include.my_rack_id": "rack_one,rack_two"
}
```
인덱스 템플릿이나 ILM 정책과 연동해 관리하기 좋기 때문에 클러스터 단위보다 사용처가 넓다.

#### 데이터 티어 구조 (Hot-Warm-Cold-Frozen)
데이터 노드를 용도 및 성능별로 티어로 구분해 클러스터를 구성하는 방법이다. `node.roles`에 `data` 대신 전용 역할 지정:

| 역할 | 설명 |
|---|---|
| data_content | 시계열 아닌 데이터, 실시간 서비스용. 시스템 인덱스가 요구하므로 필수 |
| data_hot | 가장 최근 시계열 데이터. 업데이트/읽기 가장 많음. 데이터 스트림 새 인덱스 기본 배정 |
| data_warm | data_hot보다 오래된 인덱스. 드물게 작업, 성능 덜 요구 |
| data_cold | 업데이트 없는 읽기 전용 인덱스. 드물게 조회 |
| data_frozen | 검색 가능한 스냅샷으로 변환된 인덱스. 엔터프라이즈 등급에서만 사용 가능 |

**티어 배정**
```json
PUT my_index/_settings
{
  "index.routing.allocation.include._tier_preference": "data_warm,data_hot"
}
```
먼저 data_warm에 할당 시도, 실패하면 data_hot에 할당.

ILM의 migrate 액션이 자동으로 `_tier_preference` 설정을 변경해 티어 이동 수행.

## 8. 롤링 리스타트 절차

### 정의
클러스터를 운영하는 상태에서 노드를 순차적으로 하나씩 재기동하는 방법. 클러스터 전체를 한꺼번에 내리지 않고도 설정 변경, 플러그인 적용, 버전 업그레이드 등을 수행할 수 있다.

### 왜 필요한가
동적으로 변경할 수 없는 설정의 적용, 플러그인 설치나 삭제, 버전 업그레이드, 장애 상황에서 문제를 일으키는 노드 재기동 등 다양한 상황에서 필요하다.

### 절차 (바이블 6.5절 기준)

**1단계: 샤드 할당 비활성화**
노드 종료 시 클러스터가 복제본 샤드 재생성 작업을 수행하는 것을 방지.

```json
PUT _cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "primaries"
  }
}
```

`enable` 설정값:
- `all`: 모든 종류 샤드 할당 허용 (기본 상태)
- `primaries`: 주 샤드의 할당만 허용
- `new_primaries`: 새로 생성된 인덱스의 주 샤드 할당만 허용
- `none`: 모든 샤드 할당 불허

주의: 엘라스틱서치 6.7 미만 구버전은 `primaries` 지정 불가, `none`으로 수행해야 함.

데이터 노드가 아니라면 이 과정 생략 가능.

**2단계: flush 수행**
translog를 비우고 데이터를 디스크에 안전하게 기록. 재기동 시 샤드 복구 시간 단축.

```
POST _flush
```

주의: 엘라스틱서치 7.6 미만 구버전은 `synced flush` 수행:
```
POST _flush/synced
```
8버전부터는 synced flush 삭제됨.

**3단계: 노드 재기동**
- 프로세스를 kill하고 새로 띄움
- 반드시 기존 프로세스가 완전히 종료된 후 기동
- 클러스터 합류 확인: `GET _cat/nodes`
- 클러스터 상태 불량 시 직접 로그에서 `started` 확인

**4단계: 샤드 할당 활성화**
```json
PUT _cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "all"
  }
}
```

**5단계: green 상태까지 대기**
```
GET _cat/health?v
```

green이 되면 다시 샤드 할당 비활성화하고 다음 노드 재기동. 필요한 만큼 반복.

### 설계 트레이드오프
| 관점 | 내용 |
|---|---|
| flush 필요성 | 필수는 아니나 재기동 시 샤드 복구 비용 절감 가능 |
| 구버전 주의 | 6.7 미만은 primaries 불가, 7.6 미만은 synced flush 사용 |
| 데이터 노드 | 데이터 노드가 아니면 샤드 할당 제어 과정 생략 가능 |

## 9. 버전 업그레이드 전략

### 정의
엘라스틱서치 클러스터를 현재 버전에서 더 높은 버전으로 업그레이드하는 과정. 마이너 버전 업그레이드와 메이저 버전 업그레이드로 구분된다.

### 왜 필요한가
버전에 따라 동작 방식이 달라지고, 오래된 인덱스의 하위 호환성 제한이 있어 계획적으로 접근해야 한다.

### 동작 원리

**마이너 버전 업그레이드**
롤링 리스타트 방식으로 수행. 클러스터를 내리지 않고 노드를 순차적으로 업그레이드.

**메이저 버전 업그레이드**

메이저 버전이 올라가면 인덱스의 내부 구조도 많이 변경된다. **메이저 버전이 둘 이상 차이나는 클러스터에서 생성된 인덱스에 대해 하위 호환을 지원하지 않는다.**

예시:
- 5버전 → 6버전 업그레이드: 문제 없음
- 6버전 → 7버전 업그레이드: **5버전대에서 생성한 인덱스가 남아있으면 문제 발생**
  - 6버전에서 해당 인덱스 삭제 또는 reindex 후 7버전 업그레이드 필요

**스냅샷 하위 호환성**
- 스냅샷도 메이저 버전이 둘 이상 차이나면 하위 호환 불지원
- 5버전에서 생성한 스냅샷을 7버전에서 복구 불가
- 해결책: 6버전 임시 클러스터를 세팅해 5버전 스냅샷에서 복구 후 필요 작업 수행

**업그레이드 전 확인사항**
- 오래된 메이저 버전에서 생성된 인덱스 존재 여부 확인
- `_source` 활성화 상태 확인 (비활성화 시 reindex 불가)
- 클러스터 green 상태 확인

### 설계 트레이드오프
| 관점 | 내용 |
|---|---|
| 인덱스 호환 | 메이저 버전 2단계 이상 차이 시 재생성 필요 - alias와 reindex로 사전 준비 |
| 시계열 인덱스 | 오래된 인덱스가 자연스럽게 삭제되는 구조면 업그레이드 용이 |
| 스냅샷 | 오래된 스냅샷은 버전 호환 확인 필요 - 임시 중간 버전 클러스터 준비 고려 |
