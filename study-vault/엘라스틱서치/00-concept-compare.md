## 1. match vs term

### 구조적 차이

| 구분 | match | term |
|------|-------|------|
| 풀려는 문제 | 자연어 텍스트 검색 | 정확한 값 일치 검색 |
| 동작 원리 | 검색어에 애널라이저를 적용한 후 역색인에서 조회 | 검색어를 그대로(분석 없이) 역색인에서 조회 |
| 대상 필드 타입 | `text` | `keyword`, `integer`, `date` 등 |
| 대소문자 | 애널라이저 설정에 따라 처리 | 입력 그대로 비교 |
| 사용 시점 | "엘라스틱서치 검색"처럼 자연어를 입력받을 때 | status = "active", category = "book"처럼 코드/분류값을 비교할 때 |
| 트레이드오프 | 유연하지만 분석 오버헤드 있음 | 빠르고 정확하지만 분석 없으므로 텍스트 부분 일치 불가 |

### 왜 헷갈리는가

`term`은 "단어 하나와 매치"처럼 읽히지만 실제로는 "분석 없이 값 그대로 비교"를 의미한다. 반면 `match`는 "단어 하나씩 분리해서 매치"처럼 동작하므로, 이름과 동작이 서로 바뀐 것처럼 느껴진다.

또한 `text` 필드에 `term` 쿼리를 쓰면 색인 시 애널라이저가 소문자로 변환한 값을 저장하는데, 검색어는 원문 그대로 전달되어 대소문자 불일치로 결과가 나오지 않는 함정이 있다.

### 핵심 구분 기준

> **애널라이저가 개입하는가?** match는 O, term은 X.

## 2. Query Context vs Filter Context

### 구조적 차이

| 구분 | Query Context | Filter Context |
|------|--------------|----------------|
| 풀려는 문제 | "얼마나 관련성이 높은가" 순위 계산 | "조건을 만족하는가" Yes/No 판별 |
| 동작 원리 | 매칭 + BM25 점수(_score) 계산 | 매칭만, 점수 계산 없음 |
| 캐시 | 캐시되지 않음 (점수가 문서마다 다를 수 있음) | 결과를 bitset으로 캐시 (반복 실행 시 빠름) |
| 사용 시점 | 검색 결과 정렬이 필요한 핵심 검색 조건 | 날짜 범위, 상태값, 가격 범위 같은 필터링 조건 |
| bool 쿼리 위치 | `must`, `should` | `filter`, `must_not` |
| 트레이드오프 | 관련성 점수 제공, 느리고 캐시 안 됨 | 빠르고 캐시됨, 점수 없음 |

### 왜 헷갈리는가

`must`와 `filter` 둘 다 "조건을 충족해야 한다"는 의미처럼 보이지만, `must`는 점수를 계산하고 `filter`는 점수를 무시한다는 차이가 있다. "필터이니까 필터링(=조건 충족)만 하면 되는 것"이라고 이해하면 filter 절이 왜 더 빠른지 납득된다.

### 핵심 구분 기준

> **_score에 기여하는가?** Query Context는 O, Filter Context는 X.

- [05 검색 쿼리 - bool 쿼리와 컨텍스트](./05-검색-쿼리/concepts.md)
- [13 내부 동작 - BM25 스코어링](./13-내부-동작/concepts.md)

## 3. Primary Shard vs Replica Shard

### 구조적 차이

| 구분 | Primary Shard | Replica Shard |
|------|--------------|---------------|
| 풀려는 문제 | 데이터의 실제 저장소이자 쓰기 처리 주체 | 가용성 보장과 읽기 부하 분산 |
| 동작 원리 | 문서 색인 요청을 직접 처리한 후 Replica에 복제 | Primary 색인 완료 후 동일 데이터 복사본 유지 |
| 쓰기 | 가능 (유일한 쓰기 대상) | 불가. Primary → Replica 단방향 복제만 |
| 읽기 | 가능 | 가능 (읽기 부하 분산) |
| 장애 시 | Primary 장애 → Replica 중 하나가 Primary로 승격 | Replica 장애 → 서비스 영향 없음 (Primary는 정상) |
| 개수 변경 | 인덱스 생성 후 변경 불가 | 언제든지 동적 변경 가능 |
| 트레이드오프 | 수를 늘리면 병렬처리 향상, 줄이면 오버헤드 감소 | 수를 늘리면 가용성·읽기 성능 향상, 줄이면 저장소 절약 |

### 왜 헷갈리는가

둘 다 "샤드"이고 동일한 데이터를 가지고 있다. 그래서 읽기 관점에서는 구분이 의미 없어 보인다. 차이는 **쓰기 경로**에서 드러난다.

### 핵심 구분 기준

> **쓰기 요청을 직접 처리하는가?** Primary만 O.

## 4. Refresh vs Flush (ES 용어)

### 구조적 차이

| 구분 | Refresh | Flush |
|------|---------|-------|
| 풀려는 문제 | 색인한 문서를 검색 가능 상태로 만들기 | 데이터 영속성 보장 (재시작 후에도 유실 없음) |
| 동작 원리 | 인메모리 버퍼 → 파일 시스템 캐시 이동 (새 세그먼트 생성) | 파일 시스템 캐시 → 디스크 fsync + translog 초기화 |
| 비용 | 상대적으로 가벼움 | fsync 발생으로 무거움 |
| 기본 주기 | 1초 (`refresh_interval`) | 필요 시 자동 (translog 크기 초과 또는 시간 기준) |
| 직후 효과 | 문서가 검색됨 | 데이터가 디스크에 안전하게 저장됨 |
| 트레이드오프 | refresh_interval을 늘리면 색인 성능 향상 (세그먼트 생성 빈도 감소) | flush 빈도를 줄이면 translog가 커지지만 쓰기 성능 향상 |

### 왜 헷갈리는가

두 개념 모두 "메모리에서 디스크 방향으로 데이터가 이동한다"는 느낌이지만, refresh는 파일 시스템 캐시까지만 가고 flush는 진짜 디스크까지 간다는 차이가 있다. 또한 루씬 용어에도 flush가 있어 혼동된다(아래 5번 참조).

### 핵심 구분 기준

> **검색 가능성** → Refresh. **영속성(fsync)** → Flush.

## 5. 루씬 Flush vs ES Flush

### 구조적 차이

| 구분 | 루씬 Flush | ES Flush |
|------|-----------|---------|
| 주체 | 루씬 라이브러리 내부 | 엘라스틱서치 레이어 |
| 동작 | 인메모리 세그먼트를 파일 시스템에 쓰기 (새 세그먼트 파일 생성) | 파일 시스템 캐시 → 디스크 fsync + translog 비움 |
| ES 관점에서의 대응 | ES의 **Refresh**와 유사한 효과 (검색 가능 세그먼트 생성) | ES 고유 개념. 루씬에는 대응 개념 없음 |
| 검색 가능성 | O (루씬 flush 후 검색 가능) | O (flush 이전에도 refresh가 먼저 실행됨) |
| 디스크 영속성 | X (OS 페이지 캐시까지만) | O (fsync 포함) |

### 왜 헷갈리는가

ES 문서와 루씬 문서를 함께 읽으면 "flush"라는 단어가 다른 레이어에서 다른 의미로 사용된다. ES의 refresh = 루씬의 flush라는 대응 관계를 알지 못하면 매우 혼란스럽다.

### 핵심 구분 기준

> ES 문서의 **Refresh ≈ 루씬 Flush**. ES 문서의 **Flush = fsync + translog 비움**.

### 관련 개념

- [01 핵심 아키텍처 - 세그먼트와 translog](./01-핵심-아키텍처/concepts.md)
- [13 내부 동작 - 루씬 내부 구조](./13-내부-동작/concepts.md)

## 6. scroll vs search_after

### 구조적 차이

| 구분 | scroll | search_after |
|------|--------|--------------|
| 풀려는 문제 | 특정 시점 스냅샷 기반 전체 결과 순회 | 실시간 데이터 기반 커서 페이지네이션 |
| 동작 원리 | 최초 검색 시 컨텍스트(스냅샷)를 서버에 유지. scroll_id로 다음 페이지 요청 | 정렬 기준의 마지막 값(search_after)을 다음 요청에 포함 |
| 서버 상태 | Stateful (scroll context를 서버가 유지) | Stateless (클라이언트가 커서 값 보관) |
| 실시간 반영 | X (최초 스냅샷 고정) | O (요청 시점 데이터 반영) |
| 메모리 | scroll context 유지로 힙 소모 | 없음 |
| 사용 시점 | 데이터 마이그레이션, 일괄 처리 등 변경 없는 전체 순회 | 실시간 피드, 무한 스크롤 UI |
| 폐기 여부 | 7.x 이후 비권장 (Point in Time + search_after 권장) | 현행 권장 방식 |
| 트레이드오프 | 서버 메모리 부담, 유지 시간 설정 필요 | 중간 삽입/삭제 시 결과 중복/누락 가능성 있음 |

### 왜 헷갈리는가

둘 다 "10,000건 이상의 대용량 결과를 순회할 때 쓴다"는 목적이 같다. 차이는 서버가 상태를 유지하느냐(scroll) 클라이언트가 커서를 보관하느냐(search_after)에 있다.

### 핵심 구분 기준

> **배치/마이그레이션 → scroll (또는 PIT + search_after)**. **실시간 UI 페이지네이션 → search_after**.

## 7. alias vs data stream

### 구조적 차이

| 구분 | alias | data stream |
|------|-------|-------------|
| 풀려는 문제 | 여러 인덱스에 단일 이름을 부여해 무중단 전환 또는 통합 검색 | 시계열 데이터(로그, 메트릭)의 자동 Rollover와 ILM 관리 |
| 동작 원리 | 하나 이상의 인덱스를 가리키는 가상 이름. 수동으로 대상 인덱스 관리 | 내부적으로 backing index들로 구성. Rollover 조건 충족 시 자동으로 새 backing index 생성 |
| 쓰기 대상 | `is_write_index: true`로 지정한 인덱스 하나 | 가장 최근 backing index 자동 선택 |
| ILM 연동 | 별도 설정 필요 | 기본 설계에서 ILM과 긴밀하게 연동 |
| @timestamp 필드 | 요구하지 않음 | 필수 (모든 문서에 @timestamp 포함) |
| 사용 시점 | 버전 전환, 다중 인덱스 통합 검색, 범용 용도 | 로그, 이벤트, 메트릭 같은 시계열 데이터 |
| 트레이드오프 | 유연하지만 Rollover를 수동 관리해야 함 | 자동화가 강력하지만 @timestamp 필수 + 구조 변경이 어려움 |

### 왜 헷갈리는가

data stream도 내부적으로 alias를 사용한다. 그래서 `GET _data_stream/my-stream`을 보면 alias처럼 보이는 구조가 나온다. 차이는 data stream이 자동 Rollover와 @timestamp 강제를 추가로 제공한다는 점이다.

### 핵심 구분 기준

> **시계열(로그/메트릭) → data stream**. **범용/버전 전환 → alias**.

- [09 운영 전략 - alias, rollover, data stream](./09-운영-전략/concepts.md)

## 8. ILM vs SLM

### 구조적 차이

| 구분 | ILM (Index Lifecycle Management) | SLM (Snapshot Lifecycle Management) |
|------|----------------------------------|-------------------------------------|
| 풀려는 문제 | 인덱스를 핫→웜→콜드→삭제 단계별로 자동 관리 | 스냅샷(백업)을 주기적으로 자동 생성 및 보존 관리 |
| 관리 대상 | 인덱스 자체 (데이터의 저장 위치와 수명) | 스냅샷 (인덱스의 백업본) |
| 설정 위치 | `PUT _ilm/policy/{name}` | `PUT _slm/policy/{name}` |
| 주요 액션 | rollover, shrink, forcemerge, freeze, delete | 스냅샷 생성 주기, 최소/최대 보존 개수, 보존 기간 |
| 연동 | data stream, alias와 함께 사용 | 스냅샷 저장소(repository)와 함께 사용 |
| 복구 지원 | X (데이터 삭제 방지가 목적이 아님) | O (스냅샷으로 인덱스 복구) |
| 트레이드오프 | 설정이 복잡하지만 인덱스 관리 자동화 강력 | 간단하지만 복구 시간(RTO)은 인덱스 크기에 비례 |

### 왜 헷갈리는가

둘 다 "Lifecycle Management"이고 둘 다 자동화를 담당한다. 하지만 ILM은 **살아있는 인덱스의 상태 전이**를, SLM은 **백업 스냅샷의 생성과 삭제**를 관리한다는 근본적 차이가 있다.

### 핵심 구분 기준

> **인덱스 수명 관리 → ILM**. **백업 스냅샷 관리 → SLM**.

## 9. 메트릭 집계 vs 버킷 집계

### 구조적 차이

| 구분 | 메트릭 집계 (Metric Aggregation) | 버킷 집계 (Bucket Aggregation) |
|------|--------------------------------|-------------------------------|
| 풀려는 문제 | 문서 집합에서 단일 수치 계산 | 문서를 그룹(버킷)으로 분류 |
| 출력 | 숫자 값 (평균, 합계, 최대, 최소, 개수 등) | 버킷 목록 (각 버킷에 문서 포함) |
| 대표 유형 | avg, sum, max, min, cardinality, percentiles | terms, date_histogram, range, histogram |
| 중첩 가능 | 버킷 집계의 하위 집계로 사용 가능 | 버킷 집계 하위에 다른 버킷 집계 중첩 가능 |
| 사용 예 | "전체 주문의 평균 금액은?" | "카테고리별 문서 수는?" |
| 트레이드오프 | 단순하고 빠름 | 중첩 깊어질수록 메모리와 CPU 소모 증가 |

### 왜 헷갈리는가

실무에서는 항상 버킷 집계 안에 메트릭 집계를 중첩한다("카테고리별 평균 금액"). 이 조합이 자연스럽다 보니 둘을 별개로 구분하는 것을 잊기 쉽다.

### 핵심 구분 기준

> **숫자 하나 → 메트릭 집계**. **그룹으로 나누기 → 버킷 집계**.

## 10. doc_values vs fielddata

### 구조적 차이

| 구분 | doc_values | fielddata |
|------|-----------|-----------|
| 풀려는 문제 | 집계·정렬을 위한 열 지향 데이터 구조 | text 필드의 집계를 위한 임시 데이터 구조 |
| 저장 위치 | 디스크 (인덱싱 시점에 함께 생성) | JVM 힙 메모리 (쿼리 시점에 동적 생성) |
| 대상 필드 | `keyword`, `numeric`, `date`, `boolean` 등 | `text` 필드 전용 |
| 기본 활성화 | O (기본적으로 켜져 있음) | X (기본 비활성화. 명시적으로 켜야 함) |
| 성능 | 디스크에서 읽어 빠르고 안정적 | 힙 소모로 GC 압력, 대용량 시 OutOfMemory 위험 |
| 비활성화 시점 | 집계·정렬이 전혀 필요 없는 필드 (`doc_values: false`로 공간 절약) | 기본값이 비활성화이므로 text 필드 집계 시에만 활성화 |
| 트레이드오프 | 색인 시점 저장 비용, 집계 시 빠름 | 첫 쿼리 시 힙에 올라가는 지연(warm-up), 이후 캐시 활용 |

### 왜 헷갈리는가

둘 다 "집계를 위한 데이터 구조"이지만, 대상 필드 타입이 다르다. `keyword` 필드는 doc_values를 쓰고, `text` 필드는 fielddata를 써야 하는데, 이를 모르면 "text 필드를 집계하려 했더니 에러가 난다"는 상황에 빠진다.

### 핵심 구분 기준

> **keyword/numeric → doc_values (디스크, 기본 활성)**. **text 집계 → fielddata (힙, 기본 비활성)**.
