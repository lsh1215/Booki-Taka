## 1. 장애 발생 시 초기 대응 순서

### 정의
장애 발생 시 클러스터를 안정화하고 피해를 최소화하기 위한 표준 절차.

### 단계별 대응 흐름

**1단계: 빠른 기본 상황 파악**
```
GET _cat/health   → 클러스터 상태(red/yellow), 미할당 샤드 수, 노드 수 확인
GET _cat/nodes    → 누락 노드, 마스터 선출 여부, 부하/힙/메모리/CPU 확인
GET _nodes/stats  → fs 항목으로 디스크 사용량 확인
```
- 모니터링 시스템 응답이 없으면 직접 cat API로 확인
- 주요 체크포인트: 노드 이탈 여부 / 미할당 샤드 발생 여부 / 마스터 선출 상태

**2단계: 샤드 할당 비활성화 (가장 시급)**
```json
PUT _cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "primaries"
  }
}
```
- 노드가 빠지면 → ES가 복제본 샤드를 새로 할당 → 부하 가중 → 연쇄 장애 위험
- 옵션:
  - `none`: 모든 할당 차단 (가장 심각한 상황)
  - `primaries`: 주 샤드 할당만 허용 (범용적, 권장)
  - `new_primaries`: 새로 생성된 인덱스의 주 샤드만 허용
- 요청 실패 시(마스터 미선출, GC STW 등): 실패해도 다음 단계로 진행

**3단계: 클라이언트 트래픽 차단**
- 조정 전용 노드(coordinating-only node)를 내리는 방식이 가장 깔끔
- 읽기/쓰기 전용 조정 노드를 분리해두면 선택적 차단 가능
- 샤드 복구 중 색인이 계속 들어오면: 복구 중인 복제본에 변경분 재처리 필요 → 복구 지연

**4단계: 로그 확인 및 조치**
- 상황별 확인 경로:
  - HTTP 연결 불가 → 프로세스 상태 확인 (`ps`) → 서버 로그 확인
  - 마스터 미선출 → 마스터 후보 노드 서버 로그 확인
  - `GET _cat/nodes` 응답 지체 → STW 의심 → GC 로그 확인 → `kill -15` 후 재기동
  - STW가 심각하면 `kill -15`, 안 죽으면 즉시 `kill -9`

**5단계: 샤드 복구 대기**
```json
PUT _cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "all"
  }
}
```
- all로 변경 후 green 복귀 대기
- 복구 중 부하로 노드 추가 이탈 가능 → 7.4절 복구 전략 적용

**6단계: 상세 원인 분석**
- 모니터링 수치 / 서버 로그 / GC 로그 / 슬로우 로그 분석

### 왜 필요한가
연쇄 장애 방지: 노드 이탈 → 샤드 재할당 → 부하 가중 → 추가 노드 이탈의 도미노를 끊어야 함.

## 2. 과도한 요청 인입

### 정의
키바나를 통한 무거운 집계/대시보드 요청이 클러스터에 과부하를 유발하는 상황.

### 발생 패턴
- 통제되지 않은 사용자가 긴 기간 대시보드를 짧은 주기로 자동 리프레시
- 서비스 오픈/개편 직후 여러 명이 동시에 대시보드 접속
- 장애 상황 단체 대화방에 대시보드 링크 공유

### 대응
1. 임시로 키바나를 내리거나 조정 전용 노드를 내려서 추가 요청 차단
2. 슬로우 로그로 키바나 쿼리 여부 확인 (키바나 자동 생성 쿼리는 불필요한 속성이 많아 식별 용이)
3. 인입 주기, 무거움 정도 파악 후 대응

### 예방
- 키바나를 통제되지 않은 사용자에게 오픈하지 않음
- 사용자 사전 교육 및 협조 요청

## 3. GC로 인한 Stop-The-World

### 정의
Full GC 발생으로 JVM이 일정 시간 완전히 멈추는(Stop-The-World, STW) 상태.

### 왜 문제가 되는가
- STW 시간이 길어지면 노드가 클러스터에서 제외(timeout)
- OOM으로 노드 프로세스 사망 가능
- 노드 이탈 → 복제본 재할당 → 부하 가중 → 연쇄 STW 위험

### 원인
- `fielddata`를 이용한 집계/정렬
- 키바나에서 인입되는 무거운 집계 작업

### 대응
1. 샤드 할당 비활성화 (최우선)
2. 추가 트래픽 유입 차단
3. 문제 노드 재기동
   - 재기동 전 `ps`로 두 프로세스가 동시에 뜨지 않도록 확인
   - `kill -15` → 안 죽으면 즉시 `kill -9`
   - 의도적 롤링 리스타트와 달리 사전 flush 없이 바로 kill

### 버전 개선 사항
- G1GC 기본 적용 + 실제 메모리 서킷 브레이커가 추가된 7 버전 이상에서 크게 개선
- 7 버전 미만 사용 중이면 메이저 버전 업그레이드 강력 권장

## 4. 디스크 풀 상황

### 정의
디스크 사용량이 임계치를 넘어 인덱스가 read-only 상태로 전환되는 상황.

### 동작 원리
- 디스크 사용량이 **95%** 초과 → ES가 인덱스에 워터마크(watermark) 설정
- 워터마크가 설정된 인덱스: 읽기와 삭제만 가능, 색인 요청은 모두 거부
- 디스크에 여유가 생기면 **7.4 버전 이상**은 자동 해제
- **7.4 버전 이하**: 수동 해제 필요

### 대응 절차
1. 디스크 사용량이 급증한 원인 파악
   ```bash
   du -sh /path/to/data          # OS 레벨
   ```
   ```
   GET _cat/indices?v&s=store.size:desc    # 큰 인덱스 확인
   ```
2. 삭제 가능한 파일/인덱스 삭제
3. 워터마크 해제 (7.4 이하 또는 자동 해제 안 된 경우)
   ```json
   PUT [인덱스 이름]/_settings
   {
     "index": {
       "blocks": {
         "read_only_allow_delete": false
       }
     }
   }
   ```
   - `null`로 설정: 설정 항목 자체가 사라짐 (깔끔)
   - `false`로 설정: 워터마크가 붙었던 인덱스 추후 구분 가능
   - 와일드카드로 일괄 해제 가능: `PUT my-index-*/_settings`

### 예방
- 모니터링에서 디스크 사용량 알람 설정 필수
- ILM 정책으로 오래된 인덱스 자동 삭제

## 5. 미할당 샤드 처리

### 정의
샤드가 노드에 할당되지 못하고 unassigned 상태로 남아있는 상황.

### 동작 원리
샤드 할당 실패 원인이 해소되지 않으면 ES는 재시도를 반복하다가 최대 재시도 횟수를 소진하면 더 이상 시도하지 않음.

### 원인 파악
```
GET _cluster/allocation/explain?pretty
{
  "index": "my-index-20210811",
  "shard": 1,
  "primary": false
}
```
- `unassigned_info` 부분: 미할당 이유
- `node_allocation_decisions` 부분: 각 노드가 해당 샤드를 거부한 이유

### 자주 나오는 실패 원인
| 원인 | 대응 |
|---|---|
| `max_retry` 소진 (가장 빈번) | 즉시 `POST _cluster/reroute?retry_failed=true` |
| 서킷 브레이커 작동 | 클러스터 부하 감소 후 재할당 요청, 임시로 서킷 브레이커 임계치 완화 |
| `too many open files` | 클러스터 부하 감소 후 재할당 요청 |
| 댕글링 인덱스 / 데이터 없음 | 7.3.5절 댕글링 인덱스 대응 참조 |

### 재할당 요청
```
POST _cluster/reroute?retry_failed=true
# 요청 본문은 반드시 비워 둘 것
```

### 수동 샤드 이동
```json
POST _cluster/reroute
{
  "commands": [
    {
      "move": {
        "index": "test-index-es",
        "shard": 8,
        "from_node": "es02",
        "to_node": "es01"
      }
    }
  ]
}
```

## 6. 댕글링 인덱스

### 정의
노드의 로컬 디스크에 샤드 데이터는 있는데 클러스터 메타데이터에는 해당 인덱스 정보가 없는 상태.

### 발생 원인
**시나리오 1** (일반적, 큰 문제 없음):
- 노드가 클러스터에서 제외된 동안 클러스터에서 해당 인덱스 삭제
- 노드 재합류 시 로컬 데이터 ↔ 클러스터 메타데이터 불일치 → 댕글링 인덱스 발생

**시나리오 2** (심각):
- 마스터 노드 심각한 장애로 메타데이터 자체가 손상

### 인덱스 묘비 (Index Tombstones)
- 인덱스 삭제 시마다 클러스터 상태에 "이 인덱스는 삭제됐다"는 기록 남김
- 롤링 리스타트 중 인덱스 삭제가 일어나도 묘비가 있으면 댕글링 인덱스로 취급하지 않음
- 기본 최대 500개 (`cluster.indices.tombstones.size`, 정적 설정 - 재기동 필요)

### 버전별 차이
- **7.5 미만**: 댕글링 인덱스 자동 적재 (위험! 모든 샤드가 없으면 yellow/red 진입)
- **7.5 이상**: 자동 적재 비활성화, API로 수동 관리
- **7.9 이상**: 댕글링 인덱스 발생해도 티가 나지 않음 → 직접 목록 조회 필요

### 대응 절차

**목록 조회**
```
GET _dangling
```

**삭제 (더 이상 필요 없는 데이터)**
```
DELETE /_dangling/[인덱스 UUID]?accept_data_loss=true
```

**적재 (데이터를 최대한 살리려는 경우)**
```
POST /_dangling/[인덱스 UUID]?accept_data_loss=true
```
- 주의: 최신 상태를 보장하지 않음, 샤드 일부만 있으면 yellow/red 진입 가능
- 동일 이름의 인덱스가 이미 있으면 적재 실패 → 이름 충돌 해결 먼저

## 7. 날짜가 넘어갈 때 대량 인덱스 생성

### 문제 상황
시계열 인덱스 (일 단위) 사용 시 자정에 대량의 인덱스가 동시 생성 → 부하 집중 → 색인 실패.

### 왜 특히 장애 복구 중에 위험한가
복구 중 재시작된 노드는 샤드 복구가 완료되지 않아 "샤드 수가 적은 노드"로 판정됨. 새 인덱스의 모든 샤드가 이 노드에 몰릴 수 있음.

### 대응 방법

**단기 대응 1: 균형 설정 임시 비활성화**
```json
PUT _cluster/settings
{
  "transient": {
    "cluster.routing.allocation.balance.shard": 0
  }
}
```
- 기본값: 0.45 (샤드 균형 고려)
- 0으로 설정: 균형 무시, 순서대로 단순 배정
- 비상 처리 후 반드시 원래 값(0.45)으로 복구

**단기 대응 2: 미리 인덱스 분산 생성**
- 자정에 N개 생성 예정 → 21시: N/3, 22시: N/3, 23시: N/3 미리 생성
- cron이나 배치 도구 활용

**근본 해결**
- 노드 추가로 클러스터 확장
- 클러스터 용도별 분리
- 매핑 재검토 및 정리

## 8. 특정 노드 성능 저하

### 분석 절차

1. **서버 로그 + GC 로그 확인**
   - 일시적 STW인지 vs 지속적/일관적 저하인지 구분

2. **일시적인 경우**
   - 슬로우 로그로 무거운 요청 인입 여부 확인
   - 다른 노드 로그와 비교
   - 재기동으로 개선 가능하면 재기동 고려

3. **지속적인 경우**
   - 샤드 배분 상태 확인 (`GET _cat/shards`)
   - 특정 노드에 부하 집중 원인 파악:
     - 샤드 수 적고 단일 샤드 크기 큰 경우 → 샤드 수 늘려 분산
     - `cluster.routing.allocation.balance.shard` 설정 점검
     - 클라이언트가 특정 데이터 노드에만 직접 요청하는지 확인
     - 클러스터 용도별 분리 고려

4. **하드웨어 의심**
   - 부하 집중 원인 없는데 성능 저하 → 하드웨어 문제
   - 다른 노드와 하드웨어 지표 비교
   - 대규모 ES에서 가장 흔한 하드웨어 문제: 디스크 관련 (`iostat`으로 점검)

## 9. 샤드 복구 전략

### 9.1 복구 기본 원칙

- 샤드 크기/개수가 복구 속도에 큰 영향 → 너무 큰 샤드 사전 방지
- 클라이언트 트래픽 차단 시 복구 속도 향상 (복구 중 변경분 재처리 불필요)

### 9.2 복구 진행 상황 확인

```
GET [인덱스 이름]/_recovery?human
GET _recovery?human
```

**복구 단계 순서**:
| 단계 | 설명 |
|---|---|
| `INIT` | 복구 작업 아직 시작 안 됨 |
| `INDEX` | 루씬 파일 복구 중 |
| `VERIFY_INDEX` | 색인 검증 |
| `TRANSLOG` | translog에서 작업 재처리 |
| `FINALIZE` | translog 완료 후 최종 작업 |
| `DONE` | 복구 완료 |

- `index.size`, `index.files`, `translog` 항목으로 진행률 파악

### 9.3 복구 속도 조정

**동시 복구 작업 수 (네트워크 경유 - 복제본 샤드)**
```json
PUT _cluster/settings
{
  "transient": {
    "cluster.routing.allocation.node_concurrent_recoveries": 4
  }
}
```
- 기본값: 2
- 복제해 주는 수 / 복제받는 수 동시 지정

**동시 주 샤드 복구 수 (로컬 디스크 - 네트워크 불필요)**
```json
PUT _cluster/settings
{
  "transient": {
    "cluster.routing.allocation.node_initial_primaries_recoveries": 8
  }
}
```
- 기본값: 4
- 네트워크 미사용이므로 더 높은 값 가능
- 미할당 주 샤드 빠른 복구가 중요할 때 일시적으로 크게 높임

**네트워크 대역폭 제한**
```json
PUT _cluster/settings
{
  "transient": {
    "indices.recovery.max_bytes_per_sec": "512mb"
  }
}
```
- 기본값: 40MB (cold/frozen 티어는 40-250MB)
- 색인 작업 완전 차단 시: 크게 올려도 무방

### 9.4 복구 우선순위 조정

```json
PUT [인덱스 이름]/_settings
{
  "index.priority": 10
}
```
- 기본값: 1
- 값이 높을수록 먼저 복구
- 동일 priority → 인덱스 생성 시각 최신 순 → 인덱스 이름 내림차순

## 10. 서킷 브레이커

### 정의
과도한 메모리 사용이 예상되는 요청을 사전에 차단하여 노드 중단을 방지하는 메커니즘.

### 왜 필요한가
ES의 설계 철학: "처리하다 죽는 것"보다 "처음부터 과도한 요청은 거부"가 더 안전. 서킷 브레이커 발동 시 클라이언트가 재시도 책임을 가짐 (ES가 자동 재시도하지 않음).

### 종류와 기본값

| 종류 | 설정 | 기본값 | 설명 |
|---|---|---|---|
| 필드 데이터 | `indices.breaker.fielddata.limit` | 힙의 40% | fielddata 메모리에 올라갈 때 예상 메모리 |
| 요청 | `indices.breaker.request.limit` | 힙의 60% | 단일 요청의 데이터 구조 메모리 |
| 실행 중 요청 | `network.breaker.inflight_requests.limit` | 힙의 100% | transport/HTTP로 들어오는 모든 요청 크기 |
| 부모 (실제 메모리) | `indices.breaker.total.limit` | 힙의 95% (real memory 체크 시) | 실제 메모리 사용량 + 자식 브레이커 합산 |
| 스크립트 컴파일 | `script.max_compilations_rate` | 75/5m | 메모리 기반이 아닌 횟수 기반 |

### 동적 설정 변경

```json
PUT _cluster/settings
{
  "transient": {
    "indices": {
      "breaker": {
        "fielddata.limit": "30%",
        "request.limit": "70%",
        "total.limit": "80%"
      }
    },
    "network": {
      "breaker": {
        "inflight_requests.limit": "95%"
      }
    }
  }
}
```

### 부모 서킷 브레이커 특징
- 실제 JVM 메모리를 체크 (7 버전부터 추가)
- `indices.breaker.total.use_real_memory`:
  - `true` (기본): 실제 메모리 체크, 임계값 힙의 95%
  - `false`: 자식 브레이커 합산만 체크, 임계값 힙의 70%
- 7 버전 이상 업그레이드의 핵심 이유 중 하나

### 서킷 브레이커 정보 확인 (실무가이드)
```
GET _nodes/stats
```
응답의 `breakers` 섹션:
- `tripped`: 해당 브레이커가 발동된 횟수
- `estimated_size`: 현재 예상 메모리 사용량
- `limit_size`: 브레이커 임계치

### 스크립트 컴파일 브레이커
- `params` 미사용 시: 숫자만 바꿔도 다른 스크립트로 인식 → 모두 별도 컴파일
- `params`를 활용하면 한 번만 컴파일
- 설정 변경: `script.max_compilations_rate: "50/5m"`

### 트레이드오프
- 임계치를 높이면: 노드 OOM 위험 증가
- 임계치를 낮추면: 정상적인 요청도 거부될 수 있음
- 장애 복구 중 임시로 완화 후, 완료 후 원복 필요

## 11. 원활한 장애 복구를 위한 서비스 구조 설계

### 11.1 앞쪽에 메시지 큐 두기

**구조**: 서비스 → 메시지 큐 → 별도 컨슈머 → ES

**장점**:
- 장애 시 컨슈머만 내리면 자연스럽게 색인 차단
- 서비스는 장애 중에도 데이터 생산 계속 가능
- 서비스와 ES의 결합이 느슨해짐 (loose coupling)
- 장애 종료 후 오프셋을 되감아 재처리 가능 (카프카의 경우)

**선택지**:
- Apache Kafka (권장, 오프셋 되감기, 대규모 커뮤니티)
- Logstash (Elastic 공식 지원)
- Fluentd (사용자 많음) → EFK 스택
- StreamSets Data Collector (성능 요건 느슨할 때)
- Kafka Connect - Elasticsearch Connector (카프카 중심 아키텍처)
- 직접 개발 (높은 성능 요건, 개발/관리 비용 증가)

### 11.2 멱등하게 설계하기

**멱등성이 없으면**: 타임아웃/장애 후 재처리 시 정확히 어디서 실패했는지 추적해야 함 → 복잡.

**멱등한 설계 방법**:
- 색인 시 문서에 `_id` 반드시 지정
- 카프카 오프셋 조합으로 `_id` 생성: `{topic}-{partition}-{offset}`
- 주의: 카프카 다중화 구성 시 같은 조합이 다른 메시지일 수 있음

**활용 시나리오**:
- 매핑/샤드 수 변경 시: alias + reindex → reindex 중 발생 변경분은 오프셋 되감아 재처리

### 11.3 클러스터 용도별 분리

**왜 분리해야 하는가**:
- 키바나 분석 작업이 서비스 ES 장애를 유발하면 서비스 장애로 이어짐
- 중요도별로 장애 대응 전략이 달라야 함

**분리 기준**:
- 서비스 데이터 (NoSQL 스토리지) vs 로그 데이터 vs BI/분석 데이터
- 중요도 높은 클러스터: 신중하고 보수적인 복구
- 중요도 낮은 클러스터: 풀 리스타트 + 문제 데이터 삭제 같은 과감한 방법 선택 가능

**서비스 로그 클러스터의 이점**:
- 색인 순서 중요하지 않음 → 장애 후 bulk API 병렬 호출로 빠른 재처리
